核心論題

Base44 的故事是當代最濃縮的「AI-native 工程組織」樣本：一個人的 Vibe Coding 平台，在一年內被 Wix 收購，從 2 人衝到 80 人——整個過程幾乎沒有任何傳統的管理流程，卻維持了驚人的開發速度。

讓任何人都能「感覺」出一個軟體

Vibe Coding 是 2024 年末興起的概念：你不需要寫程式，只需要描述你「想要什麼感覺」，AI 就幫你把 App 建出來。Base44 就是提供這種體驗的平台。

  • 目標用戶：非技術人員（想快速驗證想法）、技術人員（想加速原型開發）

  • 關鍵里程碑：2025 年被 Wix（全球最大建站平台之一）收購

Base44 的故事說明：AI 工具讓一個人的創業公司可以比過去大十倍的團隊移動得更快。

三大核心原則

I

極度簡單（Keep It Stupid Simple）

每次面對挑戰，第一反應是「最簡單的解法是什麼？」而非立刻建流程、開委員會、寫規範。一個 Prompt 能解決的問題，不建系統。

II

從過往行為提煉品味（Encode Taste from Past Actions）

不需要會議來制定規範。讓 Claude 讀取過去的 PR 評語和實驗數據，自動提煉出「我們是什麼樣的團隊」——這比任何手寫規範都更真實。

III

吃自家狗糧（Dogfood Your Own Product）

用 Base44 這個 Vibe Coding 平台建立 Base44 的內部工具。這創造了最真實的使用者回饋迴路，同時展現了 AI-native 工具的真實能力。

"The bottleneck will keep moving. For now, our current challenge is how do we do post-validation correctly — once a pull request reaches production, how do you make sure it's moving the right needle?" 「瓶頸會持續移動。現在我們面臨的挑戰是如何正確地做上線後驗證——當 PR 進入生產環境後，如何確保它真的在推動正確的指標？」

— Gabriel, AI Engineering Manager, Base44
I

即時 Onboarding

傳統 Onboarding 依賴 Confluence 或 Notion 的靜態文件——這些文件往往在上線當天就已過時。Base44 的解法是：用兩個 Prompt，取代所有文件。

傳統方式

靜態文件 Onboarding

寫 Confluence 頁面 → 更新過期 → 寫維護腳本 → 文件繼續過期。維護文件本身就是一個全職工作。

Base44 方式

兩個 Prompt = 實時地圖

Prompt 1：「整理所有 Commit，告訴我每個人負責什麼領域」→ 即時組織地圖。Prompt 2：「給我這個組件的 Mermaid 架構圖」→ 即時架構圖。

給初學者 · 為什麼靜態文件在 AI 公司特別危險？
Vibe Coding 公司的文件半衰期：以天計算

在一個每天合入幾十個 PR 的高速團隊，架構可能一週內就完全重構。靜態文件的問題不是它會過期——而是你永遠不知道它什麼時候過期了。

  • Commit 歷史：永遠是最新的「真相」，AI 可以即時解讀
  • Mermaid 圖：Claude 可以從代碼直接生成，不需要人工維護
  • 時間成本：寫文件 ≈ 0，因為 Claude 隨時生成；維護文件 = 0，因為 Claude 每次都重新生成

「AI 即文件，Commit 即事實。」這是 Base44 的 Onboarding 哲學。

關鍵故事 · WhatsApp 整合：4 天完成「2 週任務」

新員工的第一個任務，比任何人預期都快

Base44 想在平台上加入 WhatsApp 整合——讓用戶可以透過 WhatsApp 與 AI Agent 對話。這需要：Meta API 整合、修改 Agentic 流程、前端界面更新。Yoav 和 Maor 一致估計需要 1-2 週。

"We gave that Thursday night. Sunday morning everything was ready. He onboarded on Thursday with using those simple prompts, he sent it over to PR, the PR Maor review had two, three small comments, and we were ready to move on to production."

週四晚上交任務，週日早上 PR 已過，週一進生產。這位工程師的整個 Onboarding 就是兩個 Prompt。

這個故事的啟示不只是速度——而是：好的 Onboarding 工具讓人一開始就能全力衝刺，而不是花一週「暖機」。

從創辦人 PR 評語提煉審查品味

Maor 的另一個身份是「最嚴格的後端 PR 審查員」。但他沒辦法24/7 review 每一個 PR。解法？

給初學者 · 什麼是「提煉品味」？
不需要開會制定規範——讓過去的行為說話

想像你是一個新加入的工程師，你不知道這個團隊「喜歡什麼樣的代碼」。與其讓主管花一週告訴你，Base44 直接讓 Claude 讀取 Maor 過去的所有 PR 評語，提取出「Maor 最在乎什麼」。

  • 輸入：Maor 過去 1-2 週在 GitHub 上的所有 PR 評語
  • 輸出：「Maor 最在乎的 5 件事」寫入 Claude Code Instructions
  • 效果：自動化 PR 審查，每天更新，永遠反映最新的「品味」

這是把「人類的直覺」轉化為「可執行的規則」的最直接方法。

Step 1
收集材料：用 GitHub API 撈取 Maor 過去幾週的所有 PR 評語
Step 2
提煉原則：讓 Claude 分析評語，提煉出最重要的審查原則（不是規則，而是「品味」）
Step 3
寫入 Instructions：把提煉出的原則放入 Claude Code 的 CLAUDE.md 或 Instructions
Step 4
每隔幾天更新：重新執行提煉，確保 PR 審查員的「品味」跟上 Maor 的最新想法

給初學者 · 為什麼用 Haiku 而不是 Sonnet？
對的工作，用對的模型

分類一則訊息是「高挫折感」或「低挫折感」，不需要多高的智能——這是個簡單的二元分類任務。Haiku 比 Sonnet 便宜許多，速度更快，對這個任務綽綽有餘。

  • Haiku 適用：簡單分類、快速摘要、頻繁呼叫的批量任務
  • Sonnet 適用：複雜推理、代碼生成、需要更高品質輸出的任務
  • 成本考量：在生產流量上跑 Haiku，成本可能是 Sonnet 的 1/10 以下

用最便宜的模型做最合適的任務，是 AI 產品工程的基本財務紀律。

一夜翻倍：40 人到 80 人

「我們合併了另一個 Vibe Coding 產品團隊，在一個晚上，頭數從 40 翻倍到將近 80。」這帶來了全新的挑戰——規模化實驗、真正的 Eval 系統、以及如何讓 QA 不成為瓶頸。

關鍵故事 · 一夜 40→80 的震盪

合併帶來的是人，不是流程

Base44 在 Wix 的支持下持續擴張，同時一個來自 Wix 內部的 Vibe Coding 產品團隊加入了。一夜之間，工程團隊從 40 人變成近 80 人。

"We started hiring more externally, we had more internal movers moving from Wix to Base44. And then we even merged a different product working on vibe coding. And in one single night, we doubled our head count from 40 people to almost 80."

新來的工程師有不同的背景、不同的 AB 測試直覺、不同的代碼風格。Gabriel 面對的挑戰是：如何讓 80 人像 10 人一樣快速移動？答案依然是：讓過去的行為說話。

AB 測試的民主化：PostHog MCP

給初學者 · 什麼是 PostHog 和 MCP？
PostHog：產品分析 + AB 測試平台

PostHog 是一個開源的產品分析工具，讓工程師可以追蹤用戶行為、跑 AB 測試、分析功能使用率。它有一個「MCP Server」，讓 Claude 可以直接查詢 PostHog 的數據。

  • PostHog 做什麼：記錄每個實驗的假設、結果、顯著性
  • MCP 是什麼：Model Context Protocol，讓 AI 模型可以安全地呼叫外部工具和數據
  • 組合效果：Claude 可以直接讀取 PostHog 的所有歷史實驗，自動分析模式

就像給 Claude 一個資料庫查詢工具，讓它能直接從 PostHog 的歷史學習。

問題
沒有人知道團隊的 AB 測試指南是什麼：多久需要跑？要追蹤哪些 KPIs？什麼時候可以直接上線不用測試？
選項 A
召集多部門委員會開會制定指南 → 我們真的很討厭開會
選項 B
讓 Claude 讀取過去 100 次 PostHog 實驗 + 對應 PR，提煉出指南 → 幾小時內有可用版本
結果
現在每個 PR 開啟，Bot 自動在 GitHub 評論：「可以直接上線 / 需要 AB 測試 7 天 / 需要測試 30 天（影響轉換率）」

"We figured out that our past actions, they could convey our guidelines in the best way possible." 「我們發現，我們過去的行動，能以最好的方式傳遞我們的指南。」

— Gabriel, Base44
給初學者 · 為什麼 30 天 vs 7 天？
不同功能需要不同的測試時長

AB 測試的時長取決於你要測量的指標的「信號速度」。有些指標很快能看到差異，有些需要很長時間才能確定。

  • 7 天：UI 改動、功能可用性（用戶立刻使用或不使用）
  • 30 天：轉換率、訂閱率（需要覆蓋完整的月費用戶週期）
  • 直接上線：純技術改動（效能優化、Bug 修復），不影響用戶行為

過去 100 次實驗就是最好的「老師」——它們記錄了團隊什麼時候選了什麼，結果是否正確。

Dogfooding：用 Base44 建 Base44 的內部工具

所有內部追蹤工具——實驗儀表板、QA 報告、指標看板——全部用 Base44 建立，連接 BigQuery（數據倉庫）+ PostHog + GitHub。

Dogfooding 的好處

最真實的使用者回饋

工程師每天使用自己的產品。任何體驗問題都會立刻被感受到，而不是等客戶投訴。這是最快的產品改進迴路。

技術驗證

證明平台能力

「我們用自己的 Vibe Coding 平台建所有內部工具」本身就是最有力的產品展示。如果連自己都用不好，怎麼說服客戶？

I

Eval：等待對的時機

Base44 刻意等到 80 人規模才認真投資 Eval 系統——不是因為不重要，而是因為 ROI 在之前不夠高。這個時機判斷本身就是一個重要的工程決策。

"We really value simplicity. Evals is a great example. We held it off until it was the right moment to build it, and then we went all in." 「我們非常重視簡單性。Eval 就是個好例子。我們等到了正確的時機才建立它，然後全力投入。」

— Gabriel, Base44
給初學者 · 什麼是 Eval（評估系統）？
AI 產品的「單元測試」

傳統軟體有單元測試：寫一個函數，測試它的輸入輸出。AI 產品的 Eval 也是類似的概念：給 AI 一個輸入，驗證輸出是否符合期望。

  • 為什麼難：AI 的輸出是機率性的，沒有唯一正確答案
  • Base44 的解法：不只評估模型輸出，而是評估「生成的 App 是否真的能用」
  • 關鍵洞見：如果 App 有局部錯誤，不算 Eval 失敗——讓 Agent 自己修復，觀察修復了幾輪

這個「允許修復」的設計，讓 Eval 更接近真實用戶體驗，而非完美的一次性輸出。

用戶模擬器：不只評估模型，評估整個系統

I

User Simulator（用戶模擬器）

模擬真實用戶在 Base44 上的操作：輸入需求 → 生成 App → 操作 App → 回報問題。用 Stagehand 在小盒子內執行。

II

錯誤回饋迴路

App 有局部錯誤 → 錯誤訊號回饋給 Agent → Agent 嘗試修復 → 重新評估。追蹤：修復了幾輪、每輪延遲、總成本。

III

CI/CD 整合

任何 AI 代碼改動 → 自動啟動 Eval → 真實 Base44 App 實例 → Stagehand 模擬操作 → 截圖 + 報告。

關鍵故事 · Hello World 不只是煙霧測試

最簡單的 Eval 有時候足夠有力

Base44 最重要的 Eval 是「Hello World App」——讓 AI 生成一個顯示「Hello World」文字的 App，驗證文字是否正確顯示。

"The most canonical eval we have is the Hello World app. It doesn't mean Base44 is performing the way we want — it's like a smoke test. But fun fact, these evals will pass on the smallest model you can think of, which is really cool."

這個故事的啟示：Eval 不需要從最複雜的情境開始。一個能穩定運行的簡單基準，比一個不穩定的複雜系統更有價值。先確保最基本的不會壞，再逐步增加複雜度。

II

QA 自動化：Skills + CLI Tools

測試特定訂閱等級的邊緣案例，過去需要 QA 工程師手動設置資料庫狀態。現在，Claude Code 有兩個武器——Skills 和 CLI Tools——可以自己搞定。

給初學者 · 什麼是 Skills（在 Claude Code 中）？
把「集體記憶」封裝成可重用的技能

Skills 是你教給 Claude Code 的「操作手冊」——比如「如何在 Base44 上完成新用戶註冊流程」，包含所有需要點擊的元素、等待的事件、驗證的結果。

  • 不用 Skill 時：每次 Claude 都要重新「學習」平台的界面和流程，花很多時間探索
  • 有 Skill 時：Claude 直接知道怎麼走，不需要重新摸索
  • 80% 原則：Skill 覆蓋最常用的 80% 流程，剩下 20% 讓 Claude 自己探索

這是「明確知識」和「隱性能力」的最佳組合。

QA 自動化架構：Skills + CLI + Stagehand
第一層 · 知識

Skills（流程技能）

封裝所有主要用戶流程。Claude 知道如何完成「新用戶註冊」「升級方案」「提交 App」等操作，不需要每次重新探索 selectors。

第二層 · 能力

CLI Tools（資料庫操控）

特製的 CLI 工具，讓 Claude 可以直接修改資料庫和 API，設定特定測試環境（如：把用戶設定為「達到點數上限的付費用戶」）。

第三層 · 執行

Stagehand（瀏覽器）

AI 驅動的瀏覽器自動化工具，讓 Claude 可以在真實瀏覽器中操作 Base44，截取截圖，驗證功能是否正確。

↓ 輸出：測試計畫 + 截圖報告 → 傳送到 Base44 內部看板（用 Base44 建的）

"We needed Claude to be able to go to the database and overwrite the setup so that they can just test that specific flow. So we created CLI tools that abstract our APIs and databases specifically for the use case of setting up tests." 「我們需要 Claude 能夠直接修改資料庫、覆寫測試設定，這樣它就能直接測試那個特定流程。於是我們建立了 CLI 工具，專門為測試環境設置而設計的 API 和資料庫抽象層。」

— Gabriel, Base44

80% 法則的黃金平衡

過度封裝（100%）

每個流程都寫成 Skill

維護成本高。每次界面改變都要更新 Skill。而且 Claude 的自主探索能力被過度限制——有些場景它自己能搞定。

黃金平衡（80%）

核心流程 Skill + 其餘讓 Claude 探索

80% 的常用流程有 Skill 覆蓋，確保效率。剩下 20% 讓 Claude 自己判斷，利用它的理解能力。維護成本低，彈性高。

V

Guided Discussion

六道思考題，幫助你把 Base44 的規模化經驗轉化為對自己組織的洞察。

Q1

Base44 的核心 Onboarding 哲學是「Commit 歷史即文件，AI 即導師，靜態文件不存在」。這在什麼類型的公司和團隊中最有效？在什麼情況下靜態文件仍然不可或缺？

查看建議思考方向
  • 歷史對照：傳統企業（銀行、政府）有大量流程文件需求，但這些文件的更新頻率也很低。Base44 的方法更適合高速迭代的 AI-native 公司。
  • 講者隱含論點：Yoav 說「靜態文件在上線當天就過時了」——這在 AI 開發週期比過去快 10 倍的公司特別真實。
  • 職涯啟發：「Commit 即文件」這個思維，要求工程師寫有意義的 Commit Message。這不只是技術實踐，也是溝通紀律。
  • 反向思考：有些知識天然適合靜態文件——比如「為什麼做這個技術決策」的背景脈絡。Commit 記錄了「做了什麼」，但可能缺少「為什麼」。
Q2

「挫折感指標」是一個極其聰明的「代理指標」（Proxy Metric）設計。你認為這個設計的最大局限是什麼？還有哪些 AI 產品的狀態可以用類似的「行為信號」而非「直接評估」來追蹤？

查看建議思考方向
  • 歷史對照：NPS（Net Promoter Score）也是一個代理指標——用「推薦意願」代替「滿意度」的直接測量。代理指標的風險是它可能與真實目標脫鉤。
  • 講者隱含論點：Yoav 說「挫折感高/低是個強烈的單一信號」——但一個二元信號可能捕捉不到「雖然能用但體驗不好」的情況。
  • 職涯啟發：設計一個「低成本但高信號的指標」是產品工程的核心技能。思考：你的產品裡有沒有用戶行為的「天然失敗信號」？
  • 反向思考：如果用戶沉默也可能是因為「太沮喪了懶得抱怨」怎麼辦？靜默不一定等於滿意。
Q3

Gabriel 說「我們可以從過去的行動中提煉出品味」——用 Claude 分析 100 次 PostHog 實驗得出 AB 測試指南。但「提煉出的品味」可能會固化過去的偏見，限制創新。你認為如何在「保留組織品味」和「允許突破性改變」之間取得平衡？

查看建議思考方向
  • 歷史對照：Netflix 的 A/B testing 文化也曾因「只測試增量改動」而錯過了一些根本性的產品轉型機會。歷史數據的局限性在於它只反映「曾經嘗試過的事」。
  • 講者隱含論點：Gabriel 說「指南是第一個版本，我們可以迭代」——這表明他們視指南為活文件，而非永久規則。
  • 職涯啟發：「提煉品味」最適合用於「執行層面的規範」（代碼風格、AB 測試時長），而非「策略層面的判斷」（要不要進入新市場）。
  • 反向思考：有沒有可能設計一個 Prompt，讓 Claude 不只提煉過去的模式，也主動指出「這個模式在哪些情況下可能不適用」？
Q4

Gabriel 特別強調「Eval 系統要等到對的時機才建立，然後全力投入」。在 AI 產品開發中，什麼信號告訴你「現在是建立 Eval 系統的對的時機」？你的組織現在是過早還是過晚建立 Eval？

查看建議思考方向
  • 歷史對照：傳統軟體工程中，「何時引入自動化測試」也是個長期爭論。通常答案是「當手動測試的成本超過自動化的建置成本時」。
  • 講者隱含論點：15 人時 ROI 不夠，80 人時 ROI 足夠。這說明 Eval 的「正確時機」和團隊規模、產品複雜度、以及你有多少 AI 工程師能分配去建它密切相關。
  • 職涯啟發：在 AI 產品中，「沒有 Eval」的代價是「不知道你的改動是否真的讓產品更好」。這個風險在小規模時可以靠直覺彌補，但隨著規模增長會失控。
  • 反向思考：「等到對的時機」是否有可能讓你錯過建立 Eval 的最佳窗口？如果在 15 人時就開始建立最簡單的 Eval 骨架，80 人時會不會更容易？
Q5

Base44 的 QA 方法核心是「給 Claude 改資料庫的權限（CLI Tools）」。這在安全層面帶來了什麼風險？你認為應該如何設計這個 CLI Tool 架構，在賦予 Claude 足夠能力的同時，防止意外或惡意的資料庫破壞？

查看建議思考方向
  • 歷史對照：「最小權限原則」（Principle of Least Privilege）是資訊安全的基本原則。給 Claude 改資料庫的能力，需要嚴格限制它只能做「測試設置」相關的操作。
  • 講者隱含論點：Gabriel 說「CLI 工具是特別為測試環境設計的抽象層」——這表明他們不是讓 Claude 直接訪問生產資料庫，而是有一個隔離層。
  • 職涯啟發：設計 AI Agent 的工具調用邊界，是 AI 系統安全的核心議題。思考：你的工具應該是「唯讀」的，還是有「有限的寫入」能力，還是「完全讀寫」？
  • 反向思考：在測試環境（staging）vs 生產環境（production）中，這個風險完全不同。好的架構設計應該讓 Claude 無論如何都無法意外修改生產資料。
Q6

Gabriel 說「我們用 Base44 建了我們所有的內部工具，這讓我們成為自己產品最真實的用戶」。Dogfooding（吃自家狗糧）是否適用於所有類型的產品？什麼情況下 Dogfooding 可能帶來盲點，反而讓你看不到真實用戶的問題？

查看建議思考方向
  • 歷史對照：Facebook 工程師使用 Facebook，Gmail 工程師使用 Gmail——這是科技公司的標準實踐。但醫療 AI 公司的工程師，不一定能真實模擬醫生的使用場景。
  • 講者隱含論點：Base44 是一個開發工具，工程師本身就是目標用戶之一——所以 Dogfooding 在這裡特別有效。對於面向非技術用戶的產品，工程師可能過於技術化，看不到用戶的困惑。
  • 職涯啟發：Dogfooding 的真正價值不是「找到所有 Bug」，而是建立對產品的「情感連結」——你更容易發現讓你自己惱怒的問題，即使那不是最重要的問題。
  • 反向思考：「高級用戶盲點」——技術熟練的工程師可能繞過了普通用戶會卡住的地方，從而永遠看不到那些體驗問題。
