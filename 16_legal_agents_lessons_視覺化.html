I

核心論題：「竊取」最佳實踐

Coding Agent 已經解決了很多問題。你不需要重新發明輪子——只需要學會把輪子裝到你的車上。

Jacob Emeling 是 Legora 的首席軟體工程師。Legora 為律師打造 AI 協同工作空間，今日估值超過 50 億美元，並以創紀錄速度從年收 100 萬美元衝到 1 億美元。在這場演講中，他坦率地分享了一個核心洞察：法律領域的 AI 開發者不需要從零出發，只需要有系統地「竊取（Steal）」Coding Agent 的最佳實踐。

六個月前，Legora 重新思考了整個 Agent 架構——不再使用多 LLM Pipeline，改用類似 Coding Agent 的單一主模型迴圈。結果：即使用最輕量的 Claude Haiku，也能完美完成 10 頁法律文件的逐段翻譯。

II

DocX 的噩夢：舊架構如何失敗

多 LLM Pipeline 的每個交接點，都是一個失敗的機會。

Legora 的舊架構聽起來合理：一個頂層 頂層 Agent 判斷編輯意圖，然後把個別編輯點分發給多個子模型執行。但在實踐中，這個設計有根本缺陷。

DocX 文件是一個高度複雜的壓縮檔——XML、樣式標記、字型嵌入、修訂歷史，全部混在一起。當頂層模型需要「描述」一個 50 頁文件裡哪 15 個地方需要修改時，Token 消耗驚人，而且子模型對整體脈絡的理解是碎片化的。

給初學者 · 一句話搞懂 Pipeline 架構的問題
為什麼多個「小 AI」協作反而更差？

想像你要翻譯一本書，把每章分給不同翻譯員。每個人翻得好，但第一章翻譯員不知道第五章改了一個角色名字——全書讀起來不一致。

  • Pipeline 問題：每個子模型只有局部上下文，無法保持整體一致性
  • Loop 解法：一個翻譯員（模型）翻完整本書，自己發現並修正不一致

「交接」是 AI Agent 最危險的時刻——每次交接都有資訊損失。減少交接，就能減少錯誤。

"Basically, like all the coding agents out there work in a way where they just read, edit, and verify things in a loop... And it just worked. And it didn't have all these weird exhaustiveness issues that we run into with editing legal documents in the past."

「所有 Coding Agent 基本上都用同一種方式：在迴圈中讀取、編輯、驗證……它就是管用了。那些我們過去在編輯法律文件時遇到的奇怪完整性問題，全都消失了。」 — Jacob Emeling, Legora

扁平化中間表示法（Flat IR）

給初學者 · 一句話搞懂 Flat IR
把複雜的 Word 檔「翻譯」成 AI 能讀懂的語言

DocX 對 AI 來說就像亂碼——大量的 XML 標籤、格式資訊、中繼資料混在一起。「扁平化中間表示法」是把這些亂碼轉換成 AI 熟悉的純文字格式：

  • DocX 原始：<w:p><w:r><w:t>條款一</w:t></w:r></w:p>
  • Flat IR：[PARAGRAPH_001] 條款一的完整內文...

把 Word 文件「翻譯」成 AI 的母語，AI 就能像處理程式碼一樣處理合約。

破關時刻 · Claude Haiku 的十頁翻譯

「我們扔進去一份 10 頁文件，要求逐段翻成瑞典文——它就直接做到了」

Jacob 回憶起第一次用新架構做「靈感測試（Vibe Check）」的時刻。他的同事——一位深知法律文件編輯有多難的工程師，之前還建了 Legora 的第一個 Word 插件——被叫來測試。

"他說：『就丟一個 10 頁文件進去，要它逐段翻成瑞典文。這在舊架構裡一直是個大難題。』我們就這樣做了——模型開始一段一段地翻，有時候迷失了方向，重新讀整份文件，發現『啊，我漏了上面這段』，回去補正，繼續跑了大概 10 分鐘。打開文件一看，全部翻好了。"

最令人驚訝的不是翻譯品質，而是測試用的是最輕量的 Claude Haiku 模型——不是 Sonnet，不是 Opus。這證明了正確的工具設計對模型能力的放大效應遠超模型本身的升級。

給初學者 · 一句話搞懂「工具設計 vs 模型能力」
更好的工具 > 更強的模型

很多人認為 AI 效果不好，解法是換更強的模型。Jacob 的故事說明了另一種可能：工具設計（Harness Design）可能比模型升級更重要。

  • 用強模型 + 爛工具：仍然可能失敗
  • 用弱模型 + 好工具：可能超出預期地成功

原因：好的工具設計讓模型能充分利用其已有的能力，而不是在「對抗」工具的限制上浪費計算資源。

給初學者 · 一句話搞懂 Linting
Linting：在 AI 犯錯前先攔截

ESLint 會告訴你「這個變數未定義」或「這個函式從未被呼叫」——在你執行代碼前就找到問題。法律文件 Linter 做類似的事：

  • 程式碼 Lint：「函式 A 呼叫了函式 B，但函式 B 不存在」
  • 法律 Lint：「條款 3.2 引用了附表 C，但 Agent 剛剛刪除了附表 C」

Linter 即時把這個錯誤反饋給 Agent，Agent 就能自動修正，而不是讓律師事後手動檢查。

Agent 修改合約 → Linter 自動掃描交叉參照 → 發現失效引用 → 即時反饋給 Agent → Agent 自動修正

III

盡職調查：法律界的獨特戰場

購買一家公司，你需要在三週內審完 2000 份合約——每份都要找到可能讓交易失敗的紅旗。

企業併購（M&A）的法律盡職調查是一個沒有 Coding 對應物的純法律問題。想像公司 A 正在收購公司 B——法律團隊需要在極短時間內審查 B 的所有合約：供應商協議、員工合約、租約、授權協議……通常數百到數千份。

目標是提取：每份合約的類型、關鍵當事人、重要條款、以及任何可能讓交易失敗的「紅旗（Red Flag）」。

給初學者 · 一句話搞懂「盡職調查」
DD（Due Diligence）：收購前的 X 光檢查

你在二手車市場買車前，要請技師檢查引擎——這就是個人層面的盡職調查。企業收購的盡職調查就是這個過程的企業版，但規模大得多：

  • 要審查的「文件」：全部的法律合約、財務紀錄、智慧財產
  • 要找的「問題」：可能讓收購破局的條款、隱藏負債、合規風險
  • 時間壓力：通常有嚴格的交易時間線，必須在幾週內完成

以前需要大批律師日以繼夜手動審查，現在 Legora 的 Table Review 讓 AI 初步提取，律師只需驗證。

給初學者 · 一句話搞懂「反向引用（Reverse Citation）」
讓 AI 的結論「有憑有據」

AI 說「這份合約有問題」，律師最怕的是：「你是從哪裡看出來的？你有沒有幻覺？」反向引用解決了這個信任問題：

  • AI 提取資訊時，同時記錄它從文件的哪一頁、哪一段話做出結論
  • 律師點一下，系統自動跳到原文並高亮那句話
  • 驗證從「逐字讀完」變成「數秒核對」

反向引用把 AI 的「黑盒子」變成「透明盒子」——律師永遠知道 AI 的結論從哪裡來。這是在高風險專業場景中建立 AI 信任的關鍵設計。

核心設計哲學 · Citation 讓幻覺無所遁形

「引用錨定」：法律 AI 的防幻覺護欄

Jacob 在演講中強調，「把每個答案都錨定在引用（Citation）上」是法律 AI 的必要發明，沒有 Coding 的直接對應。律師的職業責任讓他們無法接受 AI 的「我這樣認為」——他們需要「這個結論來自第 12 頁第 3 段」。

"律師可以驗證 Agent 輸出中某個聲明的來源在哪裡——這不只是方便，而是讓整個 AI 工作流程在高風險法律場景中可行的根本設計。"

Table Review 的設計讓這個驗證過程從「閱讀整份合約尋找來源」（可能需要 30 分鐘）變成「點一下，高亮即現」（3 秒）。這個差異決定了律師是否願意在實際工作中信任 AI——信任建立在可驗證性上，不是模型的聲譽或準確率統計。

規劃模式：從 Coding 直接複製的 UX

如果 Table Review 是「發明」，那規劃模式（Planning Mode）就是最純粹的「復用」——直接把 Claude Code 的 UX 搬進 Legora。

給初學者 · 一句話搞懂「規劃模式」
先規劃，再執行——為什麼這樣更安全？

想像你要重新裝修廚房，有兩種方式：直接叫工人開幹（傳統 AI），或者先給你看設計圖讓你確認再動工（規劃模式）。

  • 直接執行：快，但如果搞錯了很難復原
  • 先規劃後確認：慢一步，但避免不可挽回的錯誤

法律任務的錯誤代價極高（刪錯客戶文件、遺漏關鍵條款），所以「先規劃、律師確認、再執行」是必要的安全設計。

Coding Agent 版

輸入任務 → Claude Code 生成詳細計畫 → 開發者審閱修改 → 確認後執行 → 高風險操作先問再做

Legora 法律版

輸入任務 → Legora 生成修訂計畫 → 律師審閱修改 → 確認後執行 → 刪除客戶文件等高風險操作先問再做

"In your coding agent, you don't want your agent to execute random shell commands that are not sandboxed... And it's a similar thing in an agent working with legal stuff. You don't want it to randomly go and delete a bunch of important client documents."

「在 Coding Agent 裡，你不希望它執行未受 Sandbox 保護的隨機 Shell 命令……法律 Agent 也一樣。你不希望它隨意刪除一堆重要的客戶文件。」 — Jacob Emeling, Legora

給初學者 · 一句話搞懂為什麼 Coding 領先
為什麼 Coding Agent 比其他領域更成熟？

Jacob 提出了一個有趣的問題：為什麼 AI 在程式設計上進步最快？有幾個原因：

  • 訓練數據：網路上有幾十億行開源代碼，法律文件大多是機密
  • 自動評估：代碼能不能跑，直接測試就知道；法律文件的品質難以自動評估
  • 工程師是 AI 的最大使用者，也是最早採用者，形成反饋飛輪

但 Jacob 的論點是：這些「領先」不代表你要等——你要「竊取」已被驗證的最佳實踐。

普適性洞察 · 不只是法律

這個框架適用於所有知識工作

Jacob 刻意強調：法律和程式設計的平行性不是什麼「獨特的神奇組合」，而是任何知識工作都能與 Coding 找到類比的。

"如果你在任何其他垂直領域構建 AI，做這個練習（找出你的領域與 Coding 的平行性）是非常有意思的事。法律和程式設計的平行不是什麼神奇組合——幾乎任何知識工作都有這些特性。"

醫療記錄（病歷 = 程式碼，診斷 = Debug）、財務報告（年報 = 文檔，審計 = Code Review）、學術研究（論文 = 代碼，引用 = 依賴）——每個領域都有等待被「翻譯」的 Coding Agent 最佳實踐。

給初學者 · 三層借鑑框架實作指南
如何把這個框架用在你的領域？

如果你在建構非 Coding 的垂直 AI，Jacob 的框架是最直接的起點：

  • 第一步（復用）：列出 Coding Agent 已解決的 UX 問題（規劃、人機協同、Sandbox），直接套用，不要重新驗證
  • 第二步（翻譯）：找出你的領域的「DocX 問題」——什麼是你的「複雜格式問題」？什麼是你的「Flat IR」？
  • 第三步（發明）：找出你的領域的「Due Diligence 問題」——什麼是 Coding 沒有對應物的獨特痛點？

這個框架讓你避免重新發明輪子，同時找到真正需要創新的地方。

RL 微調的意外紅利

Jacob 分享了一個深刻的技術洞察：為什麼讓法律 Agent 的工具設計「看起來像 Coding」能帶來額外效益：

"My mental model is that you want to have the model almost feel like it's inside a coding agent harness, and it just does a legal task. Because then suddenly you get these benefits from all the reinforcement learning and fine-tuning that is done on the coding agent harnesses."

「我的心智模型是：你想讓模型幾乎感覺自己在一個 Coding Agent Harness 裡，只是在執行法律任務。因為這樣你突然就得到了在 Coding Agent Harness 上做的所有強化學習與微調帶來的好處。」 — Jacob Emeling, Legora
給初學者 · 一句話搞懂 RL 微調的「轉移學習」
讓你的 AI 繼承 Coding 訓練的紅利

Claude 等模型針對 Coding 任務做了大量的強化學習微調（RL Fine-tuning）——模型學會了在 Read-Edit-Verify 這樣的工具模式下，更精準、更徹底地執行任務。

  • 如果你的法律工具設計和 Coding 工具設計「看起來一樣」（相同的工具命名、相同的操作模式）
  • 模型就會「認出」這是它熟悉的模式，應用為 Coding 微調過的行為
  • 你不需要自己做 RL 微調，就免費繼承了成果

這是「站在巨人肩膀上」在 AI 工具設計層面的具體體現。

V

六道思考題

從 Jacob 的演講出發，思考垂直領域 AI 開發的策略選擇與技術取捨。

Q1 · Jacob 說「工具和 Harness 設計的優越性可以大幅彌補模型本體能力的不足」（Haiku 做到了 Sonnet 可能做不到的事）。這個論點的邊界在哪裡？什麼情況下，再好的工具設計也無法彌補模型的能力不足？

查看建議思考方向
  • 量化思考：有沒有辦法量化「工具設計帶來的提升」vs「模型升級帶來的提升」？在什麼任務類型上，工具設計的邊際效益遞減？
  • 講者隱含論點：Jacob 的論述暗示「架構問題」比「模型問題」更常見——你認同嗎？大多數 AI 開發失敗是模型太弱，還是工具設計太差？
  • 職涯啟發：如果工具設計比模型選擇更重要，AI 工程師應該花更多時間在哪裡？這如何影響你分配學習資源的方式？
  • 反向思考：過度依賴「工具設計」的策略有什麼風險？如果 Anthropic 下個月發布一個「會自動選擇最佳工具設計」的模型，這個技能還有價值嗎？

Q2 · Legora 的「反向引用（Reverse Citation）」功能讓律師能驗證 AI 的每個結論——這解決了幻覺問題，但也改變了律師的工作模式。這種「AI 提取 + 人工驗證」的工作流對律師的專業發展有什麼長期影響？

查看建議思考方向
  • 歷史對照：法律資料庫（如 Westlaw、LexisNexis）已經改變了律師做研究的方式——那一代律師失去了什麼能力，又獲得了什麼能力？AI 這次的改變會不同嗎？
  • 講者隱含論點：Jacob 把「反向引用」定位為「讓 AI 可信任」的工具，而不是「讓律師能跳過閱讀」的工具——這個設計哲學背後有什麼假設？
  • 職涯啟發：如果律師不再需要「逐字閱讀 2000 份合約找紅旗」，他們的核心價值在哪裡？什麼技能會更重要？什麼技能會退化？
  • 反向思考：「可驗證性」是信任 AI 的必要條件嗎？還是某些情況下，我們應該允許 AI 做出我們無法完全驗證的判斷？

Q3 · Jacob 建議所有垂直領域開發者「竊取（Steal）」Coding Agent 的最佳實踐。但法律、醫療、金融等受監管行業有嚴格的合規要求——直接套用 Coding Agent 的模式是否安全？需要哪些調適？

查看建議思考方向
  • 具體案例：法律 Agent 使用「Sandbox」（模擬執行環境）有什麼意義？在法律場景中，什麼是「不能沙盒化」的操作？和程式碼執行的 Sandbox 有何根本差異？
  • 講者隱含論點：Jacob 把「危險工具呼叫核准」視為從 Coding 直接復用的 UX——但刪除客戶文件的「危險性」和執行危險 Shell 命令的「危險性」是同一種嗎？
  • 職涯啟發：在你的工作領域，「受監管的 AI 部署」和「技術可能性」之間的落差有多大？如何成為能在這個落差中創造價值的人？
  • 反向思考：「先借鑑、再調適」的策略可能導致法律領域「過度模仿 Coding」，從而忽視法律工作的獨特性——有哪些法律需求可能被這個框架遺漏？

Q4 · Legora 把 DocX 轉換為「扁平化中間表示法（Flat IR）」讓 AI 能處理它。這個「中間表示法」設計是否有通用模式？在你的領域，有哪些「AI 難以直接處理的格式」需要類似的轉換層？

查看建議思考方向
  • 類比擴展：PDF、Excel、CAD 文件、醫療影像——哪些格式有類似 DocX 的問題？它們的「Flat IR」長什麼樣？誰在做這個領域的工作？
  • 講者隱含論點：Jacob 說 Flat IR 讓模型「感覺在 Coding Agent Harness 裡」——格式的轉換不只是技術問題，也是讓模型能「認識」任務類型的介面設計問題。
  • 職涯啟發：「格式轉換層」是 AI 應用中被嚴重低估的技術價值——有哪些公司或開源工具已經在做這個工作？能否成為你的創業機會或技術優勢？
  • 反向思考：從源格式直接學習（如 Multimodal AI 直接理解 PDF 圖片）vs 通過 Flat IR 轉換——哪個方向長期更有潛力？兩者的代價和收益各是什麼？

Q5 · Legora 創下「ARR 1 → 100 百萬美元」的創紀錄增速。Jacob 把這歸功於六個月前的架構轉型。但技術改進真的能直接解釋業績增長嗎？其他因素（市場時機、銷售策略、品牌）的貢獻有多少？

查看建議思考方向
  • 量化思考：有什麼方法可以分離「技術改進」和「市場時機」對業績增長的貢獻？如何設計一個自然實驗來測試這個問題？
  • 講者隱含論點：Jacob 的演講把技術架構轉型和業績增長並置在一起，但沒有明確建立因果關係——這是確認偏誤（Confirmation Bias）的例子嗎？
  • 職涯啟發：在 AI 創業中，「技術領先」和「市場時機」哪個更重要？ Legora 的成功對你選擇垂直領域或技術策略有何啟示？
  • 反向思考：如果 Legora 沒有做架構轉型，用舊的 Pipeline 架構，能否達到類似的業績增長？如果答案是「可能可以」，Jacob 的演講論點有什麼問題？

Q6 · Jacob 說法律領域落後 Coding Agent 約六個月——但他在 2026 年 5 月說這話。考慮到 AI 發展速度，「六個月差距」在今天意味著什麼？這個差距是在縮小還是在擴大？

查看建議思考方向
  • 歷史對照：回顧 2024 年到 2026 年，不同垂直領域的 AI 採用速度有何差異？有沒有哪個垂直領域的採用速度比 Coding 還快？原因是什麼？
  • 講者隱含論點：「六個月差距」假設了固定的「跟隨速度」——但如果其他領域的開發者都學會了「竊取 Coding 最佳實踐」，差距會不會快速消失？
  • 職涯啟發：如果你是法律、醫療或金融 AI 的開發者，「六個月時間窗口」意味著什麼機會？如何把握這個窗口期建立競爭優勢？
  • 反向思考：「差距」的存在是否因為某些根本性的行業差異（法規、數據可用性、職業文化），而不只是「沒學到 Coding 的最佳實踐」？如果是，Jacob 的樂觀預期是否過度？

本場演講高頻術語
  • DocX：Microsoft Word 文件格式，本質上是含大量 XML 的壓縮檔，AI 難以直接操作

  • Lazy LLM：模型在處理長文件時中途跳過段落的「偷懶」現象
  • Table Review：Legora 的盡職調查工具，結構化提取 + 反向引用高亮
  • Reverse Citation：AI 提取結論時同時記錄來源，用戶點一下即可驗證
  • Due Diligence（盡職調查）：企業收購前對目標公司的全面法律與財務審查
