I

電光石火的一年

去年三月，Jeremy 加入 Anthropic 時，Opus 4 是最先進的模型，Claude Code 尚未正式發布，能力曲線還只是剛起步。今天，那個起點看起來已像史前時代。

Jeremy 在倫敦開場問了一個問題：「誰這週合併了一個完全由 Claude 撰寫、自己完全沒讀過的 PR？」幾乎一半的手舉了起來。這不是炫耀，而是警訊——技術地基正在快速移動，開發者必須重新學習如何站穩。

"Opus 4 is a dinosaur, almost a distant memory, and models are so much better." 「Opus 4 已成化石，幾乎是遙遠的記憶，而現在的模型遠遠更強。」

— Jeremy, Anthropic PM

Anthropic CEO Dario 已公開表示：Anthropic 絕大多數程式碼現在是由 Claude 撰寫，包括 Claude Code 自身的程式碼。我們正處在一個完全新的典範中——不是「AI 輔助寫程式」，而是「人類輔助 AI 寫程式」。

給初學者 · 一句話搞懂能力曲線
什麼是「能力曲線（The Capability Curve）」？

想像你每三個月收到一個更聰明的同事來幫你寫程式。這個同事不只變快，而是跳躍式地變聰明——從只能寫函式，到能規劃整個系統。能力曲線就是在描述這個加速提升的過程。

  • 一年前：Claude 能解決 60% 的標準 GitHub 問題
  • 現在：Claude 能解決 87%，且最新模型讓標準測試完全飽和
  • 對開發者的意義：你昨天認為 AI 做不到的事，今天可能已經做得到了

所以「駕馭曲線（Ride the Curve）」的意思是：不要用舊眼光限制 AI 的野心。

10 個核心洞見

01 SWE-bench 已被飽和：最前沿模型（Mythos Preview）讓業界最重要的軟體工程基準測試失去衡量意義——模型進步已超越了測試難度。
02 三大核心突破：自主規劃（Planning）、錯誤恢復（Error Recovery）、長文本專注力（Long-horizon Coherence）三者疊加，催生真正的自主智慧體。
03 告別 Doom Looping：模型遇到錯誤不再重複同樣失敗的嘗試，而是調用 Thinking Tokens 深度思考後換方向——這個困擾業界多年的問題已大體解決。
04 Bun Rust 改寫案例：Bun 創辦人（Anthropic 員工 Jared）不懂 Rust，讓 Claude Agent 自主運行一週，將整個引擎改寫為 Rust 並通過 100% 測試。
05 Evals 是競爭護城河：有完善評估體系的公司在新模型上線時可快速驗證收益；沒有的公司只能看 Twitter 猜測效果，慢上好幾週。
06 Frankenstein Prompt 是新型 Bug：模型越聰明，越會嚴格執行每條指令——過去為舊模型疊加的「科學怪人提示詞」會被新模型當成字面意思執行，製造新問題。
07 Auto Mode 分類器：不需要人類審核每一個工具調用——讓 AI 自行評估工具調用是否安全，只在關鍵變更時才引入人工確認。
08 閉環自我優化：讓 Claude Code 接入你的 Agent 系統，給它跑 Evals 的權限，它會自主改進 Prompt、工具設計——讓 AI 幫你優化 AI。
09 「Actually」時刻：新一代推理模型在思考過程中會自我糾錯，說「Actually」或「Never mind」後改變方向——這是 Test-Time Compute 的可觀察外在表現。
10 更有野心：不要在任務開始前就替 Claude 設限。以前做不到的長任務，現在交給 Claude 試試——你可能會被驚到。

給初學者 · 一句話搞懂飽和問題
為什麼「基準測試飽和」是個大問題？

想像你拿一張高中數學考卷來測大學數學教授——他會考 100 分，但你無法分辨他有多厲害。SWE-bench 現在對最前沿模型就是這個情況。

  • 問題：舊 Eval 滿分了，不代表新模型沒進步，代表測試太簡單了
  • 後果：公司誤判「新模型只進步 1%」，其實是 Eval 跟不上了
  • 解法：不斷「調高難度」——把真實失敗案例加進 Eval

Jeremy 提醒：必須隨模型演進不斷更新你的評估集，否則你的 Eval 就是在衡量昨天的問題。

II

第一支柱：自主規劃與推理

以前模型像衝動的工程師直接動手——不看說明書就組 IKEA 家具。現在模型會先閱讀、先規劃，然後才動手。

Jeremy 用 IKEA 家具比喻完美描述了舊模型的症狀：先行動後思考。Sonnet 3.7 會拿到任務就開始寫程式，遇到問題才回頭看 spec，通常已經走錯方向。

"You'll notice as the models reason through, they'll say things like 'actually' or 'never mind,' and change their approach as they're developing a plan." 「你會注意到當模型在推理時，它們會說『事實上…』或『算了先這樣』，並在制定計畫的過程中改變方法。」

— Jeremy
給初學者 · 一句話搞懂 Test-Time Compute
什麼是 Test-Time Compute（推理期計算）？

一般 AI 訓練完就固定了，但 Reasoning 模型可以在「回答你問題的那一刻」投入更多計算力思考。就像同一個人，時間充裕時能解更難的數學題。

  • 比喻：普通模型 = 限時考試；推理模型 = 開卷考試，你自己決定要想多久
  • 代價：需要更多 Token，也花更多時間
  • 適用場景：複雜的軟體工程、企業智慧體——值得投入更多思考

Effort Parameter 越高 = 模型花更多計算力 = 更難的任務越有可能成功。

新模型具備強大的 Test-Time Compute，在行動前會自主閱讀規格、制定高成功率計畫，甚至在內省規劃時會自我糾錯。這讓開發者不再需要用繁複的 Prompt 強迫模型「先想清楚再動手」。

Sonnet 3.7 舊模式

IKEA 衝動組裝法

拿到任務立刻寫程式。遇到問題才回頭讀 spec。浪費時間在方向錯誤的程式碼上。

Opus 4.7 新模式

資深工程師規劃法

先讀完所有 spec。制定計畫，自我審查（「Actually…」）。帶著高成功率計畫才開始執行。

給初學者 · 一句話搞懂 Adaptive Thinking
什麼是 Adaptive Thinking（自適應思考）？

讓模型自己決定要花多少時間思考——簡單問題快速回答，難問題深度推理。就像讓員工自行判斷一個任務值得花幾小時。

  • 不開啟：模型每次都用固定的思考深度，簡單任務浪費算力，難任務不夠用
  • 開啟：模型自動校準，用合適的計算力解決每個問題

對開發者的啟示：讓 Adaptive Thinking 開著，給模型空間自我校準。

II

第二支柱：錯誤恢復與告別死循環

Doom Looping——重複嘗試同樣失敗解法的死循環——曾是 AI Agent 最令人抓狂的問題。Jeremy 宣告：這個問題已大體解決。

給初學者 · 一句話搞懂死循環
什麼是 Doom Looping（死循環）？

想像你的同事遇到 bug，宣稱「我修好了！」，你一看 code——根本沒變。然後他再說一次「我修好了！」，還是沒變。這就是 Doom Looping。

  • 症狀：AI 說修好了 → 執行還是失敗 → AI 說修好了 → 執行還是失敗……
  • 代價：浪費大量 Token，卻無法突破
  • 舊的解法：開發者手動中斷，重新給指令
  • 現在的解法：模型遇到失敗會調用 Thinking Tokens，真正換方向

Jeremy 的說法：「我們已大體解決死循環問題。」這是 2025-2026 最重要的突破之一。

"We've sort of solved doom looping for the most part, because models are able to try a tool call, receive the results from the environment, use some thinking tokens, use some test-time compute to spend more computation and figure out how it should actually respond to that failure." 「我們已大體解決死循環問題——模型現在能夠嘗試工具調用、從環境接收回傳結果，然後使用思維 Token 與 Test-Time Compute 來投入更多計算，進而搞懂該如何真正應對那個錯誤。」

— Jeremy

模型收到工具回傳的錯誤後，會啟動 Thinking Tokens 進行深度思考，而不是重複同樣的解法。結果是：Token 浪費大幅降低，任務完成率顯著提升。

模型演進的見證

同一個任務，12 個月後的驚人差距

Jeremy 展示了一個實驗：給 Sonnet 4 和 Opus 4.7 相同的 Prompt——「一鍵重建整個 Claude.ai 網站」。

Sonnet 4：寫了 2,000 行程式碼，UI 陽春，聊天功能完全失效，對話無法產生回應。
Opus 4.7：只用 1,700 行更精簡的程式碼，完美還原側邊欄、對話歷史、Canvas、Mermaid 圖表，聊天正常運作——還加了深色模式。

Jeremy 笑說「它甚至加了深色模式，就像一個真正的工程師」。這不只是能力的進步，更是工程思維的進步——更少代碼、更完整功能、更主動的工程判斷。

II

第三支柱：百萬 Token 的持續專注

過去模型在處理大型代碼庫時會「迷失主線」。現在 Claude 可以在百萬 Token 的任務中保持高度前後連貫性。

一年前，如果你讓模型跨整個代碼庫做重構，消耗幾十萬 Token 後，它往往會忘記任務目標、遺漏重要細節，或無法遵循一開始的複雜規格。

現在，Long-horizon Coherence 讓模型在 100 萬 Token 以上的任務中保持主線。這意味著開發者不再需要把任務切成細碎的小塊，也不需要每 20 萬 Token 就介入提醒模型它在做什麼。

給初學者 · 一句話搞懂長文本連貫性
為什麼「迷失主線」是個嚴重問題？

想像你給一個工讀生一份 50 頁的工作說明，讓他做一個月的專案。半途他忘了第 12 頁的重要限制，結果整個方向錯了。這就是模型「迷失主線（Lose the Plot）」。

  • 以前的症狀：規格文件有 30 條規定，跑到一半只記得前 5 條
  • 現在的能力：規格文件全部都記得，100 萬 Token 跑完還沒忘
  • 對開發者的意義：可以把整個代碼庫一次給 Claude，讓它自己找相關的部分

Jeremy 的建議：「不要替 Claude 設限。把整個代碼庫給它，看它能做什麼。」

Long-horizon

三大支柱疊加的結果：真正意義上的長時序自主智慧體（Long-horizon Autonomous Agents）——能執行跨數小時、橫跨整個代碼庫的複雜任務，過去需要多人團隊幾週才能完成的工作。

III

長時序智慧體的工作方式

當規劃、錯誤恢復、長文本連貫性三者合一，Claude 可以「長跑」——數小時的端對端任務，只需要人類在關鍵節點做確認。

Jeremy 描述了長時序智慧體的典型循環：從計畫開始，執行，用環境驗證（跑測試），如果測試不通過則迭代，每隔一段時間對比目標——持續運行，不需要持續人工介入。

給初學者 · 一句話搞懂長時序智慧體
什麼是 Long-horizon Agent（長時序智慧體）？

普通 AI 助手就像叫外賣——你說要什麼，它給你什麼，一次性的。長時序智慧體更像一個自主工作幾天的工讀生——你給它目標，它自己規劃、執行、遇到問題自己解決，偶爾向你確認關鍵決定。

  • 舊模式：AI 做一步，你確認，AI 再做一步，你再確認……
  • 新模式：給 AI 一個目標 + 測試套件，讓它自主執行直到測試全通過
  • 關鍵要素：好的驗證環境（測試）+ 清楚的目標 + 允許模型自主運行

長時序智慧體讓一個人能完成過去需要一個小團隊才能做的專案。

III

傳奇案例：Bun 引擎一週 Rust 改寫

一個不懂 Rust 的工程師，讓 Claude Agent 自主運行一週，把整個 JavaScript 引擎改寫為 Rust，通過近 100% 的測試套件——然後合併了 PR。

長時序智慧體的極致展示

Jared 的一週冒險：不懂 Rust，卻讓 Bun 成為 Rust 引擎

Bun 的創辦人 Jared（現為 Anthropic 員工）有一天厭倦了 JavaScript 引擎的記憶體錯誤。他問自己：「如果我把整個 Bun 引擎改寫成 Rust，這些問題就消失了——但我不懂 Rust。」

"Because Bun has a great test suite with nearly 100% coverage of the entire engine, he was able to get Claude to run over the course of an entire week."
「因為 Bun 擁有近乎 100% 覆蓋率的完整測試套件，他能讓 Claude 持續運行整整一週。」

Jared 把 Claude Agent 接上 Bun 的測試套件，給了指令，然後讓 Agent 自主工作。Agent 不斷根據測試失敗的報錯修正程式碼，反覆迭代，在一週內將整個引擎改寫為 Rust 並通過測試——然後 Jared 合併了這個 PR。現在的 Bun 引擎是 Rust 寫的。

這個案例的啟示：好的測試套件是 AI 自主開發的燃料。沒有 100% 覆蓋率的測試，Jared 就無法信任 Claude 的工作。測試品質決定了 AI 能夠有多自主。

給初學者 · 一句話搞懂為什麼測試如此關鍵
為什麼測試套件是 AI 自主開發的關鍵？

沒有測試，AI 就像在黑暗中摸索——它不知道自己改對了沒有。有完整的測試套件，AI 就有了清晰的成功標準：「測試全過 = 任務完成。」

  • Jared 能讓 Claude 自主一週的前提：Bun 擁有近 100% 測試覆蓋率
  • 如果測試不完整：AI 可能「通過測試」但破壞了未被測試的功能
  • 教訓：想讓 AI 更自主，先讓你的測試更完整

Jeremy 說：「給 Claude 能驗證整個軟體系統的東西，它能做的事是不可思議的。」

來自真實客戶的驗證

Opus 4.7 的代碼質量躍升

IV

策略一：建立真正有用的 Evals

Evals（評估體系）是 AI 時代的單元測試——沒有 Evals 就像軟體工程師不寫測試，你根本不知道自己的應用是否正常運作。

"Evaluations are just the unit tests and the regression tests of the AI era." 「評估就是 AI 時代的單元測試與回歸測試。」

— Jeremy
給初學者 · 一句話搞懂 Evals
什麼是 AI Evals（評估體系）？

就像軟體有單元測試，AI 應用也需要「Evals」——一套測試題，評估 AI 在你的實際使用場景中表現得好不好。

  • Evals ≠ 學術基準（SWE-bench、MMLU）：那些是通用測試，不是你的場景
  • 好的 Evals = 來自真實用戶失敗案例 + 你實際關心的任務
  • 例子：如果你做金融 Agent，就收集用戶真正搞錯的財務問題來當測試

有 Evals 的公司在新模型上線時，一個腳本就知道新模型好不好用；沒有的公司只能靠感覺。這是競爭護城河。

三個 Evals 常見陷阱

❌ 用學術基準代替真實 Evals：SWE-bench、MMLU 是通用測試，無法反映你的應用場景。金融 Agent 應該用真實財務問題測試，不是程式碼問題。
❌ Eval 飽和後不更新：當新模型只進步「1%」時，可能是你的 Eval 太容易了，不是模型沒進步。必須持續「調高難度」。
❌ 因為「麻煩」就不做 Evals：很多團隊認為 Evals 是研究員的工作。Jeremy 反駁：就像開發者不會說「單元測試太麻煩了，我不寫」——開始就行了。
IV

策略二：精簡你的 Frankenstein Prompt

模型越聰明，越嚴格執行每條指令。過去為舊模型缺陷疊加的複雜提示詞，在新模型手中會變成 Bug 的溫床。

給初學者 · 一句話搞懂 Frankenstein Prompt
什麼是 Frankenstein Prompt（科學怪人提示詞）？

想像一份工作說明書，被不同的前任主管反覆修改了三年。每次 AI 出現問題，就加一條「千萬不要做 X」，結果變成一個充滿矛盾、前後不一致的龐大文件。新員工（新模型）看完這份說明書會完全搞混。

  • 症狀：Prompt 超過 1000 行，充滿「因為之前 Claude 會 XXX 所以加了這條」
  • 新問題：新模型會嚴格執行每一條，包括那些舊的、不再有效的指令
  • 案例：Jeremy 的 Claude.ai 引用格式 bug——舊格式範例留在 Prompt 裡，新模型嚴格照做，產出錯誤格式

解法：定期「大掃除」，把只為舊模型弱點存在的指令全部刪掉，精簡到真正的需求。

Claude.ai 引用格式 Bug 事件

當新模型太「聽話」反而製造問題

Jeremy 在準備 Claude 4 上線、調整 Claude.ai 應用程式時，發現一個奇怪的問題：輸出的引用格式全錯了。

"The model was following instructions more effectively than previous ones. And there were pieces of the Claude.ai prompt that it was following that we didn't even expect it to follow."
「模型執行指令的效率比以前更高，甚至去執行了某些我們壓根沒預期它會去執行的提示詞片段。」

原來 Prompt 裡有一個舊版引用格式的範例，這個格式早就廢棄了，但一直沒清掉。新模型太「聽話」，嚴格按照這個舊範例輸出，反而產出了錯誤格式。

修法：只是拿掉那個舊範例，微調幾個字。整類問題消失。教訓：Prompt 裡的每個字都是對模型的指令，不再有效的指令要刪掉。

"When you get a new model, you want to cut down your prompt, figure out what's not necessary anymore, and describe what you actually intend for your application." 「拿到新模型時，你要削減 Prompt，找出什麼已不再必要，描述你的應用真正的意圖——而不是堆疊應付舊模型弱點的 workaround。」

— Jeremy
IV

策略三：給模型空間工作

不要替 Claude 設限。允許它思考、允許它自主運行、允許它在你的系統中閉環優化自己。

I

開啟 Adaptive Thinking，調高 Effort

對複雜的軟體工程任務，將 Effort Parameter 調至最高。讓模型自己決定要花多少時間思考。不要追求「最快回應」——追求「最好結果」。

II

引入 Auto Mode 分類器

不需要人類審核每一個工具調用。建立類似 Claude Code 的 Auto Mode：AI 評估每個工具調用是否安全，一般操作自動放行，僅在危險變更時引入人工審查。

III

閉合智慧體循環（Close the Agent Loop）

讓 Claude Code 接入你的 Agent 系統，給它跑 Evals 的權限。它會自主讀懂你的代碼、運行測試、修改 Prompt——讓 AI 幫你優化 AI，達到自我提升的閉環。

給初學者 · 一句話搞懂 Auto Mode 分類器
什麼是 Auto Mode 分類器？

給每個工具調用打分：「這個操作安全嗎？」如果安全（如讀取文件），自動執行；如果危險（如刪除生產資料庫），才停下來問你。

  • 沒有 Auto Mode：每個操作都問你確認 → AI 每兩分鐘打擾你一次 → 你沒法去做別的事
  • 有 Auto Mode：安全操作自動過，危險操作才通知你 → 你可以完全放手讓它跑幾小時
  • Anthropic 內部：幾乎所有軟體工程師都在用 Auto Mode 讓 Claude 自主工作

你可以把 Auto Mode 的邏輯應用到你自己的應用中，實現更高程度的自主運行。

"If you can give Claude the ability to iterate on your own system, then you can sort of get to the point where you're almost self-improving." 「如果你能給 Claude 迭代你自己系統的能力，你就能達到近乎自我提升的境界。」

— Jeremy
給初學者 · 一句話搞懂閉合智慧體循環
什麼是 Close the Agent Loop（閉合智慧體循環）？

讓 AI 幫你優化 AI 的工作——就像讓一個工程師讀完你的所有測試報告，然後建議如何改進程式碼，而這個工程師本身就是 AI。

  • 第一層：Claude 幫你寫程式（普通 Copilot 模式）
  • 第二層：Claude 幫你改進讓 Claude 寫更好程式的環境（Meta 優化）
  • 實作：把 Claude Code 接入你的 Agent 系統 → 讓它跑 Evals → 讓它自己改 Prompt → 看分數提升

這是 AI 開發的終極型態：不只是使用 AI，而是讓 AI 自我演進。

策略 核心動作 競爭優勢

V

思考題 · 六個值得深想的問題

Jeremy 的演講挑戰了很多開發者的舊有假設。這六個問題沒有標準答案，但認真想過之後，你對 AI 開發的判斷力會顯著提升。

Q1 · Jeremy 說 SWE-bench Verified 已被「飽和」——最前沿模型讓這個最重要的基準失去意義。如果模型進步的速度永遠快於基準測試的更新速度，我們該怎麼持續衡量 AI 的真實能力？

查看建議思考方向
  • 歷史對照：圍棋的 Elo 系統最終也「飽和」了，解決方案是 AI 自弈（AlphaGo → AlphaZero）。AI 能力評估是否也需要「AI 對 AI」的動態基準？
  • 講者隱含論點：Jeremy 暗示基準飽和問題要靠「真實場景 Evals」而非更難的學術基準解決——因為學術基準永遠落後於真實需求。
  • 職涯啟發：如果你建立了一套優質的真實場景 Evals，你就擁有其他公司沒有的能力感知器——這是護城河。
  • 反向思考：也許「衡量能力」本身就是錯誤的框架——也許問題應該是「哪些任務已能自動化」，而不是「模型得了幾分」。

Q2 · Bun Rust 改寫案例之所以成功，前提是 Bun 已有近 100% 的測試覆蓋率。這意味著「測試品質」將成為 AI 自主開發能力的上限。你的代碼庫有多少測試覆蓋率？這個比例如何影響你現在能讓 Claude 做什麼？

查看建議思考方向
  • 歷史對照：Google 工程文化強調測試覆蓋率已超過 20 年。現在看來，這不只是工程品質問題，更是 AI 可自主化程度的基礎設施。
  • 講者隱含論點：Jeremy 說「給 Claude 能驗證整個系統的東西，它能做的事是不可思議的」——測試套件就是 AI 的感知系統。
  • 職涯啟發：提升測試覆蓋率不只是工程師的傳統美德，而是讓你能更好使用 AI 工具的直接投資——ROI 在 2026 年已大幅提升。
  • 反向思考：如果測試覆蓋率低的舊代碼庫難以讓 AI 自主操作，這是否會加速「重寫舊系統」的商業動機？舊技術債是否有新的清償方式？

Q3 · Jeremy 提出「Frankenstein Prompt」問題——模型越聰明，越嚴格執行每條指令，導致舊 Prompt 積累的 workaround 變成 Bug。你現在最重要的一個 Prompt 有多長？裡面有多少是為了修補以前模型的缺陷而加的？

查看建議思考方向
  • 量化參考：Jeremy 提到「3000 行」的 Prompt 並不罕見。如果你的 Prompt 超過 500 行，可能有相當比例已不再必要。
  • 講者隱含論點：Prompt 工程有「技術債」——每次「快速修補」就是一行不透明的指令。新模型上線是清償這種技術債的最佳時機。
  • 職涯啟發：能夠系統性審計並精簡 Prompt 的工程師，將在 AI 產品迭代速度上遠超只會「加指令」的工程師。這是一種需要訓練的能力。
  • 反向思考：「精簡」和「保留必要規則」之間的邊界在哪裡？如何用 Evals 驗證一條指令是否可以安全刪除？

Q4 · Jeremy 說「閉合智慧體循環」——讓 Claude 接入你的 Agent 系統，讓它自主改進 Prompt 和工具設計。這種「AI 優化 AI」的閉環有什麼潛在風險？應該設置哪些防護機制？

查看建議思考方向
  • 歷史對照：Google DeepMind 的 AlphaCode 已能自主生成程式碼並驗證。但「自我優化的系統」在 AI Safety 社群一直是重要研究課題——目標漂移（Reward Hacking）是已知風險。
  • 講者隱含論點：Jeremy 說要「在受控的方式下讓模型自主」——他預設了 Evals 是防護機制，但 Evals 本身也可能被「遊戲」（teaching to the test）。
  • 職涯啟發：AI 系統設計師需要同時思考「讓系統更強大」和「讓系統更可控」——這兩件事有時對立，找到平衡是核心能力。
  • 反向思考：完全閉合的自我優化循環在理論上是終極效率，但人類如何維持對系統行為的理解和監督？「可解釋性」在閉環系統中變得更還是更難？

Q5 · Jeremy 展示 Opus 4.7 重建 Claude.ai 網站比 Sonnet 4 用了更少程式碼（1700 vs 2000 行），卻做出更多功能。這說明了什麼？「程式碼行數」作為工程生產力的衡量指標，在 AI 時代還有意義嗎？

查看建議思考方向
  • 歷史對照：Bill Gates 曾說「衡量程式設計進度用程式碼行數，就像衡量飛機製造進度用重量」。AI 時代的「生產力」應該怎麼定義？
  • 講者隱含論點：Jeremy 特意提到 1700 < 2000 這個數字，強調「更少代碼 + 更多功能 = 更聰明的工程判斷」。新模型不只是快，而是有更好的抽象能力。
  • 職涯啟發：如果 AI 能生成更精簡的程式碼，人類工程師的價值是什麼？可能是「設計更清晰的需求」和「維護代碼庫的可測試性」——這兩件事都比「寫程式碼」更重要了。
  • 反向思考：更精簡的程式碼是否更難維護？AI 生成的「優雅」程式碼對人類的可讀性是否有影響？

Q6 · Jeremy 說「能力曲線不是關於任何單一模型，而是整個軌跡——地基在我們腳下持續移動」。這對你的開發策略有什麼意義？你現在的架構設計能否輕鬆換掉底層的 AI 模型？

查看建議思考方向
  • 歷史對照：智慧手機時代的 app 設計師：在 iPhone 4 時代為「低效能」設計，反而在 iPhone X 時代被自我限制。類比：今天為「不夠聰明的 AI」設計的 scaffolding，明天可能成為瓶頸。
  • 講者隱含論點：Jeremy 說「每隔幾個月模型就顯著更聰明」——這暗示了「模型無關性（Model-agnostic）架構設計」是未來的最佳實踐。
  • 職涯啟發：擁有強大 Evals 的公司可以在幾天內切換到更強模型，並立即驗證收益。沒有 Evals 的公司需要幾個月的人工測試。這是 2026 年最重要的工程能力差距之一。
  • 反向思考：如果模型每幾個月就大幅升級，過度投資「讓舊模型做得更好」的 Prompt Engineering 是否是錯誤策略？也許更值得投資的是「讓系統更容易換模型」？
給初學者 · 一句話搞懂 Effort Parameter
什麼是 Effort Parameter（努力參數）？

就像滑桿一樣，從 0（快速回應）到 1（最大思考）。你可以控制模型「願意花多少時間思考」。

  • Effort 低 = 快速但可能不夠深思熟慮（適合簡單問答）
  • Effort 高 = 慢但思考更深（適合複雜軟體工程、企業 Agent）
  • Jeremy 的建議：對智慧敏感型用例（軟體工程、企業決策），調到最高

代價：更多 Token 費用。但對於一個能在一週內改寫整個引擎的 Agent，這個代價通常是值得的。

給初學者 · 一句話搞懂 Scaffolding
什麼是 Scaffolding（腳手架）？

建築蓋好後要拆掉的外部支撐結構。在 AI 開發中，Scaffolding 是圍繞核心 LLM 的所有「輔助結構」——Prompt、工具、執行環境、額外指令——幫助舊模型做到它本來做不到的事。

  • 當模型更聰明：很多舊的 Scaffolding 就不再需要了
  • 問題：工程師常常忘了拆，Scaffolding 越加越多，變成 Frankenstein
  • 最佳實踐：每次模型升級，先試著把 Scaffolding 削到最少，再用 Evals 確認不影響表現

Jeremy 的核心訊息：模型越強，你需要的 Scaffolding 越少——這是進步的標誌，不是退步。
