I

核心論題：讓 Agent 不再失憶

每一次任務結束，記憶歸零。這是 AI Agent 的原罪——直到今天。

Anthropic API 知識團隊負責人 Ravi 在 Code with Claude London 2026 的演講中，正式發布了兩項重量級功能：Memory（記憶）與Dreaming（作夢）。這不是增量功能更新，而是 Agent 從「工具」走向「學習主體」的根本性轉變。

核心洞察在於：Anthropic 不再試圖在 Harness 層硬建記憶工具，而是將記憶建模為 Claude 的原生檔案系統——讓頂尖模型（如 Opus 4.7）用其本已精通的虛擬環境導航與 Bash 能力，自主決定「未來的自己需要讀什麼筆記」。

十條核心洞見

IAgent 任務時序每 7 個月翻倍（Meter 2025）——長週期 Context 管理是當前最緊迫的技術瓶頸。
IIMemory 突破「每次從同一起跑點出發」的根本限制，讓 Agent 可從常見策略、錯誤、工具使用中持續學習。
IIIMemory 建模為檔案系統，不是 Key-Value 靜態存取——Claude 自主決定結構與呈現方式（"Let it cook"）。
IV多 Agent 記憶分兩層：組織級唯讀（高穩定、所有人共享）+ 任務級讀寫（細粒度、動態更新）。
V樂觀並行控制（OCC）解決多 Agent 同時寫入衝突——比悲觀鎖定更適合超長時序任務。
VI企業控制 API：版本 Diff、寫入歸屬、去識別化、審計追蹤——記憶可管、可審、可匯出。
VIIDreaming = 帶外非同步精煉迴路，解決多 Agent 場景下「各自為政、局部最佳化」的根本問題。
VIIIDreaming 三大優勢：識別全域模式、清晰目標分離、零熱路徑延遲——三者缺一不可。
IXDreaming 本身建構在 Cloud Managed Agents 之上，啟動子 Agent 陣列並行分析逐字稿——以 Agent 治理 Agent。
X記憶系統是解鎖 Agent 走向「跨日、跨週超長時序運行」的核心關鍵——這是 2026 年最重要的 AI 基礎建設。

II

記憶作為檔案系統

不是資料庫，不是 Key-Value Store——Memory 是 Agent 的個人工作目錄。

給初學者 · 一句話搞懂 Memory
Agent 為什麼會「失憶」？

想像你每天上班前都要把所有記憶清空——你昨天的客戶叫什麼名字？上週遇到的 Bug 怎麼解？你完全不知道。這就是傳統 AI Agent 的處境：每次啟動都從空白開始。

  • 沒有 Memory：Agent A 和 Agent B 各自踩同一個坑
  • 有了 Memory：Agent A 解決了問題 → 寫到「日記」→ Agent B 讀到 → 直接避開

Memory 就是讓 Agent 有辦法寫日記、讀日記的基礎設施。

"But one pattern we're seeing is that as models improve we really just want to get out of Claude's way... Opus 4.7 is state-of-the-art at file system based memory and it's increasingly capable of discerning which context is most important to save for its future self."

「隨著模型進步，我們只想讓開路讓 Claude 自己來……Opus 4.7 在基於檔案系統的記憶方面已達業界頂尖，它越來越有能力判斷哪些 Context 對未來的自己最重要。」 — Ravi, Anthropic

這段話揭示了一個深刻的設計哲學轉變：與其在框架層設計精細的記憶工具，不如信任模型本身的能力，讓它用早已精通的檔案操作技能自由組織記憶。Anthropic 稱之為「Let it cook」——放手讓它發揮。

給初學者 · 一句話搞懂「檔案系統記憶」
為什麼是「檔案系統」而不是「資料庫」？

傳統記憶系統像字典查表：你給 Key，它給 Value，結構固定死板。Claude 的記憶是一個硬碟：你可以建資料夾、寫 Markdown、建索引檔、用 Bash 搜尋——完全靈活。

  • 資料庫記憶：search("錯誤訊息") → 固定格式回傳
  • 檔案系統記憶：Claude 自己決定要建 /errors/2026-05/ 目錄，每個錯誤一個 .md 檔

因為 Claude 天生就擅長 Bash 和虛擬環境導航，這個設計讓模型能力完全發揮。

記憶架構三層

Anthropic 的 Memory 系統由三個核心組成部分構成：

Memory 架構三層
第一層 · 儲存層（Storage Layer） 數據如何被管理、變更如何被追蹤——版本控制、Diff、歸屬標記（Attribution）
第二層 · 結構層（Structure） 記憶格式優化成讓 Claude 最大功效發揮的形式——檔案系統建模，Agent 自主組織
第三層 · 雲端處理層（Cloud-driven Processing） 記憶的更新機制——Agent 工作時即時寫入（局部），以及 Dreaming 帶外精煉（全域）
給初學者 · 一句話搞懂「儲存層」
版本控制：記憶的「Git」

如果 Agent 寫錯了什麼，怎麼辦？Memory 的儲存層就像 Git——每次變更都有紀錄，可以回溯、可以比對（Diff）、可以查誰寫的（Attribution）。

  • 版本歷史：看記憶如何隨時間演變
  • Diff 比對：哪個版本改了什麼
  • 歸屬標記：是 Agent A 還是 B 寫的？

這讓 Memory 在企業環境中可稽核、可追責。

多 Agent 記憶架構

記憶層級架構：從組織到任務
組織級記憶 · 唯讀
  • SLO 政策、Runbooks
  • On-call 人員映射
  • 組織標準與最佳實踐
  • 更新頻率：極低 · 所有 Agent 可讀

  • 發現的錯誤與解法
  • 跨 Session 協調狀態
  • 更新頻率：頻繁 · 有寫入控制
給初學者 · 一句話搞懂「樂觀並行控制」
OCC：不搶鎖，但提交前先確認

假設 Agent A 和 Agent B 同時要更新同一個記憶檔案。悲觀鎖定是「一人用，其他人等」——效率極差。樂觀並行控制（OCC）是「都先改，提交時比對版本號」。

  • Agent A 讀取記憶 v1.0，修改後準備提交
  • Agent B 也讀了 v1.0，比 A 先提交，記憶變 v1.1
  • A 提交時發現版本不符 → 觸發衝突解決機制

這避免了長時間任務被「鎖住」等待，又確保數據一致性。超長時序任務的必備設計。

真實案例 · Rakuten 首次錯誤率

樂天：從每次重蹈覆轍到一次學習、永久記住

日本電商巨頭 Rakuten（樂天）在生產環境中部署了 Cloud Managed Agents，並開啟了 Memory 功能。在舊架構下，Agent 每次啟動都無法記住上次踩過的坑——常見的 API 呼叫錯誤、系統特定的行為模式，每次都要重新「摸索」。

Memory 讓 Agent 能累積領域特定的工具使用策略與系統知識，每次任務都在前次基礎上前進，而不是原地踏步。

結果：生產環境 Agent 的首次嘗試錯誤率顯著降低。每個 Agent 都站在前人的肩膀上——哪怕「前人」就是昨天的自己。

企業級控制 API

Memory 不只給 Agent 用——它有完整的獨立 API，讓開發團隊從任何地方管理：

給初學者 · 一句話搞懂「去識別化」
Redactions：讓 Agent 記憶符合法規

企業 Agent 在處理任務時，記憶中可能殘留用戶個資（姓名、信用卡號）。「去識別化」讓系統能自動偵測並遮蔽這些敏感內容——記住「有一個 VIP 客戶抱怨付款問題」，但不記住那個客戶的真實姓名。

  • GDPR / CCPA 合規的必備功能
  • 讓 Agent 記憶可以長期保存、跨部門共享

企業要把 Memory 用在生產環境，這是繞不開的門檻。

III

局部最佳化的困境

Agent 在任務中即時記筆記——這個模式在單一 Agent 時運作良好。多 Agent 並行時，問題出現了。

Ravi 坦承，當記憶架構擴展到複雜的多 Agent 場景時，他們觀察到一些令人沮喪的模式：

重複犯錯

不同 Session 各自獨立踩同一個坑，無法從彼此的失敗中學習

效率低下模式

多個 Agent 表現出相同的次優行為，沒有機制把「最佳解」傳播出去

碎片化記憶

每個 Session 局部最佳化地寫入，導致重複、矛盾、難以查找的記憶碎片

給初學者 · 一句話搞懂「局部最佳化」
為什麼「當下最好」不等於「整體最好」？

想像 20 個工程師在同一個 Wiki 上各自寫筆記：每個人都在自己的筆記裡寫得很好，但全局的 Wiki 變成了重複的垃圾堆——同一個問題被解決了 15 次，每次解法略有不同，讀的人不知道看哪個。

  • 局部最佳化：我這篇文章寫得很清楚 ✓
  • 全局問題：重複內容、矛盾資訊、查不到想要的 ✗

多 Agent 記憶面臨的正是這個「分散式 Wiki 問題」。Dreaming 就是那個會定期整理 Wiki 的管理員。

"The general theme was memory was being updated in a locally optimal way but it wasn't globally optimal... each dreaming run analyzes session transcripts, it inspects the existing state of memory and it proposes optimizations to the memory."

「整體的問題是：記憶以局部最佳化的方式在更新，但它在全局上並不是最佳的……每次 Dreaming 運行都會分析 Session 逐字稿，檢查記憶的現有狀態，並提出最佳化方案。」 — Ravi, Anthropic

Dreaming：帶外精煉迴路

Dreaming 架構圖：帶外非同步精煉

為什麼必須「解耦」？

Dreaming 與 Agent 主迴圈完全分離——Agent 只管做任務，Dreaming 只管精煉記憶，互不干擾。

滑鼠移到圖上，看精煉迴路轉動

Dreaming 的三大結構性優勢

①識別全域模式：單一 Agent 在孤立的 Session 中無法察覺「CPU 飆升 60 秒後必發特定告警」這種跨越數天、多環境的統計規律。Dreaming 批次分析歷史 Session，站在全局視角去碎片化。
②清晰的目標分離：如果讓運行中的 Agent 同時負責「精煉記憶」，它被迫在「把任務做好」與「把記憶寫好」之間做 Token 預算的權衡。Dreaming 獨立出去，讓任務 Agent 目標無污染。
③零延遲：Dreaming 是後台進程（可設定每晚、每小時或事件觸發），完全從 Agent 熱路徑移除，不增加任何線上延遲。
給初學者 · 一句話搞懂 Dreaming
為什麼叫「作夢」？

人類的大腦在睡眠時會整理白天的記憶——去掉不重要的，加強重要的，把零散的片段整合成結構化的知識。這就是「Dreaming」的設計靈感。

  • 白天（工作時）：Agent 即時寫入原始筆記
  • 夜晚（帶外）：Dreaming 分析所有筆記，精煉出更好的記憶
  • 隔天（下次任務）：Agent 用的是更高質量的記憶

人類需要睡眠才能「消化」知識，AI Agent 需要 Dreaming 才能做到同樣的事。

Dreaming 與 Test-time Compute 的類比

"With dreaming, agents are doing the same thing. They're spending some work up front to curate and produce higher quality memory and... pays dividends for all downstream agent performance... dreaming is the bridge between memory as we know it today and organization scale memory and knowledge."

「透過 Dreaming，Agent 也在做同樣的事：預先投入一些工作來整理並產生更高質量的記憶，而這會為所有下游 Agent 的效能帶來回報……Dreaming 是我們今天所知的記憶與組織規模知識之間的橋樑。」 — Ravi, Anthropic
給初學者 · 一句話搞懂 Test-time Compute
預先投入計算，換取更好的結果

Test-time Compute（TTC）是讓模型在回答問題時，花更多 Token 「思考」——探索多條路徑、自我糾錯——從而輸出更好的答案。Dreaming 把這個概念擴展到記憶層：預先投入計算整理記憶，未來所有 Agent 都受惠。

  • 傳統 TTC：我花更多 Token 思考這個問題 → 這次回答更好
  • Dreaming TTC：我花計算整理過去的記憶 → 未來所有回答都更好

這就是從「個人學習」到「組織知識積累」的關鍵躍遷。

里程碑案例 · Harvey 法律 AI

Harvey 的 6 倍突破：記憶讓法律 AI 不再「每次重讀案例」

Harvey 是全球頂尖的 AI 法律平台，服務對象包括大型律所和企業法務部門。在接入 Dreaming 之前，每個 Agent Session 都需要從頭理解相關法律框架、先前案例的判決思路和客戶偏好風格——即使這些都是重複的工作。

"法律工作的核心是知識積累——你做過一個複雜的 M&A 交易，下次看到類似結構就應該更快。但 AI Agent 以前做不到這點。"

接入 Memory 與 Dreaming 後，Harvey 在自家的法律基準測試中看到了 6 倍的任務完成率提升。Dreaming 在跨案件、跨 Session 中識別出重複的法律推理模式，把最佳實踐固化到共享記憶中，每個新 Agent 都站在巨人肩膀上出發。

給初學者 · 一句話搞懂「帶外處理」
Out-of-band：後台的管理員

「帶外」（Out-of-band）的意思是：這個程序不在主要工作流程的路徑上執行，它是一個獨立的「背景任務」。

  • In-band（帶內）：Agent 正在做任務時，同時整理記憶 → 分心、延遲增加
  • Out-of-band（帶外）：Agent 完成任務後，Dreaming 在後台靜悄悄地整理 → 零干擾

就像圖書館員不在讀者借書時整理書架，而是閉館後統一整理——讀者永遠看到整潔的書架，卻感受不到整理的過程。

給初學者 · 一句話搞懂 SRE
SRE 輪值：程式設計師的消防員

SRE（Site Reliability Engineering）的核心工作之一是「輪值（On-call）」——系統出問題時，輪到你的工程師要半夜被叫醒處理。AI Agent 正在接手這份工作：

  • 以前：告警 → 叫醒工程師 → 工程師查 Runbook → 手動修復
  • 現在：告警 → Agent 查記憶 + 分析 → 自動修復 → 更新記憶

但如果 Agent 每次都從空白記憶出發，它就跟一個「每天都第一天上班」的工程師一樣。Memory 解決了這個問題。

設計洞察 · Dreaming 吃自己的狗糧

Dreaming 本身建構在 Managed Agents 上

一個讓 Ravi 特別強調的細節：Dreaming 不是用特殊的底層系統實作的，它本身就是一個 Cloud Managed Agent。

"Dreaming 本身就是構建在 Cloud Managed Agents 之上的。所以它是為 Cloud Managed Agents 打造、且本身也基於 Cloud Managed Agents 的功能。"

這意味著開發者可以像觀察任何其他 Managed Agent 一樣觀察 Dreaming 的運行——完全相同的 UX、完全相同的可觀測性工具。Anthropic 用自己的技術解決自己的問題，這是「狗糧測試」通過的最好證明。更重要的是：這說明 Managed Agents 的抽象層已經足夠強大，連 Anthropic 自己的核心功能都建構在上面。

給初學者 · 一句話搞懂「審計追蹤」
Audit Trail：記憶的「監視錄影」

企業部署 AI Agent 時，最怕的問題是：「這個錯誤決定是誰做的？」Memory 的版本控制讓每次變更都有完整紀錄：

  • 時間戳：何時寫入
  • 歸屬（Attribution）：哪個 Agent Session 寫的
  • Diff：改了什麼

當出現問題時，可以精確定位「是哪個 Agent 在什麼時候寫入了什麼錯誤的記憶」——讓 AI 系統在企業中真正具備可問責性。

Wise Docs 跨 Session 驗證

真實案例 · Wise 文件驗證

Wise：讓 Agent 記住「這個文件格式我見過」

國際匯款平台 Wise 在其文件驗證流程中部署了 Managed Agents，面臨一個典型問題：相同類型的文件（如某國的銀行對帳單）會有特定的格式問題和驗證邏輯，但每個 Agent Session 都要重新「摸索」這些規律。

Memory 讓 Agent 能夠記住跨 Session 的常見問題，避免在每次新的文件驗證工作流中重複相同的錯誤。

接入 Memory 後，Wise 的文件驗證流程中跨 Session 的常見問題顯著減少——Agent 學習了「某類文件通常缺失第 3 頁」、「某國銀行帳單的簽章位置不標準」等知識，並在下次看到類似文件時直接應用，不再從零偵測。

給初學者 · 一句話搞懂「跨 Session 記憶」
Session 是什麼？跨 Session 是什麼？

一個 Session 是 Agent 的一次「工作週期」——從啟動到完成任務。就像你開啟一個新的無痕視窗，所有狀態從零開始。

  • 沒有 Memory：Session 1 解決了問題，Session 2 又從頭解決一遍
  • 有 Memory：Session 1 把解決方法寫入記憶，Session 2 直接讀取 → 跳過重複工作

「跨 Session」記憶就是讓這些獨立的工作週期能夠互相傳遞知識，形成一個持續學習的智慧體。

六道思考題

從 Ravi 的演講出發，深入思考 Memory 與 Dreaming 對 AI 開發、產品設計與組織知識管理的長期影響。

Q1 · Ravi 強調「Let it cook」——信任模型自主管理記憶，而不是設計精細的記憶工具。這個哲學有什麼風險？在哪些場景下「退出 Claude 的通道」是不明智的？

查看建議思考方向
  • 歷史對照：對比「工具設計 vs 信任模型」在不同 AI 能力層次的適用性——GPT-3 時代為什麼需要精細工具，Opus 4.7 時代為什麼可以放手？能力閾值在哪裡？
  • 講者隱含論點：Ravi 的論述暗示「模型能力提升比工具工程更可靠」，但這對 Agent 系統的長期可維護性有何影響？當模型更新換代時，記憶結構是否需要遷移？
  • 職涯啟發：如果 AI 系統越來越傾向於「信任模型而非設計工具」，傳統的「AI 工程師」角色如何演變？什麼技能會更有價值？
  • 反向思考：「讓它自由發揮」的邊界在哪裡？企業合規、安全審計、可預測性——這些需求是否天然與「讓 Claude 自主決定記憶結構」相衝突？

Q2 · Dreaming 的「帶外非同步精煉」解決了局部最佳化問題，但引入了新的延遲——記憶只在 Dreaming 運行後才會改善。在高頻率、時效敏感的業務場景（如金融交易監控、即時客服），這個延遲能接受嗎？

查看建議思考方向
  • 量化分析：Dreaming 可觸發頻率（每小時 → 每 5 分鐘？）與計算成本之間如何權衡？有無「即時 Dreaming」的技術可能性，其代價是什麼？
  • 講者隱含論點：Ravi 選擇「帶外處理」而非「即時精煉」，背後的根本原因是什麼？這是技術限制，還是系統設計選擇？
  • 職涯啟發：不同行業對「記憶更新延遲」的容忍度不同，你所在的領域處於哪個區間？如何根據業務需求調整 Dreaming 頻率策略？
  • 反向思考：「接受延遲換取全局最佳化」vs「犧牲全局最佳化換取即時性」——哪種架構的長期技術債更高？

Q3 · 「Agent Swarm 共同維護組織知識」的願景令人興奮，但也暗藏風險：如果 Agent 集體形成了某個錯誤的「共識記憶」（例如一個錯誤的最佳實踐被多個 Agent 強化），如何被發現和糾正？

查看建議思考方向
  • 歷史對照：對比人類組織中「制度性錯誤知識」的形成與修正——公司手冊上的過時流程、行業的「最佳實踐」被新研究推翻的案例。AI 記憶系統有無天然的「自我糾錯」機制？
  • 講者隱含論點：Ravi 提到 Dreaming 輸出是「供 Agent 選擇性採納的快照」，這個「選擇性」是如何實作的？誰有否決權？
  • 職涯啟發：設計 Memory 系統時，如何建立「知識品質評估」機制？是否需要人類在迴路中定期審計 Agent 的共享記憶？這是新的工作崗位嗎？
  • 反向思考：「版本控制 + 歸屬標記」能解決這個問題嗎？還是需要主動的「知識健康度監控」系統？

Q4 · Ravi 說「Dreaming 是從今天所知的記憶到組織規模知識的橋樑」，但現有公司都有自己的知識管理系統（Confluence、Notion、Wiki）。Memory 系統如何與這些現有工具共存或競爭？

查看建議思考方向
  • 歷史對照：電子郵件、Wiki、Slack——每個新的知識工具都說自己會取代前者，卻都共存至今。AI 記憶系統為何會（或不會）不同？
  • 講者隱含論點：Memory 是 Agent 自主生成和維護的知識，傳統 Wiki 是人類寫的——這個差異是否意味著它們服務不同的使用者群體而非競爭？
  • 職涯啟發：對於知識工作者，「協助 AI Agent 維護和審計共享記憶」是否會成為一種新的工作形態？如何為此準備？
  • 反向思考：如果 Memory 最終取代了人類維護的 Wiki，組織失去了什麼？知識創造過程中人類的「詮釋」角色是否無可替代？

Q5 · 樂觀並行控制（OCC）在多 Agent 高並發寫入時，「衝突重試」的成本是多少？在極端場景（數百個 Agent 同時更新同一記憶文件），OCC 是否仍然是正確的選擇？

查看建議思考方向
  • 量化分析：OCC 的衝突率受哪些因素影響（並發數、寫入頻率、記憶文件大小）？和悲觀鎖定比較，在什麼並發規模下 OCC 開始失效？
  • 講者隱含論點：Anthropic 選擇 OCC 而非悲觀鎖定，主要論據是「超長時序任務不能等」——但 Dreaming 的帶外架構是否能降低即時寫入衝突的概率？
  • 職涯啟發：分散式系統設計（CAP 定理、Eventual Consistency）的知識在 AI Agent 時代變得更重要——如何將傳統分散式系統思維應用到 Agent 架構設計中？
  • 反向思考：「記憶的一致性」和「記憶的可用性」之間存在根本張力（CAP 定理）——Anthropic 的設計選擇了哪一邊？這對不同業務場景是否合適？

Q6 · Ravi 預測 2026 年將會看到 Agent 運行跨日、跨週的超長時序任務，而 Memory 是關鍵。但跨週任務意味著 Agent 的「目標」可能在過程中需要調整——Memory 如何支持「目標本身的演化」，而不只是「知識的積累」？

查看建議思考方向
  • 歷史對照：人類長期項目（如建設一座城市、執行 10 年戰略）如何在保持核心目標的同時調整子目標——這是否能類比到 Agent 記憶設計？
  • 講者隱含論點：Ravi 的框架主要關注「知識積累型記憶」（學習策略、避免錯誤），但「目標演化型記憶」（這個任務的優先級變了）需要不同的設計嗎？
  • 職涯啟發：如果 Agent 能跨週工作，它的「工作記憶」和「長期目標記憶」需要如何分層？這對你目前的 AI 系統設計有何啟發？
  • 反向思考：「目標不變、知識積累」是 Anthropic Memory 的隱含假設嗎？如果這個假設不成立（目標也需要更新），現有架構是否需要根本性重設計？
給初學者 · 名詞速查表
本場演講高頻術語
  • Memory：讓 Agent 跨任務保留學習成果的持久化記憶系統，建模為檔案系統
  • Dreaming：帶外非同步精煉迴路，批次分析跨 Session 逐字稿，全域最佳化記憶
  • OCC（樂觀並行控制）：允許多個 Agent 自由讀寫，提交時比對版本號解決衝突
  • Out-of-band（帶外）：不在主要 Agent 工作流路徑上執行的獨立後台進程
  • Hot path（熱路徑）：系統中最頻繁執行、對延遲最敏感的核心程式碼路徑
  • Test-time Compute（TTC）：推理期投入額外計算換取更好輸出的技術，Dreaming 是其記憶層延伸
  • Attribution（歸屬）：記錄哪個 Agent Session 寫入了哪部分記憶的標記機制
  • Redactions（去識別化）：遮蔽記憶中敏感個資，確保隱私合規
