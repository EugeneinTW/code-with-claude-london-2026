I

HeroCorp 的成本危機

每次啟動 HeroCorp 的 AI 儀表板，費用高達 31 英鎊——整合了 Web、Slack、Gong、Jira 的海量數據，全部塞給 Opus 4.7。這樣的 Agent 在商業上根本無法存活。

Punit Sha 一開場就問了一個殘酷的問題：「誰已經把自己的 Agent 推向生產環境，而且對品質、成本、速度都完全滿意？」現場寥寥幾隻手。這就是問題所在。

大多數開發者能讓 Agent 「工作」，但無法讓它「有利潤地工作」。本場演講的三大技術正是為此而生：不改變 Agent 的智能輸出，卻讓運行成本降至十分之一。

II

Prompt Caching：90% 折扣的商業利器

每次呼叫 Claude API 時，如果系統 Prompt 或工具定義沒有改變，為什麼要重新處理它們？快取正是為了解決這個問題——預處理一次，重複使用，把節省的計算成本還給你。

"If you remember nothing else from this session, think about prompt caching. You get a 90% discount because we're not re-processing them. We pass those savings on to you." 「如果你從這場演講只記住一件事，請記住 Prompt Caching。你拿到 90% 的折扣，因為我們沒有重新處理它們。我們把節省的成本轉讓給你。」

— Punit Sha
給初學者 · 一句話搞懂 Prompt Caching
什麼是 Prompt Caching（提示詞快取）？

想像每次打電話給客服，都要重新報一遍你的帳號、地址、過去的所有問題——太浪費時間了。Prompt Caching 就是讓系統記住這些固定資訊，每次只需傳遞「這次新增的部分」。

  • 沒有快取：每次呼叫都傳送完整 System Prompt → 全部重新處理 → 費用 × N 次
  • 有快取：固定部分預處理一次，新增部分才計費 → 省下 90% 輸入 Token 費用
  • 額外紅利：快取 Token 不計入速率限制，等效提升 5 倍吞吐量

Punit 說：「這是生產級 Agent 不能不用的基礎設施。」

快取命中率的三大效益

效益 1

90% 成本折扣

快取命中的輸入 Token 不需重新處理。Anthropic 把這份節省 pass through 給開發者——對 Token 密集的 Agent 而言，這幾乎等於降低 90% 的 API 費用。

效益 2

5× 速率限制倍增

快取命中的 Token 不計入速率限制。若你的 Agent 有 80% 快取命中率，你的實際可用速率等效提升 5 倍——對高流量生產系統是決定性的。

效益 3

降低首字延遲（TTFT）

快取命中不需要模型重新讀取長 Prompt，回應速度更快。對即時互動的 Agent 應用，體感速度顯著提升。

頂尖客戶標準

Replit / Cursor / Perplexity：90%+

這三家知名 AI 工具公司的快取命中率都超過 90%。這是工程設計的結果：他們刻意讓系統 Prompt 的動態部分降到最低。

HeroCorp Demo 實測

一行程式碼，成本砍半

HeroCorp 儀表板在加入 Prompt Caching 之前：快取命中率 0%，每次加載費用 £31。

加入快取代碼後，輸出結果完全相同，但快取命中率提升至 58%，費用直接降到約 £15——接近砍半。

Punit 強調：輸出完全沒有改變，只是不再重複支付處理同一份系統 Prompt 的費用。這是最直接的「免費午餐」之一。

給初學者 · 快取失效的隱藏陷阱
什麼會讓快取失效（Cache Miss）？

快取像一把鑰匙——如果 Prompt 的任何一個字改變了，就換了一把新鑰匙，快取完全失效，相當於沒有快取。

  • 時間戳記：System Prompt 裡加入「現在時間：2026-05-20 14:32:01」→ 每秒都是新鑰匙 → 快取永遠 0%
  • 隨機 ID：每次請求加入 UUID → 每次都是新鑰匙
  • 動態用戶資料：把用戶名字塞進系統 Prompt → 每個用戶不同鑰匙
  • 解法：把動態資料移到 User Turn，保持 System Prompt 靜態

Anthropic Console 現已提供分析工具，能幫你找出是什麼東西導致快取失效。

III

Context Engineering：把垃圾擋在脈絡之外

擁有 100 萬 Token 的上下文窗口不是塞越多越好——把不相關的資訊塞進去，模型效果反而更差、費用更高。Context Engineering 是一門「精準放入」的藝術。

"Because you're putting less gunk into context, you're putting more relevant stuff there only, it turns out the models actually perform better." 「因為你放進脈絡的垃圾更少、只放相關資訊，結果發現模型表現反而更好。」

— Punit Sha
給初學者 · 一句話搞懂 Context Engineering
什麼是 Context Engineering（上下文工程）？

就像給工人下指令：你可以把整個倉庫的資料都搬到他面前（費錢、讓他分心），或只給他當前任務需要的那份文件（省錢、更專注）。Context Engineering 就是讓 AI 只看到「它現在真正需要的東西」。

  • 問題：Agent 整合了 Web、Slack、Gong、Jira 的海量資料，全部塞進去？
  • 後果：Token 消耗爆炸、費用飛漲、模型被噪音干擾、效果更差
  • 解法三招：Tool Search（按需載入）、Programmatic Filtering（API 回傳先過濾）、Compaction（定期清空舊脈絡）

Context Engineering 的目標：讓脈絡窗口裡只有「高信噪比」的內容。

🔍
減少 10% Token · Lovable 驗證

Tool Search（工具搜尋）

不預先載入幾十個工具的完整定義，而是給模型一個「搜尋工具定義的工具」。模型有需要時才動態載入對應工具的 Schema——只有當前任務需要的工具才進入脈絡。

🐍
精簡 API 回傳 · Quora 驗證

Programmatic Tool Calling（程式化工具調用）

讓 Claude 寫輕量 Python 腳本，先過濾外部 API 回傳的雜訊再送給模型。Gong 60 分鐘的電話錄音？Python 腳本先壓縮成情緒摘要，再交給 Claude。不是整份文字，是核心精華。

🗜️
無限對話體感 · Hex 驗證

Compaction（上下文壓縮）

設定閾值（如 40 萬 Token）。當對話接近上限時，引導模型自動摘要關鍵事實，清空舊對話歷史，重置為精簡版本。對話連續性保留，脈絡費用大幅降低。

給初學者 · 一句話搞懂 Tool Search
為什麼要用 Tool Search 動態載入工具定義？

想像一個秘書有 50 個工具手冊。如果你每次都把 50 本手冊全堆在他桌上，他會搞混。如果只在他需要某個工具時才遞給他那本手冊，他更專注、更準確。

  • 舊方法：把 50 個工具的 JSON Schema 全部塞進 System Prompt（大量 Token）
  • 新方法：給一個「搜尋工具定義」的工具 → 模型說「我需要 Gong 工具」→ 動態載入 Gong 的 Schema
  • 效果：Lovable 減少 10% Token 消耗，精準度也同步提升

更少 Token + 更高精準度 = 雙重紅利。這就是 Context Engineering 的魔法。

HeroCorp 上下文工程 Demo

Gong 錄音壓縮：從一小時的文字到情緒摘要

HeroCorp 儀表板整合了 Gong 客戶通話資料，每次加載時要把 30-60 分鐘的逐字稿塞進脈絡——這本身就是一個巨大的 Token 消耗。

「透過 Python 腳本過濾，將原本龐大的文字區塊壓縮成精簡的情緒摘要，使單次加載成本進一步降至 £11。」

做法：讓 Claude 生成一個輕量 Python 腳本，在 Gong API 的文字回傳到模型之前，先把它轉換為情緒指標（正面/負面/中性 + 關鍵時刻）。Claude 接收到的是 100 字的精華，不是 6000 字的原文。

同時，加入 Compaction 機制：當儀表板的上下文窗口增長到 40 萬 Token 的閾值，自動觸發摘要並清空舊歷史。操作者可以在側邊欄看到上下文增長速度明顯放緩，以及在閾值時的自動回落。

給初學者 · 一句話搞懂 Compaction
什麼是 Compaction（上下文壓縮）？

想像一本筆記本快滿了，你不是扔掉它，而是把重要內容用一頁摘要整理好，再繼續記新東西。Compaction 就是讓 AI 自動做這件事。

  • 沒有 Compaction：對話太長 → 超出上下文限制 → 對話強制中斷 → 用戶體驗崩潰
  • 有 Compaction：接近閾值時 → 自動摘要歷史 → 清空舊脈絡 → 對話無縫繼續
  • Hex 的體驗：加入 Compaction 後，他們的代碼也大幅簡化了（不需要手動管理上下文）

Compaction 讓有狀態的長期 Agent 成為可能，而不只是一次性的短對話。

IV

Advisor Strategy：資深帶菜鳥的模型路由

成本降到 £11 之後，對於低利潤業務仍然偏高。根本原因是整個 Agent 都在使用 Opus 4.7。解決方案是讓 Sonnet 做 80% 的日常工作，Opus 只在「奇怪案例」出現時充當顧問。

"There are senior engineers paired with junior engineers that make that junior engineer so much more powerful. Pairing the kind of Sonnet/Haiku gets you kind of the prices of those models approaching even sometimes Opus intelligence." 「就像資深工程師搭配菜鳥工程師，讓菜鳥的能力大幅提升。Sonnet/Haiku 搭配 Opus 顧問，能以接近 Sonnet 的費用，逼近 Opus 的智能。」

— Punit Sha
給初學者 · 一句話搞懂 Advisor Strategy
什麼是 Advisor Strategy（顧問策略）？

就像一個公司請了一位頂級顧問（很貴），但不是讓他做所有的日常工作——他只在出現複雜、高風險的決策時才被請出來。日常工作由普通員工（便宜）負責。

  • 全 Opus 4.7：頂尖智能，但費用極高，利潤薄到無法商業化
  • 全 Sonnet 4.6：費用低，但在關鍵決策上可能漏掉隱藏風險
  • Advisor Strategy：Sonnet 執行日常 + Opus 審查「奇怪案例」→ Pareto 最優

Bolt 的測試顯示：這個組合能以遠低於 Opus 的費用，在複雜案例上達到接近 Opus 的智能效果。

什麼是 Oddball Cases（奇怪案例）？

大多數任務都是「正常形狀」——模式清晰，Sonnet 可以輕鬆處理。但偶爾會有「奇怪形狀」——需要深度推理、涉及隱藏風險、或有異常複雜的規則——這時 Sonnet 可能漏掉關鍵細節。

  • 正常案例（Sonnet 適合）：常規合約審查、日常數據摘要、標準問答
  • 奇怪案例（需要 Opus）：合約中有隱含衝突的條款、多方利益相關的決策、涉及高風險的邊緣情況
  • 關鍵設計問題：如何讓 Sonnet 知道「這個案例超出我的能力範圍」？需要設計觸發條件。

Punit 的建議：從 Opus 開始，觀察什麼情況下 Sonnet 的結果不夠好，把那些情況設計為 Opus 觸發條件。

Metropolis 合約續約驚魂

西瓜效應：外綠內紅，差點釀成大錯

HeroCorp 正在審查大客戶 Metropolis 的合約續約。使用純 Sonnet 架構時：

「Sonnet 表面評估為安全（綠燈）——合約條款看起來沒問題，一切正常。」

切換到 Advisor Strategy 後，Sonnet 把脈絡交給 Opus 顧問進行深度審查。Opus 敏銳地發現了一個隱藏風險：合約指定英雄 Cryothane 必須出席某個關鍵場合，但 Cryothane 當天根本沒空。

這就是「西瓜效應」——外表綠（合約看起來安全），內部紅（有一個隱藏的執行風險）。Opus 成功推翻了 Sonnet 的綠燈評估，發出警示，讓 HeroCorp 即時介入處理，挽救了這張關鍵續約。

如果沒有 Advisor Strategy，這個風險會被忽略，合約可能因為 Cryothane 缺席而引發客戶不滿。

V

思考題 · 六個值得深想的問題

Punit 的演講展示了三個降低成本的技術。但技術背後有更深層的設計哲學值得探討——從商業模型的建構，到 AI 系統的信任邊界。

Q1 · Punit 說 Replit、Cursor、Perplexity 的快取命中率都超過 90%。這表示頂尖 AI 產品團隊把「系統 Prompt 的靜態化」當作核心工程目標。你現在最重要的 Agent 的系統 Prompt 中，有哪些部分是動態的（會因為請求不同而改變）？把這些部分移出系統 Prompt 需要什麼代價？

查看建議思考方向
  • 量化參考：快取命中率每提升 10%，意味著大約 10% 的 API 費用降低。從 0% 提升到 90% 理論上節省了 81% 的輸入 Token 費用（90% 折扣 × 90% 的 Token）。
  • 講者隱含論點：Punit 說頂尖客戶這樣做不是偶然——他們意識到 System Prompt 的「靜態性」是商業可持續性的基礎設施。
  • 職涯啟發：能夠系統性審計 Prompt 結構，識別哪些部分可靜態化的工程師，在 AI 產品成本優化方面有明顯優勢。這是 2026 年最被低估的工程能力之一。
  • 反向思考：極高的快取命中率是否意味著系統 Prompt 「太固定」，缺乏對用戶個性化的彈性？靜態化和個性化之間如何取得平衡？

Q2 · Advisor Strategy 的西瓜效應案例顯示：Sonnet 把高風險合約評估為安全，Opus 才找到問題。但在真實系統中，你怎麼知道什麼情況下需要升級到 Opus？「Oddball Cases」的觸發條件如何設計？

查看建議思考方向
  • 歷史對照：人類組織中「什麼需要升級給主管」的判斷一直是管理學研究課題。AI 多模型路由面對同樣的問題——規則化的邊界 vs. 學習到的邊界哪個更可靠？
  • 講者隱含論點：Punit 的建議是「從 Opus 開始，觀察 Sonnet 失敗的模式，把那些模式設計為升級觸發條件」——這是資料驅動的方法，而非規則驅動。
  • 職涯啟發：設計高品質的「升級觸發條件」需要同時理解業務邏輯和模型能力邊界——這種混合能力在 2026 年非常稀缺。
  • 反向思考：如果 Sonnet 無法識別「自己不知道的事」（Known Unknowns vs Unknown Unknowns），基於 Sonnet 判斷的升級觸發機制是否有系統性盲點？

Q3 · Punit 展示 Context Engineering 讓「垃圾更少 → 模型表現更好」。但在真實業務中，「什麼是垃圾、什麼是相關信息」往往不清楚——只有當模型失敗時才知道少了什麼。你如何系統性地決定哪些資訊應該進入上下文？

查看建議思考方向
  • 量化參考：Lovable 的 10% Token 減少是透過 Tool Search 達成的。他們如何知道哪些工具定義是「不必要的垃圾」？可能是透過工具調用頻率分析——很少被用到的工具定義是候選刪除對象。
  • 講者隱含論點：Punit 的 Programmatic Tool Calling 案例暗示：「讓 Claude 自己決定要壓縮哪些 API 回傳」是一個可行策略——Meta 使用 AI 決定什麼值得告訴 AI。
  • 職涯啟發：Context Engineering 是 2026 年一個快速成熟的領域。能夠建立「信息相關性評估框架」的工程師，將成為 Agent 架構師角色的核心競爭力。
  • 反向思考：如果你的「Context Engineering」過於激進，過濾掉了某些不常用但關鍵的信息，可能在邊緣案例中導致嚴重的模型失誤。過度優化和信息完整性之間如何平衡？

Q4 · Punit 展示了三大技術讓成本從 £31 降到 £3——降低了 90%。在你的業務或應用中，AI 成本的瓶頸在哪裡？這三個技術中，哪一個對你的場景最有可能帶來最大的投資回報？

查看建議思考方向
  • 量化參考：Prompt Caching 效果最容易量化（快取命中率直接對應費用減少）；Context Engineering 效果最複雜（需要實驗）；Advisor Strategy 效果取決於「奇怪案例」的頻率。
  • 講者隱含論點：Punit 建議按順序嘗試：先快取（零智能損失，立竿見影），再脈絡工程（需要一些工程投資），最後考慮模型路由（需要評估和監控）。
  • 職涯啟發：能夠快速估算「每個優化技術的 ROI」並排列優先序的工程師，在和 PM 溝通資源分配時有巨大優勢。不是所有優化都值得投入相同的工程時間。
  • 反向思考：有些時候，「成本不是問題」——如果你的應用創造的價值遠超 AI 費用，過度優化成本反而是機會成本的浪費。你確定成本優化是當前最重要的事情嗎？

Q5 · 演講開場 Punit 問：「誰對自己 Agent 的品質/成本/速度三者都完全滿意？」幾乎沒人舉手。為什麼這三個目標在現實中很難同時達到？它們之間的張力本質是什麼？

查看建議思考方向
  • 歷史對照：軟體工程中的 CAP 定理（Consistency, Availability, Partition tolerance）類似地指出三個目標最多只能同時滿足兩個。AI Agent 的品質/成本/速度是否也有類似的「三選二」的本質性張力？
  • 講者隱含論點：Punit 展示的三大技術打破了「三選二」的假設——在某些條件下（特別是快取），你可以同時改善三個指標。這意味著「三選二」不是絕對的，而是取決於架構設計。
  • 職涯啟發：理解品質/成本/速度三者之間的動態平衡，並能根據業務場景設計合適的取捨策略，是 AI 產品工程師區別於純模型用戶的核心能力。
  • 反向思考：也許「完全滿意」本身就是錯誤的目標——優化是持續過程，隨著模型演進、業務增長，「平衡點」會不斷移動。能夠持續迭代這個平衡的組織能力比一次性「達到滿意」更重要。

Q6 · Punit 預告 2026 年即將推出「Auto Caching」自動快取功能。如果平台層的優化都能自動化（不需要開發者手動設計快取策略），Context Engineering 還有多少人工設計的空間？平台智能化會讓這些技巧變成「過時知識」嗎？

查看建議思考方向
  • 歷史對照：資料庫查詢優化（Index 設計、查詢計畫）在 20 年前需要大量人工優化，現在資料庫引擎能自動完成大部分。但高階的 Schema 設計和查詢架構仍然需要人類判斷。類比：自動快取解決「策略」，但「什麼進脈絡」仍需人工設計。
  • 講者隱含論點：Punit 說「自動快取」即將來臨——暗示 Anthropic 認為快取策略可以被自動化。但他同時展示了 Context Engineering 的複雜性（Tool Search、Compaction 的設計）——這些更難自動化。
  • 職涯啟發：自動化的邊界是「已知的、可規則化的最佳實踐」。深度的業務邏輯判斷（什麼是垃圾、什麼是關鍵信息）不能自動化。投資後者的能力比前者更有未來價值。
  • 反向思考：平台功能的自動化讓基礎優化「民主化」（人人都能達到合理水準），但同時拉高了競爭起跑線。當自動快取普及後，差異化的競爭優勢來自哪裡？
給初學者 · 名詞速查
本場演講關鍵術語
  • Prompt Caching：預處理固定 Prompt 部分，90% 折扣 + 速率提升
  • Cache Hit Rate：快取命中率，越高代表越省錢
  • TTFT：Time to First Token，首字延遲，越低代表越快
  • Tool Search：按需動態載入工具定義，而非全部預載
  • Programmatic Tool Calling：讓 Claude 寫腳本預過濾 API 回傳
  • Compaction：接近脈絡上限時自動摘要並清空舊歷史
  • Advisor Strategy：低成本模型執行 + 高級模型在複雜案例顧問
  • Oddball Cases：需要高級模型深度推理的邊緣複雜案例
  • Pareto Optimal：在成本和智能之間找到最優平衡點，不可能同時再改善兩者
給初學者 · 明天可以做的事
三個「今晚就能開始」的行動
  • 1. 前往 console.anthropic.com 查看你的 Prompt Cache 命中率。如果低於 50%，找出是什麼在動態改變系統 Prompt。
  • 2. 計算你的工具列表大小：你的 Agent 有幾個工具？如果超過 10 個，考慮是否可以改用 Tool Search 動態載入。
  • 3. 估算成本轉換效益：如果你的應用改用 Sonnet 執行 80% 任務、只在關鍵決策用 Opus，費用大概能降低多少？

Punit 的核心訊息：「讓 AI 成本可持續，才能讓商業價值真正成立。平台技術是實現這個目標的工具，而不只是優化細節。」
