I

「AI 保姆」困境

當模型越來越聰明，我們卻把越來越多時間花在盯著螢幕等 Claude 完成工作——這是一個生產力悖論。

Sid Bidasaria 是 Claude Code 的創始工程師。他在開場白中點出了一個讓所有 AI 開發者都心有戚戚的現象：「隨著模型不斷變強，我們花在盯螢幕等 Claude 完成工作的時間，以及充當『榮耀品保測試員』的時間，正在不成比例地增加。」

這不只是讓人不滿足，更是一種效率的浪費。本場演講的目標，是給你具體策略，讓你把這部分時間拿回來。

"As models have been getting smarter, I've noticed that we're increasingly spending a larger percentage of our time staring at the screen waiting for Claude to finish its work, or just acting as a glorified QA tester for Claude." 「隨著模型不斷變強，我注意到我們越來越多的時間花在盯著螢幕等 Claude 完成工作，或者只是充當 Claude 的榮耀品保測試員。」

— Sid Bidasaria，Claude Code 創始工程師
給初學者 · 一句話搞懂「AI 保姆」現象
為什麼我們反而更忙了？

想像你雇了一個很有能力但不知道「完成了要怎麼確認」的實習生。他每完成一步都要來問你：「這樣對嗎？」你反而比自己做還累。

  • 以前：你自己寫程式，邊寫邊驗證，節奏自己掌控
  • 現在：Claude 寫程式，但每個步驟都需要你在旁盯著、審查
  • 目標：教 Claude 如何自己驗證，讓你只在真正重要時介入

「停止保姆式監控」不是減少使用 AI，而是升級 AI 使用方式。

三大先決條件

在進入進階策略之前，Sid 列出了每個 Claude Code 使用者都應該完成的基礎設定——缺一不可。

I

高品質 CLAUDE.md 檔案

「這是你能做的最高槓桿率的一件事。」好的 CLAUDE.md 讓 Claude 一開始就了解你的專案脈絡、程式碼規範、常用指令。

II

連接你的工具

Slack、Asana、Linear、DataDog、BigQuery——凡是對你日常工作有用的工具，對 Claude 也有用。工具越多，Claude 能縫合的上下文越豐富。

III

設定遠端環境（Claude Code Web）

讓運行 Claude Code 的算力與你的筆電脫鉤。筆電關閉、沒電、被潑水——你的 Claude Session 仍在雲端持續運行。

給初學者 · 一句話搞懂「CLAUDE.md」
你給新同事的入職手冊

CLAUDE.md 就是你寫給 Claude 的「新人指南」——告訴它這個公司（專案）有哪些規矩、喜歡怎麼做事、有哪些禁忌。

  • 放在專案根目錄的 Markdown 檔案
  • 每次對話開始時 Claude 會自動讀取
  • 內容：程式碼規範、測試要求、部署流程、重要提示

一份好的 CLAUDE.md 等於免費的「永久上下文」——不需要每次都重新解釋專案背景。

I

三層核心策略

Sid 的整個演講架構在三個遞進層次上——每一層都建立在前一層的基礎之上，合起來形成一個真正「自駕」的開發系統。

Layer III
背景迴圈 · Background Loops
鍵盤不再是瓶頸，Claude 自動處理例行事務
Layer II
多工並行 · Multi-Clauding
在有限注意力下同時管理多個 Claude
Layer I
驗證迴圈 · Verification
教 Claude 自我檢查、自我修復

關鍵洞見：這三層策略不是互相替代，而是相互依存。你必須先讓 Claude 能可靠地驗證自己的工作（Layer I），才能有信心同時跑多個 Claude（Layer II），才能進一步讓它們在背景無人值守地自動運行（Layer III）。

I

核心問題：工具鏈需要重新設計

所有現有的開發工具——從 linter 到 IDE 到 compiler——都是為人類設計的。但現在寫程式的主力已經是 Agent。

"What does an agent need from your codebase that a human takes for granted?" 「一個 Agent 需要從你的程式碼庫中得到哪些東西，而這些東西是人類理所當然擁有的？」

— Sid Bidasaria

好消息：Prettier、linter、符號伺服器等工具，對 Agent 同樣有效——Claude 能很好地利用它們。

壞消息：我們的工具鏈有盲點。人類工程師有很多「不言而喻的知識」，例如如何啟動開發伺服器、如何登入測試帳號、如何重置資料庫狀態——Claude 沒有這些，需要明確告訴它。

給初學者 · 一句話搞懂「Agent 的工具盲點」
給陌生人寫食譜 vs 給朋友說「你知道的那樣做」

你告訴朋友「把那個東西放進去」，他知道你在說什麼。但對一個第一次來你家的人，你得說「打開冰箱第二格，拿那瓶醬油」。Claude 是那個陌生人。

  • 人類懂「npm run dev 啟動前端」——需要明確寫在 CLAUDE.md
  • 人類懂「先登入 admin@test.com」——需要寫成身份驗證腳本
  • 人類懂「看一下 console 有沒有報錯」——需要給 Claude 瀏覽器工具

所謂「工具整合」，就是把人類理所當然的知識，變成 Claude 可以使用的工具和指令。

II

人類怎麼驗證工作

在教 Claude 驗證之前，先搞清楚我們自己是怎麼驗證的——因為同一套流程，Claude 也可以做。

Sid 請聽眾回想自己最近做過的軟體專案：「你是怎麼邊做邊確認的？」他發現大多數軟體工程任務可以拆解成以下幾個步驟的某種組合：

01
✏️
設計 & 寫碼
Design & Code
02
🔨
編譯 & 建置
Build & Compile
03
▶️
執行應用
Run Executable
04
🔍
檢查副作用
Check Side Effects
05
🧪
跑單元測試
Unit Tests
06
🚀
部署
Deploy / Staging
↩ 當任何步驟失敗 → 自動回到上一步修正 → 形成自我驗證迴圈
給初學者 · 一句話搞懂「驗證迴圈」
就像洗衣機的漂洗循環

洗衣機不是洗一遍就停——它會重複「加水、洗、脫水」的循環直到乾淨。驗證迴圈也一樣：Claude 寫程式、測試、如果有問題就修、再測、直到成功。

  • 傳統做法：Claude 寫完 → 你測試 → 你告訴 Claude 哪裡錯 → 重來
  • 驗證迴圈：Claude 寫完 → Claude 自己測試 → Claude 自己修 → 你只看最終結果

驗證迴圈的關鍵：給 Claude 正確的工具和指令，讓它能自己跑這個循環。

II

建立前端驗證迴圈：四個步驟

理論很美好，但實際要怎麼做？Sid 把 UX/Frontend 驗證迴圈拆成四個具體動作。

01

啟動應用程式

告訴 Claude 怎麼啟動開發伺服器（例如 npm run start）。這是一切的起點——如果 Claude 不知道怎麼跑起來，後面都免談。

02

使用瀏覽器工具

給 Claude 一個瀏覽器 MCP 工具，讓它能實際操作網頁介面。推薦：/chrome MCP、Playwright、或其他 Browser Control MCP。

03

證明有效 / 無效

教 Claude 截圖「修復前」和「修復後」並比較。不只是「程式跑起來了」，而是「行為確實符合預期」。

04

解決阻塞問題

常見阻塞：身份驗證（給 Claude 一個測試帳號）、狀態設定（預先填入測試資料）。這和 E2E 測試的 Setup Script 概念一樣——只是要讓 Claude 動態使用它。

真實 Demo · 修復 Monkeytype 的 Signup 按鈕

Claude 如何在 Monkeytype 上偵測並修復一個壞掉的按鈕

Sid 以 Monkeytype（打字速度測試 App）作為 Demo 平台。他的個人網站上 Signup 按鈕突然停止工作。

「它開始寫一些程式碼，建置了我的 App，點擊了 Signup 按鈕，開啟瀏覽器，發現點下去沒有任何反應、不會跳到任何地方。所以它決定讀取 Log，找到了問題所在，修復程式碼，重新載入 App，不斷循環，直到達到成功狀態。最後產出了一個確實有效的 PR。」

重點：Claude 自己發現問題、自己讀 Log、自己修復、自己驗證——整個循環無需人類介入。這就是驗證迴圈的實際效果。

Demo 中還發生了一個精彩的額外環節：Claude 被要求製作「每次打錯字就出現彩紙動畫」的功能。它寫好後發現有兩個 ESLint 錯誤，自動修復了這些錯誤，再次驗證，最終成功——完全沒有請求人類協助。

II

技能（Skill）：讓驗證可傳播

學會如何驗證還不夠——你需要把這個能力打包成一個可以傳給同事、傳給未來的自己的「技能包」。

給初學者 · 一句話搞懂「Claude Code Skill」
像公司的 SOP 手冊

一個公司不可能每次換人都重新教一遍所有流程。所以有 SOP（標準作業程序）手冊。Claude Code Skill 就是你給 Claude 的 SOP 手冊。

  • 一個放在 .claude/skills/ 目錄下的 Markdown 檔案
  • 描述如何完成某個特定任務（如「如何驗證 Frontend」）
  • Claude 執行任務時會自動參考
  • 可以設定為「自我更新」——遇到新阻塞就更新自己

Skill 的神奇之處：它不只是知識的儲存，還可以是會自我進化的 SOP。

建立 Skill 的流程非常直接：先「手把手」帶著 Claude 做一遍驗證流程，然後告訴它：「把你剛才學到的東西整理成一個 Skill 檔案。」

# Demo 中的實際指令
「Take everything we learned and put it into a
skill file in .claude/demo-verification/skill.md」
# 生成的 Skill 包含：
1. 啟動 Stack 的指令（Docker Compose etc.）
2. 載入 Chrome MCP 工具
3. 煙霧測試（Smoke Test）的具體步驟

"Every time someone runs into a blocker, the skill will go back in and edit itself." 「每當有人遇到阻塞，這個 Skill 就會自動回去編輯自己。」

— Sid Bidasaria

這是 Claude Code 團隊內部的實際做法：他們有一個共用的驗證 Skill 檔案，明確指示 Claude 「持續記錄自己的學習」。每次有人遇到新問題、找到解法，Skill 就會更新——變成一個能在整個團隊之間共享的集體知識庫。

給初學者 · 一句話搞懂「自我改進的 Skill」
會自我更新的活文件

想像一份食譜，每次你做菜遇到問題、找到更好的方法，食譜就自動更新——下次你或任何人拿到的都是最新版本。

  • 在 Skill 檔案加入指令：「每次遇到新阻塞，把解決方案記錄在本檔案」
  • Claude 遇到問題 → 解決 → 自動更新 Skill → 未來不再踩同樣的坑
  • 整個團隊共享同一個 Skill → 集體智慧不斷累積

自我改進的 Skill 是「文件化」最無痛的方式——工程師不需要專門花時間寫文件，它自然而然就發生了。

II

驗證的三種口味

前端 UX 只是其中一種驗證。同樣的邏輯可以應用到後端 API、整個 App 的端到端流程，甚至基礎設施。

🖥️ Frontend / UX 驗證

啟動開發伺服器 → 使用瀏覽器 MCP → 截圖比對 → 檢查 UI 狀態

適用：React/Vue/HTML 介面、互動功能、視覺正確性

⚙️ Backend API 驗證

啟動後端服務 → 發送 HTTP 請求 → 驗證 response body → 檢查資料庫狀態

適用：REST API、GraphQL、背景工作、資料管道

🔗 End-to-End 驗證

整個 Stack 啟動 → 從前端操作 → 驗證後端資料變化 → 確認一致性

適用：關鍵用戶流程、跨服務整合、部署前最終確認

🏗️ Infrastructure 驗證

部署 IaC 變更 → 確認資源狀態 → 測試健康檢查 → 驗證連通性

適用：Terraform、K8s、Docker Compose、雲端資源

Sid 強調：「核心概念始終相同——給 Claude 正確的工具和指令讓它進入迴圈。一旦你搞清楚這一點，這三種口味自然合而為一。」

II

驗證迴圈：最重要的一張投影片

核心洞見 · 迴圈才是讓一切轉動的機制

Hill Climbing 的比喻：Claude 如何朝成功狀態爬升

Sid 說：「迴圈是讓整件事轉動的核心。這可能是這場演講最重要的一張投影片。」

「一個迴圈本質上是你可以為 Claude 完成的一條自主電路。它允許 Claude 在給定的任務或成功標準上不斷 Hill Climbing（向山頂爬升）。」

具體來說：給 Claude 驗證工具和寫程式工具 → Claude 寫程式碼 → 檢查是否失敗 → 如果失敗，debug 後再寫 → 不斷循環直到成功 → 你收到的 PR 品質更高、確實能用。

啟示：我們的目標不是讓 Claude 一次就寫對（那是不現實的），而是讓它能快速迭代、自我修正，就像有經驗的工程師一樣。驗證迴圈就是這種能力的具體實現。

III

注意力是稀缺資源

當你的 Claude 夠可靠了，下一個問題是：如何同時管理多個 Claude？問題的核心是注意力管理，而不是運算資源。

「當你試圖同時跑太多 Claude 時，它們全部都在消耗你的注意力。注意力是一種稀缺資源。」Sid 坦承，他個人的上限是 4-5 個並行 Session——超過這個數字，大腦就開始不堪負荷。

並行 Session 數 vs 注意力負荷

1–2 Sessions
低
3–4 Sessions
適中
5–6 Sessions
高
7+ Sessions
極限

解決方案不是「減少同時跑的 Claude」，而是用更好的工具讓每個 Session 消耗更少的注意力。

給初學者 · 一句話搞懂「注意力管理 vs 運算資源」
多工並行的真正瓶頸不是電腦，而是你

電腦可以同時跑 100 個 Claude Session。但你只有一個大腦，只有一雙眼睛。

  • 運算資源：幾乎無限（雲端可以一直擴充）
  • 人類注意力：極度有限（認知科學研究顯示 7±2 個工作記憶上限）
  • 解法：減少每個 Session 需要你「主動監看」的程度

多工工具（Desktop App、Claude Agents）的核心設計目標：讓你用最少的注意力，管理最多的 Claude Sessions。

III

四種多工工具

Anthropic 提供了四種不同的介面來管理多個 Claude Session，各有優缺點，適合不同使用場景。

GUI · 桌面版

Claude Code Desktop App

左側 Sidebar 列出所有 Session（本地 Terminal + 雲端 + 所有 Git Repo）。可 Pin、重命名、上色分類。是管理多 Session 的中央控制台。

適合：喜歡 GUI 的使用者、需要跨多個專案管理

NEW · 一週前上線

Claude Agents（終端機視圖）

指令：claude agents。在終端機內提供類似 Desktop 的多 Session 視圖，並依「需要注意程度」自動排序——被阻塞的 Session 排在最上面。

適合：熱愛終端機的開發者

WEB · 雲端

Claude Code on Web

前往 claude.ai/code。算力跑在 Anthropic 雲端，而非你的筆電。關電腦、沒電、在車上——Session 持續運行。

適合：需要長時間跑任務、不想被綁在電腦前

★ 最愛功能

Remote Control（遠端控制）

在任何 Session 輸入 /remote-control。Claude 被阻塞時，你的手機會震動通知。你可以在車上、會議中、任何地方用手機給 Claude 指令。

適合：所有人——真正實現「不被綁在螢幕前」

給初學者 · 一句話搞懂「Remote Control」
就像有了一個可以隨時呼叫你的 AI 助手

Remote Control 讓 Claude 能在需要你的時候主動聯繫你，而不是你一直盯著螢幕等它。

  • 設定：在 Claude Code Session 輸入 /remote-control
  • Claude 遇到需要許可、需要決策的時刻 → 你的手機震動
  • 你可以在任何地方用手機給它指令並繼續
  • 就像「寫字板的 Claude 打你電話讓你確認」

這是真正的「非同步工作」——你不需要持續在場，只需要在真正需要人類判斷的時候出現。

III

從 tmux 到 Claude Agents：進化史

Sid 分享了自己多工方式的演進——從粗糙的 tmux + Worktree 組合，到現在更優雅的解決方案。

舊

tmux + Git Worktrees 手動管理

一個 tmux 視窗分成四個窗格，每個窗格跑不同 Worktree 上的 Claude。「有效，但要管理 Worktree 和 tmux，也不是最方便的事。」

新

Claude Agents：自動化版的 tmux

同樣的並行能力，但 Claude Agents 自動處理「哪個 Session 最需要你的注意」這件事，並以視覺化方式排序。你不需要自己在四個窗格間跳來跳去。

這個演進的核心洞見：工具進化的方向是「把注意力從機制管理（管理 tmux/worktree）轉到任務管理（決定 Claude 下一步做什麼）」。

IV

工程師的日常瑣事清單

即使有了驗證迴圈和多工並行，我們每天仍然花大量時間在不需要「真正創意」的例行工作上。

Sid 列出了現代 AI 驅動的工程師每天面對的瑣事：

①

PR 保姆（PR Babysitting）

「現在我們每天能產生更多 PR，這些 PR 需要合併。但合併前要處理審查意見、合併衝突、CI 失敗。如果你一天有 20-30 個 PR 要合併，輕鬆就花掉好幾個小時。」

②

更新文件（Updating Docs）

出貨速度加快，文件的更新速度也必須跟上。但這是重複性的工作，不需要人類在迴圈中。

③

分類、監控、保持 CI 綠燈

Issue 分類、監控 Feedback、確保 CI 持續通過——每天都要做，但沒有一件是需要人類創意的。

"Your keyboard is not the bottleneck anymore." 「你的鍵盤不再是瓶頸了。」

— Sid Bidasaria
給初學者 · 一句話搞懂「背景迴圈」的價值
就像自動繳款——設定好就不用管了

你不需要每個月手動去銀行繳電費——設定自動繳款，它就自己運作。背景迴圈對程式碼工作的意義一樣：設定好規則，Claude 就自動在背景處理例行任務。

  • 不需要：你盯著螢幕等
  • 不需要：你手動觸發每個任務
  • 只需要：設定一次觸發條件和指令

這就是「你的鍵盤不再是瓶頸」的具體意義。

IV

/loop 指令：背景迴圈的最簡形式

/loop 讓你以固定時間間隔自動執行一個 Prompt——不需要複雜設定，立刻可用。

# 基本語法
/loop 10m "babysit my open PRs"
# 每 10 分鐘喚醒一次
# 執行：「處理我的 PR 保姆工作」
# 如果 CLAUDE.md 和工具設定正確
# → Claude 自己知道要做什麼

/loop 指令 的強大之處在於它的前提：如果你的 CLAUDE.md 和工具整合做得夠好，Claude 會根據指令自動判斷需要做什麼——你不需要在指令裡把每個步驟都寫清楚。

給初學者 · 一句話搞懂「/loop」
定時鬧鐘 + AI 助手的組合

想像你設定了每 10 分鐘的鬧鐘，每次響起時，AI 就自動醒來、看看有什麼需要做、做完後繼續睡覺，等下次鬧鐘響。

  • /loop 10m：每 10 分鐘喚醒一次
  • /loop 1h：每小時喚醒一次
  • 每次喚醒都執行你設定的 Prompt
  • 你去做其他事，迴圈自動在背景跑

最好的 /loop 使用場景：任何「需要定期檢查但不需要你親自動手」的任務。

IV

Routines：雲端版的 /loop

Routines 把 /loop 的概念搬到雲端，並增加了事件驅動觸發——讓你的自動化不受筆電狀態影響。

Routines 和 /loop 的核心區別：/loop 在你的電腦上跑，電腦關掉就停止。Routines 在 Anthropic 的雲端容器中跑，永遠在線。

⏰ 時間觸發（Time-based）

設定定時執行。例如：每天早上 9 點更新文件、每 6 小時整理 Issue、每晚檢查 CI 狀態。

⚡ 事件觸發（Event-based）

某件事發生就觸發。例如：有新的 PR 建立 → 自動進行初步審查、有 CI 失敗 → 立即嘗試修復。

Claude Code 團隊的實際應用

Routines 在 Anthropic 內部是這樣被用的

Sid 分享了 Claude Code 團隊自己使用 Routines 的真實案例：

「我們有一個 Routine 每天更新 Claude Code 的文件。我們還有一個 Routine 每六小時查看我們的 Issue 和用戶 Feedback，並把摘要 Post 到我們的 Slack 頻道。」

這兩個任務有什麼共同點？它們都是「有用但重複」的工作——如果讓工程師每天手動做，既耗時又讓人沮喪。但對 Claude 來說，這正是它最擅長的事。

啟示：把「有用但重複、不需要真正創意」的任務列成清單，然後逐一把它們變成 Routine。這是提升工程團隊整體速度的低成本高效益方式。

給初學者 · 一句話搞懂「Routines vs /loop」
本地鬧鐘 vs 雲端定時服務

你手機的鬧鐘需要手機開著才能響。但 Google Calendar 的提醒不管你的手機在不在、電量夠不夠，都會發出通知。

  • /loop = 手機鬧鐘：你的電腦要開著才會跑
  • Routines = 雲端服務：永遠在線，不受你的裝置影響
  • 設定方式：Claude Code Web App 或 Desktop App 的「Routines」頁籤

如果你設定的自動化任務很重要、不能因為你關電腦就停止，用 Routines；如果只是臨時性的，/loop 就夠了。

IV

三層策略的完整閉環

當你把驗證迴圈、多工並行、背景自動化三層全部疊在一起，你到達了一個全新的工作狀態。

Sid 在演講結尾描述了這個狀態：

「當你把這三個技能疊加在一起，你最終會得到一個能在你不必手動敲鍵盤的情況下完成大量工作的系統。這才是終極目標——你可以把注意力和時間放在你真正在乎的任務上，其他一切都可以高可靠度、高信心地委託給 Claude。」

「高可靠度、高信心」——注意這兩個詞。這就是為什麼驗證迴圈必須是第一層。沒有可靠的驗證，你不敢讓 Claude 在背景自由運行。三層策略的順序不是隨機的——它反映了建立信任的必要過程。

給初學者 · 一句話搞懂「三層策略的整體效益」
從「監工」到「策略家」的角色轉換

傳統工程師角色：執行者（寫程式）+ 監工（確保程式對）。三層策略的目標：把「監工」的工作自動化，讓工程師只做「策略家」（決定做什麼）。

  • Layer I 驗證：Claude 自己當監工
  • Layer II 多工：你管理多個「自動監工」的 Claude
  • Layer III 背景：連管理都自動化，你只在關鍵時刻被通知

最終結果：你的時間和注意力，只花在真正需要人類判斷和創意的地方。

V

六道引導思考題

Sid 的演講提出了幾個值得深思的問題——關於工具設計、人機協作邊界、以及 AI 時代的工程師角色。

Q1 · Sid 說「最重要的一件事是建立高品質的 CLAUDE.md」。但 CLAUDE.md 需要人工維護，如果代碼庫快速演進，它很快就會過時。有沒有辦法讓 CLAUDE.md 也自動更新？這是一個自我參照的矛盾嗎？

查看建議思考方向
  • 歷史對照：傳統 README 和 Wiki 同樣面臨「文件腐爛」問題——軟體業幾十年來一直沒有解決。AI 時代是否提供了新的解方？
  • 講者隱含論點：Sid 說 Skill 可以「自我更新」——相同邏輯是否也適用於 CLAUDE.md？可以在 CLAUDE.md 裡加入「當你發現本文件過時時，主動更新它」的指令嗎？
  • 職涯啟發：「讓文件自我維護」是一個值得探索的技術方向，也許是未來的開源工具機會。
  • 反向思考：也許 CLAUDE.md 的「過時」不完全是壞事——它可以作為「人類確認節點」，強迫工程師定期重新審視自己的工作流程假設。

Q2 · 驗證迴圈需要「給 Claude 正確的工具」（瀏覽器、日誌、資料庫存取）。但這也意味著 Claude 有很高的系統存取權限。驗證迴圈設計中，安全邊界應該在哪裡劃定？

查看建議思考方向
  • 歷史對照：CI/CD 系統（Jenkins、GitHub Actions）幾十年來也面臨相同問題——自動化需要存取權限，但存取權限帶來安全風險。業界如何處理？
  • 講者隱含論點：Sid 說驗證迴圈和 E2E 測試類似——傳統 E2E 測試也有專用的「測試帳號」和「隔離環境」，驗證迴圈是否應該沿用相同框架？
  • 職涯啟發：「AI 代理的最小權限原則」可能成為下一個重要的安全工程子領域。
  • 反向思考：限制太多會讓驗證迴圈失效（Claude 沒辦法「真的」測試）；限制太少有安全風險。也許這個張力本身就是 AI 時代安全工程的核心挑戰。

Q3 · 「自我改進的 Skill 檔案」是個強大的想法——但如果 Claude 持續改寫 Skill，有沒有可能改進了錯誤的方向？誰來審查 Skill 的演進？

查看建議思考方向
  • 歷史對照：Wiki 和共享文件（Google Docs、Confluence）也有類似問題——誰有權編輯，如何防止「善意的錯誤更新」積累成問題？
  • 講者隱含論點：Sid 強調 Skill 需要「全團隊共享」，這暗示需要某種 Governance 機制。Git 版本控制是顯然的答案嗎？
  • 職涯啟發：設計「AI 可編輯但人類可審計的系統」可能是新型軟體架構的重要模式。
  • 反向思考：也許自我改進的錯誤累積本身不是最大風險——更大的風險是「沒有自我改進的僵化文件」導致的低效。找到正確的風險平衡點更重要。

Q4 · Sid 提到「注意力上限是 4-5 個 Sessions」。但 Remote Control 和背景 Routines 的設計目標是讓你能「不主動監看」——這是在突破注意力上限，還是在重新定義「注意力」這個概念？

查看建議思考方向
  • 歷史對照：手機通知（Email、Slack）已經把我們的注意力從「主動查看」變成「被動響應」——生產力研究如何評估這個轉變？
  • 講者隱含論點：Remote Control 把 Session 監控從「主動」變成「被動」（只在被通知時才需要注意），這在認知科學上是否真的減少了注意力負荷？
  • 職涯啟發：「非同步 AI 協作」可能是未來最重要的工作技能之一——設計一套適合自己的「通知系統」而非「監看系統」。
  • 反向思考：如果你有 50 個背景 Routines，每個偶爾通知你一次——這個通知密度可能比「主動監看 5 個 Sessions」更耗費注意力。量的增加最終還是會質變。

Q5 · Sid 說工具鏈需要「從為人類設計，轉變為為 Agent 設計」。但在相當長的時間內，兩者必須共存。你怎麼設計一套同時對人類和 Agent 都友善的開發環境？

查看建議思考方向
  • 歷史對照：UX 設計中的「無障礙設計」（Accessibility）也面臨類似挑戰——為特殊需求設計的功能，往往也讓一般使用者受益（例如電梯按鈕的高度）。
  • 講者隱含論點：Sid 說「為 Agent 好的工具，對人類也好」——這個「雙贏」是真實存在的，還是過度樂觀？
  • 職涯啟發：「Agent-friendly API/CLI 設計」可能是下一代開發工具的重要標準——先想想你正在使用的工具有哪些地方對 Claude 不友善。
  • 反向思考：有些「對人類好」的設計（互動式確認、視覺化 UI）對 Agent 來說反而是障礙。兩者可能存在真正的張力，而不只是工程問題。

Q6 · Sid 展示了 Claude 能自動修復 Lint 錯誤並繼續完成任務。如果 Claude 自動「解決」了一個你其實想要親手決定的問題，你怎麼知道？如何在「自動化效率」和「人類控制感」之間取得平衡？

查看建議思考方向
  • 歷史對照：自動駕駛汽車的「Mode Confusion」問題——駕駛不清楚車子現在是「自動」還是「手動」模式，導致事故。這個問題在 AI 編程工具中如何類比？
  • 講者隱含論點：Sid 強調驗證迴圈讓你能「以高信心委託工作給 Claude」——這種信心需要透明度（知道 Claude 做了什麼）才能成立。
  • 職涯啟發：制定個人的「Claude 自動決策邊界」清單——哪些事 Claude 可以自動做，哪些事必須通知你後再做。
  • 反向思考：也許「控制感」本身就是一種幻覺——在複雜的程式碼庫中，你真的完全理解每一個 Commit 嗎？AI 只是讓這個「不完整控制」的現實更加顯性化。
