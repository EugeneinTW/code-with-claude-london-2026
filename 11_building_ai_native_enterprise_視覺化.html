給初學者 · 什麼是巨石架構（Monolith）？
把整個城市塞進一棟大樓 vs 蓋了一個城市

Monolith 是把所有功能都放在同一個代碼庫的架構。想像一棟大樓：登記、餐廳、銀行、醫院都在同一棟裡面。優點是早期快速，缺點是越來越大後很難動它的任何一部分。

  • Monolith 問題：改一個功能可能影響整個系統；AI Agent 難以找到清晰的「進入點」
  • 微服務：每個功能是獨立的小城市，可以單獨更新，AI 更容易整合
  • 現實：三家公司都沒有「等待拆分完成」才開始 AI 化

關鍵洞見：與其等待架構完美，不如用 API 和封裝層「隔離」Monolith 的複雜性，讓 AI 可以透過 API 存取功能。

共同的核心轉型原則

01
不等待完美條件：三位講者一致建議「現在就開始」，選一個重複性的苦差事，直接讓 AI 全力處理

05
失敗率是金礦：Ruslan 每天深挖 AI 失敗案例，從中發現新產品機會

"Stop waiting for perfect conditions, perfect use cases, also enough AI-ready work... pick a thing tomorrow, maybe a piece of work that your team is doing repetitively and it's toil and kind of is a burden — just unleash Claude full on and see full potential." 「停止等待完美條件、完美使用案例、或足夠 AI 就緒的工作……明天就選一件事，也許是你的團隊重複在做的苦差事——直接全力釋放 Claude，看看它的全部潛能。」

— Ruslan, Engineering Leader, monday.com

I

HeroGen：自主軟體交付

Delivery Hero 的 HeroGen 是目前最先進的企業級「自主軟體交付系統」——從 Jira ticket 到可合入生產的 PR，全程無需工程師介入，除了最終的 Code Review。

給初學者 · 什麼是自主軟體交付系統？
讓 AI Agent 當工程師的「打字機」

傳統軟體開發：PM 寫需求 → 工程師讀需求 → 工程師寫代碼 → Code Review → 上線。HeroGen 做的事情：PM 寫需求 → AI 讀需求 → AI 寫代碼 → AI 自我審查 → 工程師最終 Code Review → 上線。

  • 工程師做什麼：最終確認 PR，而不是從頭寫每一行代碼
  • AI 做什麼：讀 ticket、理解需求、找相關代碼、寫代碼、跑測試、修錯誤、提 PR
  • HeroGen 的成績：日均 173 個 PR，85% 被工程師直接採納

這不是「AI 幫工程師」，而是「工程師監督 AI 軍團」的新模式。

給初學者 · 為什麼一個模型審查自己的代碼有問題？
盲點和偏見：AI 版的「自我審稿」問題

就像作者很難發現自己文章的錯誤——AI 模型在審查自己生成的代碼時，往往會「看不到」自己邏輯的問題，因為它的推理路徑和生成時完全一樣。

  • 單一模型問題：模型 A 生成了有 bug 的代碼，模型 A 審查時看不出來（相同盲點）
  • 多模型解法：模型 B 的訓練方式不同，它能看出模型 A 的盲點
  • 成本考量：多數評審模型可以用輕量版本，整體成本增加不多

Council of Agents 把「讓別人審稿」的人類智慧，應用到了 AI 代碼審查中。

Ulrich 和他的團隊在去年底做了一個大賭注：「讓 AI Agent 完成整個功能，從需求到 PR」——在當時，這還是一個「酷炫的想法」，但模型能力不夠。

教訓：「為下一個模型而建」不只是一個設計原則，更是一個業務策略。那些提前建好架構的人，在新模型發布的那天，直接收穫了成果。

I

架構重來，你會怎麼做？

Rebecca 問了一個最犀利的問題：「如果重來一次，你會做什麼不同的架構決策？」三位工程領導的答案驚人地一致——API-First 和 Agent 身份系統。

"API-First. As I mentioned, I think this would allow — if we invested in this earlier and made it consistent across every service and have clean boundaries — I think that would really accelerate us even more on this journey." 「API-First。如果我們更早投資這個，讓它在每個服務都保持一致並有清晰邊界，我認為這會讓我們在這條路上加速更多。」

— Ruslan, monday.com

決策一：API-First 架構

給初學者 · 什麼是 API-First？
先想「怎麼讓外部程式呼叫我」，再想「UI 怎麼顯示」

API-First 是一種設計哲學：在建任何功能時，先設計好它的 API 接口（讓外部程式可以呼叫），再建 UI（讓人類可以點擊）。

  • 傳統做法（UI-First）：先做出好的使用者介面，API 是事後加的或根本沒有
  • API-First 做法：每個功能先有 API，UI 只是呼叫 API 的一個客戶端
  • 為什麼 AI 時代很重要：AI Agent 需要透過 API 和系統互動，沒有 API 就沒有入口點

monday.com 的早期開放平台投資讓 monday vibe 可以在幾天內完成 POC——因為 AI 直接扮演「外部開發者」，調用現成的 GraphQL API。

UI-First 的代價

Agent 找不到入口

系統功能只能透過 UI 操作。Agent 要操作功能，只能靠螢幕截圖和點擊——這慢、不可靠、且每次 UI 改動都需要重新訓練 Agent。

API-First 的優勢

Agent 有清晰的接口

每個功能都有對應的 API。Agent 直接呼叫 GraphQL 或 REST API，精確、快速、穩定。monday vibe 就是直接用已有的公開 API 建起來的。

決策二：Agent 身份與授權系統

給初學者 · 為什麼 Agent 需要「身份」？
Agent 是「用戶」還是「工具」？答案決定了系統設計

傳統授權系統的設計假設：操作者是人類，有自己的帳號和權限。但 AI Agent 是機器，它可能代表不同的人類用戶執行操作，也可能是完全自主的。

  • 舊系統問題：Agent 要麼沒有任何權限，要麼用人類用戶的帳號（安全風險）
  • 一等公民設計：把 Agent 當成真正的「用戶」，有自己的 Token、細粒度的權限控制
  • monday.com 的挑戰：他們有極細緻的權限模型（誰可以看什麼工作區），這需要從頭為 Agent 重新設計

未來的企業系統，必須從設計之初就考慮「Agent 作為一等公民用戶」。

決策三：Monolith 的正確處理方式

I

不要等待拆分完成（monday.com 的做法）

用 API 層「隔離」Monolith 的複雜性。讓 AI 把舊代碼庫當成黑盒子，透過 API 接口存取功能。monday vibe 就是這樣幾天內完成 POC 的。

II

為 Monolith 提供豐富的上下文（Doctolib 的做法）

Monolith 裡有多種歷史模式（新舊寫法並存）。必須明確告訴 AI「這是舊的，這是新的，請用新的」——不能讓 AI 自己猜。

III

讓 Agent 一起規劃重構（Doctolib 的建議）

Alex 的建議：「讓 Agent 檢視現有代碼庫，告訴它你喜歡什麼、不喜歡什麼，然後讓它制定重構計畫。」把 AI 當成架構夥伴，而非執行工具。

給初學者 · 什麼是 AI Brains（脈絡層）？
讓所有 AI Agent 都能「理解這家公司」的共享記憶

AI Brains 是一個統一的上下文層：存放了公司的業務邏輯、代碼規範、架構邊界、最佳實踐——讓所有 Agent 在開始工作前，都能先「讀懂這家公司」。

  • 沒有 AI Brains：每個 Agent 從零開始理解代碼庫，重複探索，效率低
  • 有 AI Brains：共享的上下文讓所有 Agent 立刻知道：「這個架構的邊界在哪」「新舊寫法的區別是什麼」
  • 誰在建立它：最頂尖的工程師——他們現在的主要工作就是把「只有自己知道的事」結構化、文件化，放入 AI Brains

AI Brains 是企業 AI 轉型的「隱形基礎設施」——看不見，但決定了所有 Agent 的輸出品質。

"The best engineers are typically the principal engineers. And principal engineers, what are they doing usually? They're reviewing other people's code. They're usually not creating new code. This is something where we see the script being flipped — all of these people are now producing a lot of code with the help of Claude." 「最頂尖的工程師通常是 Principal Engineer。他們通常在做什麼？審查別人的代碼，通常不自己寫新代碼。而這正是我們看到劇本被翻轉的地方——這些人現在在 Claude 的協助下，產出了大量代碼。」

— Ulrich, VP Tech Foundations, Delivery Hero

Guided Discussion

六道思考題，幫助你把三家企業的轉型經驗轉化為對自己組織的行動計畫。

Q1

Delivery Hero 的 HeroGen 實現了「日均 173 個 PR 進生產環境」。這個成就的前提是什麼？你認為一家普通的企業需要具備哪些條件，才能安全地達到這個水平？

查看建議思考方向
  • 歷史對照：Google 的 Automated Testing 文化花了超過十年建立。自動化 PR 合入的前提是完整的 CI/CD + 測試覆蓋率。如果 HeroGen 沒有良好的 CI 配套，173 個 PR/day 可能是 173 個生產環境 Bug/day。
  • 講者隱含論點：Ulrich 說 Council of Agents 把成功率推到 85%，這意味著還有 15% 的失敗。在這個規模下，15% 的失敗率如何管理？
  • 職涯啟發：思考「自主軟體交付」適合的任務類型——重複性強、邊界清晰的功能 vs 需要深度創意判斷的功能，哪個更適合 Agent 自主完成？
  • 反向思考：「173 PR/day」是技術成就，但它是否也帶來了「沒有工程師真正理解系統的任何部分」的風險？
Q2

Council of Agents 用多個不同模型互相審查代碼，成本增加不顯著但成功率提升顯著。這個「多視角審查」的原則，還可以應用到 AI 工作流的哪些其他環節？

查看建議思考方向
  • 歷史對照：人類醫療中的「第二意見」制度，法律中的「對立審訊」制度，都是「不同視角發現盲點」的體現。這個原則對 AI 同樣有效。
  • 講者隱含論點：Ulrich 說「避免模型對自己的代碼有盲點」——這個問題在任何 AI 單獨做決策的場景都存在，不只是 Code Review。
  • 職涯啟發：設計一個 AI 工作流時，問自己：「這個環節的輸出需要被另一個視角審查嗎？如果 Agent A 生成了錯的東西，我如何讓系統自己發現？」
  • 反向思考：Council of Agents 增加了延遲和複雜度。在什麼情況下，單一模型的速度優先於多模型的準確性？
Q3

Ruslan 說「投資 API-First 是最值得提前做的事」，monday.com 的早期開放平台讓 monday vibe 可以在幾天內 POC。你的組織或產品現在是否具備 AI-ready 的 API 基礎設施？最大的差距在哪裡？

查看建議思考方向
  • 歷史對照：Stripe 的 API-First 策略讓它成為了「開發者的最愛」。API-First 不只對 AI 有利，對整個生態整合都有利——現在的 AI 轉型只是把這個優勢放大了十倍。
  • 講者隱含論點：monday.com 早期做開放平台是為了「外部開發者生態」，但這個決策在 AI 時代獲得了意外的回報。這說明好的架構原則往往有「超出預期的使用案例」。
  • 職涯啟發：做一個評估：你的系統中，有多少功能只能透過 UI 操作，無法透過 API 呼叫？這些功能是 AI 整合的最大障礙。
  • 反向思考：API-First 也有代價——設計好的 API 需要時間，而且一旦公開就很難改變。在快速迭代的早期，什麼時候應該開始 API-First？
Q4

Ruslan 說「Principal Engineer 的劇本翻轉了——他們從代碼審查者變成了大量產出代碼的人，同時專注構建 AI Brains」。這對工程職涯發展有什麼影響？「AI Brains 工程師」是否會成為一個新的職位類型？

查看建議思考方向
  • 歷史對照：DevOps 工程師在 2010 年代初是新興職位，現在是每家公司的標配。Platform Engineer、ML Engineer 的出現都有類似的軌跡。AI Brains 工程師可能是下一個。
  • 講者隱含論點：AI Brains 需要「最了解公司業務和代碼庫的人」來建立——這本質上是在把「資深工程師的隱性知識」轉化為「可被 AI 使用的顯性結構」。
  • 職涯啟發：如果 AI Brains 成為關鍵基礎設施，那麼「最了解整個系統的工程師」的市場價值會比過去更高，還是更低？
  • 反向思考：如果 AI Brains 可以讓 PM 和 EM 直接交付原型，工程師的「獨特價值」到底是什麼？這個問題值得每個工程師認真思考。
Q5

Ruslan 每天深挖 AI 的「失敗個案」，視之為「新產品機會的金礦」。你認為這個思維轉換（從「修復失敗」到「失敗是機會」）對於 AI 產品開發有什麼深遠影響？

查看建議思考方向
  • 歷史對照：Lean Startup 的「失敗學習」理論、Google 的「Blameless Postmortem」文化，都是把失敗視為信息而非罪責。但 Ruslan 更進一步：失敗是「產品機會」。
  • 講者隱含論點：AI 產品的「失敗案例」往往揭示了「用戶真實需求和系統能力之間的差距」——這個差距就是下一個功能的靈感。
  • 職涯啟發：建立一個追蹤 AI 失敗的系統，並分類分析：哪些失敗是可以用更好的 Prompt 解決的？哪些失敗需要添加新工具？哪些失敗暗示了一個全新的產品需求？
  • 反向思考：如果你每天都在看失敗案例，你可能會被「現有系統的局限」所束縛，而不是思考「全新的可能性」。如何在「深挖失敗」和「仰望星空」之間取得平衡？
Q6

Ulrich 說「要讓每個團隊這個季度必須端到端用 AI 完成一個功能」——這是一個強制令（Mandate）。你認為這種「強制令」方式和「自願採用」方式相比，在推動組織 AI 採用方面各有什麼優劣？什麼情況下應該選哪種方式？

查看建議思考方向
  • 歷史對照：Amazon 的 API-First Mandate（Jeff Bezos 2002 年的備忘錄）是「強制令成功改變企業架構」的教科書案例。但強制令也可能引發抵制和表面合規。
  • 講者隱含論點：Ulrich 說「我們看到過去 12 個月工程師對 AI 的抵制」——強制令是在抵制氛圍下打破僵局的工具，而不是在積極採用環境下的工具。
  • 職涯啟發：強制令的關鍵是「強制嘗試，不強制成功」——Ulrich 說的是「必須嘗試」，不是「必須用 AI 做完所有事」。這個微妙的區別讓強制令更容易被接受。
  • 反向思考：如果強制令失敗了（團隊做了一個糟糕的 AI 功能就交差），它是否反而強化了「AI 不好用」的偏見？
