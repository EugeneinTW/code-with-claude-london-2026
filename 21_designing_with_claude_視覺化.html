I

時間線被壓縮了

Claude Code 讓工程師把「幾週的工作」壓縮到「幾個小時」。當工程瓶頸消失後，新的瓶頸出現了：PM 和設計師跟不上。Dan Carry 講述的，正是 Anthropic Labs 如何解決這個問題的故事。

Dan 在演講中做了一個即時調查：「你上一次交付的新功能，從有想法到出現在用戶面前，花了幾週？」台下的答案從 1 週到 10 週不等。然後他問：「一年前，你的答案是什麼？」答案普遍大幅延長。

這個壓縮是真實的。而壓縮帶來的新問題是：當工程師能在數小時內把功能推到用戶面前時，PM 和設計師的工作流卻仍然需要幾天甚至幾週——用來寫 PRD、開規劃會議、製作線框圖。他們成了新的瓶頸。

"Once Claude Code let engineering take off, the bottleneck shifted — from building the feature to figuring out the right thing to build." 「一旦 Claude Code 讓工程起飛，瓶頸就轉移了——從『建置功能』轉移到了『搞清楚到底該建置什麼正確的東西』。」

— Dan Carry, Anthropic Labs

Claude Design：讓非工程師也能快速視覺化想法的工具

傳統的產品設計流程：PM 寫需求文件 → 設計師畫線框圖 → 開會討論 → 再次修改 → 工程師開始開發。這個流程在工程速度飆升後變得尷尬：工程師在等 PM 和設計師。

  • Claude Design：讓你與 Claude 協作，直接建立設計原型、簡報、單頁企劃書
  • 核心功能：原型生成、多人即時協作、一鍵交接給 Claude Code 開發
  • 輸出格式：標準 HTML/CSS/JS（完全開放，可以在任何工具中繼續工作）

可以把它理解為「PM 和設計師的 Claude Code」——讓非工程師也能在幾分鐘內把想法變成可互動的東西，而不是等待工程師。

I

六大核心洞見

I

文件是不精確的，原型是具體的

兩個人看同一份 PRD，腦子裡想的是完全不同的產品。而原型消除了這種解讀分歧——你可以親手觸摸它、感受它、立刻知道它對不對。

II

為自己解決痛點 = 最好的產品設計

Multiplayer 和 Hand-off 功能，最初都是 Omni 自己的開發者解決自己工作流摩擦時做出來的。用戶第一天要的功能，正好就是團隊上週為自己做的那個。

III

協作成本是速度的隱形殺手

3 人小團隊的最大優勢不是人才密度，而是協作成本趨近於零。「討論」就是轉身跟旁邊的人說話，不需要排時程、不需要對齊會議。

IV

為「差一點就能行」的東西做原型

不要為現在已經行得通的東西花時間，而是為那個差一點就能行的東西做原型。因為下一個模型發布，就會把那個「差一點」填滿——Opus 4.7 解決了大量 Claude Design 早期無法解決的問題。

V

迭代的速度決定錯誤的代價

進階控制項功能走錯方向，但從發現到拔除只花了一週。「迭代循環的小」才是降低錯誤代價的關鍵，而非「不犯錯」。

VI

工具的目標是為所有人提升工藝水平

進階用戶愛的功能，一般用戶恨。真正好的工具應該讓所有人都能做出更好的作品，而不只是給少數人更多控制桿。

II

下注工廠：前沿實驗室裡的實驗室

Anthropic Labs 的定位是「前沿實驗室裡的實驗室」。它的核心機制是「下注工廠」：小團隊、快速實驗、加倍投入或果斷放棄。Claude Code、Claude Design、MCP、Skills，都從這裡誕生。

"We ship, observe, learn. Ship, observe, learn. We just run that loop over and over again." 「我們交付、觀察、學習。交付、觀察、學習。就是一遍又一遍地跑這個循環。」

— Dan Carry, Anthropic Labs

給初學者 · 一句話搞懂「精實創業」
精實創業（Lean Startup）：讓市場告訴你對不對

傳統創業方式：花 2 年開發完整產品 → 上市 → 發現市場不要 → 失敗。精實創業的核心是把這個失敗「提前」——做最小可行的版本、立刻給用戶用、快速知道對不對、調整再迭代。

  • Build-Measure-Learn（打造-衡量-學習）：精實創業的核心循環
  • Labs 的版本：Ship-Observe-Learn，強調「行動速度」
  • 最大差異：Labs 把這個循環跑得比任何人都快，10 週 50-100 次

Dan 說 Labs 不預測未來，而是「直接去做，讓現實告訴你真相」。這種態度在 AI 快速演進的時代尤其重要——你預測的「10 年後」可能 18 個月後就到了。

兩個最強問句

問用戶：「請盡情向我抱怨！」

Dan 的用戶訪談開場白。不問「你覺得這個功能怎麼樣」，而是直接邀請批評——因為抱怨蘊含著產品改進的機會，滿意的人不會告訴你什麼需要改。

問研究員：「最近被什麼驚艷到了？」

和 Anthropic 研究員交流的開場白。讓研究員主動說出他們對模型能力的新發現——這往往就是新下注機會的源頭，用戶痛點與模型新能力的交叉點。

給初學者 · 一句話搞懂「協作成本」
為什麼小團隊反而跑得更快？

想像你要做一個決定。1 人的話，你立刻決定；10 人的話，你需要開一個會、做一個簡報、等所有人對齊、處理各種意見。

  • 協作成本：溝通、對齊、等待、重複解釋的時間總和
  • 10 人團隊：可能有 45 條溝通路徑（n×(n-1)/2）
  • 3 人團隊：只有 3 條溝通路徑——彼此都瞭解對方在做什麼

Dan 說 3 人團隊裡的「討論」，就是轉身跟旁邊的人說話。這種即時協調是大團隊無法複製的優勢。

III

廢除 PRD：原型就是唯一語言

Dan Carry 寫了 20 年 PRD。Claude Design 上線後，他停止了。不是因為 PRD 不好，而是原型更好——它是具體的、可互動的、幾分鐘內可迭代的。

"Docs are imprecise. Two people can read the same doc and imagine two completely different products. Prototypes are concrete and tactile." 「文件是不精確的。兩個人看著同一份文件，卻各自想像出完全不同的產品。原型則是具體且直觀可感受的。」

— Dan Carry, Anthropic Labs
給初學者 · 一句話搞懂「PRD（Product Requirements Document）」
PRD 是什麼？為什麼它有問題？

PRD（產品需求文件）是 PM 用文字描述產品功能的文件。在沒有 AI 工具的年代，它是對齊整個團隊的唯一方式。

  • PRD 的問題一：認知偏差：「用戶可以點擊提交按鈕」——每個人想像的按鈕位置、顏色、邏輯都不一樣
  • PRD 的問題二：製作成本高：一份好的 PRD 要花 PM 幾天時間
  • PRD 的問題三：難以更新：一旦需求改變，文件就過時了，但沒人有時間更新
  • 原型的優勢：可以親眼看到、親手體驗，消除所有歧義；Claude Design 讓原型的製作時間從幾天壓縮到幾分鐘

Dan 的新工作流：對話 + 錄音 → 轉錄文字 → 丟給 Claude Design → 3 個原型變體 → 選一個繼續迭代。

Pitch-off 大賽的那個時刻

後半場 100% 的提案都是現場做的

Anthropic Labs 有一個叫做「Pitch-offs（提案大賽）」的儀式——大家在會議中提案，試著說服其他人加入自己的團隊。在引入 Claude Design 之前，大家只是上台口頭講，效果普通。

第一次用 Claude Design 做提案大賽的時候，事情變了。大賽進行到後半段，100% 的提案都是現場在會議中即時用 Claude Design 製作的原型——有人說出一個想法，另一個人受到啟發，立刻 Fork 出去做出新的版本，現場展示。

"The back half of pitches were 100% people building in real time with Claude Design. They were being made in the meeting. This was the pivotal moment that convinced us we should double down."

這個時刻讓 Dan 確認：這個工具真的能把「從想法到原型」的時間縮短到「在對話中發生」。這就是他們決定加倍投入的訊號。

兩個「為自己做的功能」

多人即時協作（Multiplayer）

起源：團隊分享設計時，每次都要先錄影片或架設分支。這個摩擦讓人不舒服。解法：讓多人直接在同一個設計上即時迭代。用戶第一天的需求，正好就是這個。

自動交接給 Claude Code（Hand-off）

起源：設計完成後，要手動把所有對話上下文重新輸入 Claude Code。太慢太繁瑣。解法：自動攜帶所有上下文、一鍵交接。用戶第一天問的第二個問題，正是這個。

給初學者 · 一句話搞懂「Dogfooding（吃自家狗糧）」
為什麼「用自己的產品」是最好的 QA？

「吃狗糧」是科技業的俚語，意思是讓你的員工在日常工作中使用自己公司的產品，而不只是向外部客戶販售。

  • 好處：員工是最嚴苛的測試者，他們知道競品的優缺點，也知道真實工作中的痛點
  • Labs 的做法：整個團隊用 Claude Design 來設計 Claude Design——每天在開發過程中遇到的痛點，就成為明天要修復的功能
  • 結果：Multiplayer 和 Hand-off 都是這樣誕生的

Dan 說：「如果你在做任何開發者工具，而你能用你自己的工具來改善這個工具本身，這是世界上最棒的情況。」

IV

最反直覺的洞見：為「差一點」的東西做原型

在 AI 快速進化的時代，有一個看似奇怪但極其重要的建議：不要去做那些「現在已經行得通」的東西，而是要為那些「差一點就能行」的東西做原型。

"Don't build prototypes for what already works. Build prototypes for what almost works. Because the next model might just solve the rest." 「不要為那些現在已經行得通的東西做原型。要為那些差一點就能行的東西做原型。因為下一個模型可能直接就解決了剩下的問題。」

— Dan Carry, Anthropic Labs
Opus 4.7 的潮水效應

等待新模型，比工程修復更快

在 Claude Design 早期開發中，有一大批問題讓工程師百思不得其解。他們沒有寫出精妙的工程架構去修復它們，也沒有透過什麼驚人的洞察去解決它們——他們等到了 Opus 4.7 的發布，問題就迎刃而解了。

"A tide that lifts all boats — model releases will do a lot of things for you."

這個故事的含義很深遠：在 AI 工具開發中，底層模型的進步速度往往比你的工程努力更快、更有效。如果你花了大量工程資源去「駭」一個當前模型無法原生支援的功能，很可能下一個模型就輕鬆原生支援了——你的工程投入全部白費。

因此，明智的策略是：找到「幾乎行」的地方，搞清楚產品的輪廓（Shape），然後讓模型進步來填滿剩下的缺口。

給初學者 · 一句話搞懂「產品的 Shape（輪廓）」
「輪廓」比「功能清單」更重要

想像你在設計一把椅子。功能清單是：4 條腿、1 個椅背、可以坐。但「輪廓」是：這把椅子讓人感覺舒適、看起來優雅、適合辦公桌旁邊。

  • 功能清單：可以列出，但可能做出糟糕的產品
  • Shape（輪廓）：產品應該給用戶帶來什麼感覺、解決什麼核心問題
  • 為什麼難：你只能通過迭代來發現正確的輪廓，無法在開始前就知道

Dan 說 Claude Design 99% 的價值來自 10 週的迭代，而非最初的想法。這個「99% 來自迭代」就是在反覆修正輪廓——直到找到那個「對的感覺」。

IV

進階控制項的失敗：為所有人服務

不是所有的失敗都是壞事——如果你的迭代循環夠快，失敗反而是最重要的學習。進階控制項的故事是 Dan 分享的最誠實的案例。

一週的失敗，而非一季的災難

進階用戶愛了，其他人棄了

早期開發中，Claude Design 加入了一套「極度精細的像素控制功能」——進階用戶可以對每一個元素進行精確操控。幾位最活躍的測試者超級喜歡，給了大量正面反饋。

但當深入挖掘使用數據時，現實是殘酷的：除了那幾位進階用戶，幾乎所有人都討厭這個功能。不只是不喜歡——是非常討厭。這些控制項讓人困惑，並且對產品體驗產生了主動的負面影響。

"We ripped all of them out in about a week. From discovery to ripping out: one week."

如果這是一個季度開發週期的專案，這個功能可能會讓整個季度的方向走偏。但因為迭代循環夠小，一週解決問題、繼續向前。

教訓：早期的熱情反饋可能來自最活躍的用戶，而非最有代表性的用戶。 數據才是最終裁判。

三個明天就能做的行動

行動 1 · 今天就可以做

下次想寫 PRD 的時候，直接跳過它

和團隊成員或 Claude 討論想法，把對話錄音並轉錄。討論重點放在「為什麼要解決這個問題」和「好方案應該具備什麼特徵」。然後把轉錄丟給 Claude（或 Claude Design），要求產出三種不同的設計原型。

行動 2 · 這週某個下午

建一個你一直想要的內部工具

找一件你在等待、想自動化、或能加速你工作流的事情。排除一切干擾，用一個下午把它做出來。你會發現速度快得讓你自己吃驚。「大家總是在等待工具」，但在 AI 時代，為自己建工具的速度已經是驚人的了。

行動 3 · 最難的一個

24 小時內交付一個真實用戶需求

挑一個真正的功能需求（不是修 Bug），在 24 小時內完成並推到用戶面前，然後索取反饋。你會赫然發現你們現有流程的所有絆腳石：部署方式、Code Review 流程、團隊對齊機制。這是最有啟發性的診斷工具。

給初學者 · 一句話搞懂「發布後 62 項改進」是怎麼做到的
「肌肉記憶」讓發布後的衝刺自然發生

Claude Design 週五發布，週一前 62 項改進。聽起來嚇人，但 Dan 說這「一點都不累、完全自然」。原因是：在過去 10 週裡，團隊每天都在跑 Ship-Observe-Learn 循環，這個速度已經成為「正常工作方式」，而不是「緊急衝刺」。

  • 不是英雄主義：62 項改進不是通宵熬夜的結果
  • 是肌肉記憶：10 週下來，快速回應反饋已經是團隊的預設工作模式
  • 教訓：「速度文化」需要時間建立；發布前的每天迭代，就是在培養這個能力

Dan 的說法是：「這對團隊裡的每個人來說都很自然，因為在過去 10 週裡，這就是我們每天的工作方式。」快速交付不是例外狀態，而是正常狀態。

V

深度思考題

從 Dan Carry 的演講出發，六個問題探討 AI 時代的產品管理、組織設計、以及工具設計哲學。

Q1：「文件是不精確的，原型是具體的」——但原型也可能是不精確的。一個互動原型可能讓用戶關注視覺效果，而忽略底層的邏輯問題。在什麼情況下，PRD 比原型更有價值？

查看建議思考方向
  • 歷史對照：設計圍欄效應（Streetlight Effect）——人們傾向於在「看得到」的地方找問題，而不是「問題實際發生」的地方。原型的高保真度可能讓人忽視系統設計的深層問題。
  • 講者隱含的論點：Dan 強調轉錄的內容應聚焦在「問題是什麼」和「好方案的特徵」，而不是「按鈕在哪裡」。這暗示原型最有用的場景是對齊「感覺」，而非規格化「邏輯」。
  • 職涯啟發：如果你在一個需要高度合規（例如醫療、法律、金融）的行業，「原型」和「規格文件」的比重應如何調整？
  • 反向思考：有時候「用文字強迫思考」的過程本身就有價值——它讓你必須把模糊的想法轉換成精確的語言。完全跳過 PRD，是否也跳過了這個強迫自我澄清的過程？

Q2：Labs 的「下注工廠」是成功的，但它生產出的產品（Claude Code、Claude Design、MCP）都是開發者工具。這是否和「開發者正是 Labs 的用戶」有關？這個模式對非技術型產品是否同樣適用？

查看建議思考方向
  • 歷史對照：Atlassian（JIRA、Confluence）的崛起——工程師為工程師做工具，因為自己最了解痛點。但 Atlassian 最終要把工具賣給非工程師（PM、設計師），這個過渡很困難。
  • 講者隱含的論點：Dan 說「我們用 Claude Design 設計 Claude Design」——工具設計師即工具用戶的閉環，是最高效的 QA 機制。但這個閉環的前提是「開發者和目標用戶重疊」。
  • 職涯啟發：如果你在做醫療 AI 產品，但你不是醫生，你如何重現這種「自己用自己的工具」的設計優勢？有沒有替代方案？
  • 反向思考：「開發者工具由開發者做」可能產生一種偏見——過度技術化、對新手不友善、功能過多。Claude Design 引入的「進階控制項」失敗，是否就是這個偏見的體現？

Q3：Dan 的「為差一點就能行的東西做原型」建議，背後的假設是「模型進步速度夠快，能填補缺口」。但如果你正在研發的東西並非 AI 優先，這個邏輯是否成立？

查看建議思考方向
  • 歷史對照：在手機 CPU 快速提升的時代，遊戲開發者常常「先做超高規格的圖形引擎，等硬體趕上來」。這個策略在 AI 時代的對應是什麼？有哪些不同？
  • 講者隱含的論點：這個建議的關鍵假設是：你押注的能力提升（模型智能）的速度和方向是可預測的。Anthropic 作為模型提供商，對自己模型的路線圖有獨特的信息優勢。外部開發者的情況如何？
  • 職涯啟發：如果你在創業公司負責 AI 產品規劃，你如何評估「哪些問題模型進步會解決，哪些需要工程手段解決」？這是否應該成為技術選型中的標準考量維度？
  • 反向思考：如果 Anthropic 的路線圖改變，原本「差一點就行」的功能可能永遠等不到模型進步來填補。押注供應商路線圖的風險如何 hedge？

Q4：3 人團隊的超低協作成本是 Labs 成功的關鍵之一。但當產品需要從「Research Preview」擴展到「企業級產品」時，小團隊模式能否持續？或者規模化必然帶來協作成本的膨脹？

查看建議思考方向
  • 歷史對照：Basecamp / 37signals 長期堅持小團隊哲學，但他們的產品沒有達到 Google 或 Meta 的規模。Netflix 的「高人才密度，少規則」文化在擴展到萬人後面臨的挑戰。
  • 講者隱含的論點：Dan 只談 Labs 的模式，沒有談「Labs 的產品如何被更大的 Anthropic 團隊接手和維護」。這個轉換過程是什麼樣的？
  • 職涯啟發：如果你被指派帶領一個從小團隊過渡到大團隊的產品，在哪個時間點、哪些「小團隊習慣」應該保留，哪些必須捨棄？
  • 反向思考：Claude 本身是 Labs 最大的「協作成員」。隨著 AI 能力提升，「1 人 + Claude」可能逐漸取代「3 人團隊」。Labs 的團隊規模是否會越來越小？

Q5：进階控制项的失敗揭示了「早期活躍用戶反饋的代表性偏差」。但如果你的產品目標用戶就是這些進階用戶（例如開發者工具），你如何避免過度優化某個子群體？

查看建議思考方向
  • 歷史對照：Vim 和 Emacs 是「進階用戶愛到極致、新手完全進不去」的工具。VSCode 選擇了不同的定位——對新手友好，但也可以足夠進階。Claude Code 是哪種定位？
  • 講者隱含的論點：Dan 說的「為所有人提升工藝水平」，意味著 Claude Design 的目標是「降低生產好設計的門檻」，而非「讓設計師更快達到極致精細控制」。這是截然不同的產品定位。
  • 職涯啟發：在 Beta 測試設計中，如何確保你的早期用戶群能代表目標市場，而非只是最熱情的「科技愛好者」？有哪些具體的抽樣策略？
  • 反向思考：有時候「先服務好進階用戶，再慢慢下放到大眾」是正確的策略（例如 iPhone 初期定位高端）。進階控制項的失敗是否只是「時機不對」，而不是「方向不對」？

Q6：Dan 說 Claude Design 匯出的是標準 HTML/CSS/JS，因為「這些設計是屬於你的」。這種最高開放度的策略，是否會讓用戶「用完就走」，損害產品的留存率？開放性和黏著性如何平衡？

查看建議思考方向
  • 歷史對照：Figma 的開放 API 和插件生態讓它成為設計工具的標準，開放性反而創造了更強的網絡效應。Apple 的 App Store 採取相反策略——封閉生態創造護城河。在 AI 工具時代，哪個模式更適用？
  • 講者隱含的論點：Dan 說 MCP 整合將讓「任何設計工具」都能接入 Claude Design。這暗示他們的護城河不是「鎖住用戶的數據」，而是「讓 Claude 成為所有工具的最佳大腦」。
  • 職涯啟發：如果你在 B2B SaaS 公司，你如何設計數據可攜帶性（Data Portability）政策？高度開放是否反而是贏得企業客戶信任的最佳策略？
  • 反向思考：Claude Design 的開放輸出格式（HTML/CSS/JS）意味著用戶可以直接把設計帶到競爭對手的工具中。這個風險是否被「吸引更多用戶」的收益所抵消？Anthropic 的商業模式如何在這個前提下成立？
