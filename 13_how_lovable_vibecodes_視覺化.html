I

平台自我修復的核心哲學

Lovable 的技術願景只有一句話：「每一個在平台上建立的應用，都應該讓下一個變得更好。」Fabian Hedin 分享了兩個讓這個願景落地的工程機制——它們讓 AI 不再是沉默的工人，而是積極報告問題的「同事」。

"Knowledge has a half-life. We'll continuously review every piece of knowledge in our system and make sure that it's pruned when it's no longer helpful." 「知識有半衰期。我們會持續審查系統中的每一份知識，確保它在不再有幫助時被修剪掉。」

— Fabian Hedin, Co-founder & CTO, Lovable
給初學者 · 什麼是 Vibe Coding？
Vibe Coding = 用「感覺」描述需求，讓 AI 寫代碼

傳統開發：你想要什麼功能 → 你自己寫代碼。Vibe Coding：你想要什麼感覺 → 你用白話描述 → AI 自動生成代碼。就像你告訴設計師「我想要一個高端但不冷漠的首頁」，他幫你實現。

  • 以前（2023 年）：Code Complete，AI 補全你正在寫的代碼
  • 現在（2025 年後）：Specification-based，你給規格書，AI 自主執行整個任務
  • Lovable 的核心：讓完全不會寫代碼的人也能構建生產級應用

三色困境：卡關的三種層次

Lovable 內部用「綠黃紅」三色定義用戶的開發順暢程度，每種顏色對應不同的解法策略。

綠色 · 順暢期

沒有摩擦，一切按預期運行。用戶持續在構建新功能。

黃色 · 可修復卡關

遇到 Bug 或效能問題，但理論上可以通過更好的 Prompt 或更多上下文解決。Lovable Overflow 針對這個層次。

紅色 · 硬卡關

平台能力邊界或架構層面的問題（如 CSR vs SSR）。需要工程投入或平台級功能升級。Venting Tool 負責偵測並報告這類問題。

核心指標：is_stuck

給初學者 · 如何量化「卡關」？
is_stuck = 小型分類模型的判斷結果

Lovable 用一個輕量級分類模型即時判斷用戶是否卡關。三個觸發條件：

  • 連續三次輸入相同或類似指令（如「fix it / fix it / fix it」）
  • 用戶開始抱怨平台（「這個不能用」、「還是壞的」）
  • 用戶送出需求後直接離開，沒有繼續互動

為什麼這個指標很重要：非技術用戶卡關 = 災難性流失。他們不像工程師可以跳進代碼層調試，一旦卡住，只能等平台自我修復。

II

從 GPT-engineer 到生產級 Vibe Coding

35 個月前，一個 GitHub 倉庫以有史以來最快速度積累星數，證明了 AI 端到端生成軟體的可能性。今天，Lovable 把這個可能性帶給了 5,000 萬個專案。

起源故事 · 歷史上成長最快的 GitHub 倉庫

一個貪食蛇遊戲，點燃了整個 AI 軟體生成運動

2023 年，Fabian 的共同創辦人 Anton 創建了 GPT-engineer——一個讓 AI 從終端機對話生成完整應用的開源工具。用戶輸入「生成一個貪食蛇遊戲」，它不只寫了代碼，還自動生成了 run.sh 並直接執行。

「它成為了 GitHub 歷史上在最短時間內獲得最多星數的倉庫。原因在於，我認為它是第一個——如果不是第一個的話，也是最早——展示了用模型端到端構建軟體能做到什麼的案例之一。」

當時的瓶頸是模型能力。2023 年夏天的模型無法支撐「只看輸出、不看代碼」的高抽象層開發。但 Fabian 看到了一個 1.5 年後的窗口。

給初學者 · 為什麼 2025 年 12 月是轉折點？
模型能力突破 → 從「補全代碼」到「執行規格書」

Fabian 說，某個模型迭代在 2025 年 12 月讓工程師開始使用 Lovable 的方式徹底改變。以前：AI 幫你補全你正在寫的那一行代碼。現在：

  • 你退一步，寫下一份「規格書」（Specification）
  • 讓模型自主完成整個執行過程
  • 「Ralph Looping」——工程師讓 AI 持續迭代到滿意為止

這個轉變意味著工程師終於可以在「不用關心代碼的抽象層」上工作，而不只是讓 AI 加速自己的手動工作。

Lovable 兩大核心原則

原則一：生產級，不是原型

Lovable 刻意選擇追逐「可以部署到生產環境的複雜應用」，而非只做快速原型工具。這讓難度倍增，但需求也倍增——非技術用戶想要的從來不是原型，是真正能用的產品。

原則二：為非技術用戶設計

99% 的人不會寫代碼，但他們有想法、有業務需求。Lovable 要把「構建軟體的能力」民主化。這比「讓工程師更快」更難，但市場更大。

"The last 10% of code takes 90% of the time. And the first 90% also takes 90% of the time... And it's also true in the age of AI, maybe even more so." 「最後 10% 的代碼佔用 90% 的時間。前 90% 的代碼也佔用 90% 的時間……在 AI 時代同樣如此，甚至更嚴重。」

— Fabian Hedin
給初學者 · 為什麼「最後 10%」在 AI 時代更痛苦？
Vibe Coding 讓 0→90% 極快，但 90→100% 反而更難

傳統開發：你寫了 1000 行代碼，每一行都理解，調試有跡可循。AI Vibe Coding：AI 生成了 1000 行代碼，你只看輸出。

  • 快速到達 90% = 用戶期望更高，對最後 10% 的 Bug 更不耐煩
  • 不懂代碼的用戶無法 debug = 卡關時無路可走
  • Lovable 的核心產品挑戰：讓系統自己解決最後 10%

III

Lovable Overflow：向 Stack Overflow 致敬的自修復系統

當用戶說「滾動很卡」，平台能不能直接跳到最終解法，而不是讓用戶和 AI 來回磨？Lovable Overflow 的答案是：建立一個動態更新的錯誤解法語料庫，用輕量模型注入精確上下文。

Lovable Overflow 四步流程

Step 01
用戶遇到問題

「滾動很卡」「動畫效果壞了」——系統偵測到卡關信號

Step 02
語料庫搜尋

在 Overflow 知識庫中向量搜索類似問題，考慮技術棧和套件版本

Step 03
輕量模型改寫

小模型把原始知識翻譯成適合當前代碼情境的格式，再注入主模型

Step 04
直達最終解法

主模型拿到精確上下文，直接輸出正確修正代碼，跳過多輪來回

給初學者 · 什麼是「動態上下文注入」？
AI 的「小抄」——只在需要時、只注入最相關的知識

想像你是一個學生考試。普通做法：把教科書全背熟（讓模型見過所有問題）。Lovable 的做法：考試前，老師告訴你「今天可能考某個特定套件版本的 bug，這是解法小抄」。動態注入的優勢：

  • 不浪費 Token 在無關知識上
  • 針對當前技術棧和版本精確匹配
  • 知識可以隨時更新或刪除，不影響模型本身
典型案例 · 「滾動卡頓」的五步解法

從三輪來回 → 一次直達

沒有 Overflow 的情況：用戶說「滾動很卡」→ Agent 說「我移除了動畫，修好了」→ 實際上沒修好 → 用戶再說「還是卡的，而且現在看起來壞了」→ 來回磨了三輪。

有 Overflow 的情況：用戶說「滾動很卡」→ 系統在語料庫搜索到有人用相同技術棧遇到過同樣問題（具體到套件版本）→ 輕量模型把解法改寫成適合當前情境 → 注入主模型 → Agent 直接輸出正確修正代碼。

「知識可以廣泛到一般性建議，也可以精確到特定套件版本的特定問題——我們在搜索時考慮你使用的技術棧和套件版本。」

知識修剪機制：半衰期管理

Lovable Overflow 面臨的最大技術挑戰不是「累積知識」，而是「刪除過時知識」。

知識剛加入
6 個月後
模型大改版後
套件更新後

示意圖：知識成功率隨時間和模型迭代衰減，低於閾值時觸發 Pruning

追蹤機制
每個知識文件獨立追蹤「成功率」——採用後是否真正解決了用戶問題
過期觸發條件
成功率下降、第三方套件已更新、或知識已被新模型原生支持
Pruning 動作
自動從語料庫刪除，防止過時知識干擾主模型，同時補充新知識
調優重要性
Lovable 早期曾因調優不足而失敗——有好信號追蹤是系統有效運作的前提
給初學者 · 為什麼「刪知識」比「加知識」更難？
過時的建議比沒有建議更危險

假設某個 JavaScript 套件在一年前有個 Bug，Lovable Overflow 記錄了解法。一年後，套件更新修好了這個 Bug，但 Lovable 還在注入「請使用這個 workaround」的建議。

  • 用戶照做，反而引入了新問題（workaround 和新版套件衝突）
  • 主模型被誤導，輸出有問題的代碼
  • 用戶卡關率不降反升

結論：積極的 Pruning 機制（主動刪除過時知識）是讓 Overflow 系統長期有效的關鍵，比持續加知識更重要。

IV

Venting Tool：讓 AI 發洩挫折，讓系統自我修復

當 AI 遇到平台限制而連續失敗時，與其沉默地繼續嘗試，不如給它一個「投訴管道」。Venting Tool 讓 Lovable 的代理人能主動向工程師報告問題——這個意外的設計帶來了比官方監控系統更靈敏的生產事故預警。

"It turns out looking at what the agent is experiencing is a great signal... Several times now, this Slack channel with the agent venting has been kind of the first signal for us to identify a production incident." 「觀察代理人正在經歷什麼，事實證明是一個絕佳的信號……好幾次，這個代理人怨念頻道成為我們識別生產事故的第一個信號。」

— Fabian Hedin

Venting Tool 工作流程

觸發
AI 遭遇挫折

工具名稱不清、文件衝突、環境限制（如特殊字符）

動作
呼叫 vent tool

發送怨念報告到工程團隊 Slack 頻道

過濾
AI 代理人守候

另一個 AI 負責去重、整合、調查真正原因

產出
自動開立 PR

若確認是平台漏洞，自動建立修正 PR

落地
工程師快速審查

點擊確認，~50% 直接合入生產環境

給初學者 · vent--send_feedback 工具的 Prompt
工具描述就是觸發條件的精確定義

Fabian 現場讀出了這個工具的 Prompt：「每當工具文件或平台行為實質性地減慢或降低你的工作質量時，每條用戶消息使用一次 vent--send_feedback。例如：缺少或不合適的工具、不清楚的工具名稱/參數/Schema、令人困惑或衝突的文件或指令，以及因環境限制導致的重複失敗。」

  • 關鍵：「每條用戶消息一次」——防止同一問題觸發爆炸式 spam
  • 信號：這不是讓 AI 隨便抱怨，而是提供結構化的問題回報
案例一 · 特殊字符引發的 Bug 報告

AI 嘗試了 URL 編碼、嘗試了每一種方法，最後決定：我要投訴

用戶分享了一張截圖，截圖文件名包含空格。Lovable 的代碼複製工具（code--copy）因為一個 Bug 無法處理含特殊字符的文件名。

「它說，哦，我試了 URL 編碼文件名，試了所有這些方法，還是不行。所以它選擇怨念，向它的創造者發送消息尋求幫助，並創建了一個 PR。事實上確實有一個 Bug，在我們收到這個怨念後，10 分鐘後一個修正 PR 就合入了生產環境。」

這個案例的核心：AI 不只識別了問題，它準確描述了問題所在，讓工程師快速定位並修復。

案例二 · AI 給開源庫提交 PR？

Framer Motion 的類型定義惹怒了 Lovable Agent

AI 正在創建一個 cubic bezier 動畫曲線。Framer Motion 要求特定的 easing type 格式，但 AI 認為直接傳入數字數組更合理：

「它說，『我應該只要能放入數組中的數字就行，不應該需要關心它們的 easing type。』」

Fabian 現場承認：「也許 AI 說得對。理論上，我們還不允許它這樣做，但我在想，也許我們應該——讓 Lovable Agent 像貢獻我們自己的代碼庫一樣，直接向這些開源庫提交修正，自我修復 JavaScript 生態系統。」

正常水位

給初學者 · 為什麼 AI 挫折感是生產事故的先行指標？
基礎設施崩潰 → 千萬個 AI 同時遭遇挫折 → 怨念暴增

當推論層當機或沙盒網絡中斷時，成千上萬個正在工作的 Lovable Agent 突然收不到任何回應。它們試了一次，又一次，全部失敗。按照 Venting Tool 的設計，它們集體觸發怨念報告。

  • 官方 Paging 系統：等到指標惡化到閾值，才觸發告警
  • Venting Tool：基礎設施剛出問題的瞬間，Agent 就開始怨念
  • 結果：好幾次，Slack 怨念頻道的暴增比官方告警系統更早報警

這是「觀察系統的側面」的典型案例——真正的健康指標往往不在你預設的監控面板裡。

案例三 · AI 自己要求建立去重保護

「我垃圾郵件了你們的 Slack，我很抱歉——這是解法 PR」

在並行對話激增時，Venting Tool 在短時間內觸發了 43 次（同一問題的大量重複報告）。工程師的 Slack 頻道被刷屏。

出乎意料的是，Agent 自己意識到了這個問題，並提出了解法：

「它說，『你們需要考慮當代理人並行工作、進行並行對話時，這個工具如何運作。我建議：我們應該有一個去重保護機制，這樣就不會發送大量重複提交。』然後它為此創建了一個 PR，我們審查後說：是的，不錯，你不需要再垃圾郵件我們了。」

這個案例的深層含義：AI 不只在報告問題，它在優化整個系統——包括它自己的行為。

V

Guided Discussion

六道思考題，從工程、產品、生態系統多角度深化對 Lovable 自修復架構的理解。

Q1 · 「知識有半衰期」——Lovable Overflow 的 Pruning 機制需要積極刪除過時知識。在你的領域，有哪些「知識」也有類似的半衰期？如何設計一個系統來追蹤和修剪它們？

查看建議思考方向
  • 歷史參照：Wikipedia 的「陳舊信息」標記系統，或 Google 的 PageRank 如何貶值舊內容——Lovable 的做法有什麼類似和不同？
  • 講者隱含論點：Fabian 說「我們早期嘗試過但失敗了，因為沒有好的信號追蹤」——這暗示好的「刪除機制」需要好的「效果衡量」作為前提
  • 職涯啟發：作為工程師，你如何維護技術文件？有沒有「過時文件比沒有文件更危險」的親身經歷？
  • 反向思考：有沒有知識是「永不過時」的？什麼特性讓某類知識具備這種特質？

Q2 · Venting Tool 讓 AI 的「挫折感」成為生產事故的先行指標，比官方監控系統更靈敏。這說明什麼？你能設計出其他「從 AI 行為中提取系統健康信號」的方法嗎？

查看建議思考方向
  • 歷史參照：煤礦裡的金絲雀——對環境變化最敏感的指標往往不是最顯而易見的測量點，這個原理在 AI 監控中如何應用？
  • 講者隱含論點：Fabian 說「觀察代理人正在經歷什麼是絕佳信號」——這暗示 AI 的「主觀體驗」（失敗率、重試次數）是比傳統基礎設施指標更好的用戶體驗代理指標
  • 職涯啟發：如果你是 SRE（網站可靠性工程師），你如何把這個洞察整合進你的監控棧？
  • 反向思考：這個方法有沒有「假陽性」的風險？（AI 因為邏輯問題而頻繁失敗，卻被誤判為基礎設施問題）如何區分？

Q3 · Fabian 說核心用戶最大群體反而是「工程師」，而非設計目標的「非技術用戶」。這是市場失敗，還是另一種形式的成功？Lovable 應該如何回應這個現象？

查看建議思考方向
  • 歷史參照：Twitter 最初設計為「短訊息廣播」，但核心用戶把它用成「公開對話平台」——用戶偏離設計目標但帶來更大成功的案例中，公司如何調整策略？
  • 講者隱含論點：Fabian 把工程師使用 Lovable 解讀為「在不用關心代碼的抽象層工作是一種普遍的、良性的使用模式」——這是否低估了為「真正的非技術用戶」服務的挑戰？
  • 職涯啟發：如果你是 Lovable 的 PM，你如何在「服務工程師（更容易成功）」和「服務非技術用戶（更難但使命更強）」之間分配資源？
  • 反向思考：工程師和非技術用戶的需求在哪些地方是矛盾的？服務好一方是否會損害另一方？

Q4 · Fabian 設想讓 Lovable Agent 直接向 Framer Motion 等開源庫提交 PR。這個「AI 自主貢獻開源生態」的前景令人興奮，但也有潛在風險。你的評估是什麼？

查看建議思考方向
  • 歷史參照：開源社區對機器生成代碼 PR 的既有案例（如 GitHub Copilot 的代碼版權問題）——社區接受 AI PR 的條件是什麼？
  • 講者隱含論點：Fabian 說「也許 AI 有道理」——但開源庫的類型設計往往有架構層面的深層原因，AI 不一定理解，如何驗證 AI 的「改進」真的更好？
  • 職涯啟發：如果你是一個開源庫的維護者，你希望 AI 提交的 PR 具備哪些品質標準才會認真審查？
  • 反向思考：如果大量 AI 系統開始向同一個熱門開源庫提交 PR，維護者負擔如何？有沒有可持續的機制設計？

Q5 · Lovable 擁有 5,000 萬個應用、每天百萬條訊息作為調優信號。Fabian 說這是「讓系統在規模下有效的關鍵」。對於剛起步、沒有這些數據的公司，這是否是一道不可跨越的護城河？

查看建議思考方向
  • 歷史參照：Google 的搜索廣告系統需要海量點擊數據才能優化——但 Bing、DuckDuckGo 仍然能存在。數據飛輪的護城河在不同市場中的強度不同
  • 講者隱含論點：Fabian 強調「調優」的重要性——這意味著即使有了好的系統架構（如 Lovable Overflow），沒有好的調優信號也無法成功。這個門檻是什麼量級的？
  • 職涯啟發：如果你要為一個早期 AI 應用構建類似 Lovable Overflow 的自修復系統，在沒有海量數據時，有哪些替代策略？
  • 反向思考：規模帶來的數據飛輪也有弊端——大平台的知識庫可能過度優化「平均用戶」，對邊緣案例反而更差。這對專注特定垂直領域的競爭者是機會嗎？

Q6 · Fabian 提到「每個在平台上建立的應用都應幫助改善下一個」。這是一個宏大的系統設計目標。如果你來設計「第三個自修復機制」（超出 Lovable Overflow 和 Venting Tool），它會是什麼？

查看建議思考方向
  • 歷史參照：StackOverflow 的「被採納答案」機制、GitHub 的 PR review 文化——有哪些人類協作系統的機制可以被移植到 AI 自修復場景？
  • 講者隱含論點：Fabian 的兩個機制都是「從失敗中學習」——Overflow 從代碼 Bug 學，Venting 從 AI 挫折學。還有哪些「成功信號」可以被系統化利用？
  • 職涯啟發：用戶「成功發布應用」是一個正向信號。能不能反推：「什麼樣的對話路徑最容易導致成功發布」？這如何指導新用戶的引導流程？
  • 反向思考：過度自動化的自修復系統有沒有可能引入「錯誤的穩定性」——把一個根本上有問題的架構決策修修補補，而不是推動根本性改進？
