給初學者 · 一句話搞懂「PR 頻率」作為生產力指標
為什麼 PR（Pull Request）數量能反映開發速度？

PR（Pull Request）是工程師把一段代碼提交給團隊審查、準備合併進主分支的動作。PR 數量越多，代表提交了越多完整的代碼修改。

  • 傳統比喻：PR 數量 ≈ 工廠每天下線的成品數量
  • Spotify 的 PR 頻率提升 76% = 代碼產出速度翻了將近一倍
  • 但注意：PR 多不等於品質好——下文會說明這帶來的新挑戰

PR 頻率是一個代理指標（Proxy Metric），可以快速衡量開發速度，但需要與其他品質指標一起看。

代碼增長比工程師快七倍

AI 出現前，Spotify 就已面臨一個嚴峻的問題：代碼庫的增長速度是工程師招募速度的七倍。技術債在不斷累積。

工程師在那份年度工程調查中給出的最大抱怨？就是這些維護性的遷移工作。

II

Fleet Shift：自動化的基礎設施

Spotify 建立了一套名為「Fleet Management」的自動化系統，讓機器負責那些「重要但無聊」的維護工作。

建立

Fleet Shift：大規模代碼變更引擎

能跨數千個 Repo 批量執行代碼修改。輸入：一個變更腳本。輸出：數千個 PR，自動測試，自動合併。

限制

確定性腳本的天花板：Hyrum's Law

「代碼有非常非常寬的 API Surface。呼叫同一個方法有無數種寫法。當你把腳本跑過百萬行代碼，你會發現所有的邊緣情況，都必須在腳本裡處理。」

突破

用 LLM 取代確定性腳本

「確定性腳本」的問題：需要在腳本裡寫滿無數 if-else 處理邊緣情況。LLM 的優勢：理解語意，即使代碼寫法千變萬化，也能辨別「它們的意圖是相同的」。

核心故事 · Hyrum's Law 的挑戰

為什麼 Google 工程師的一條定律讓 Spotify 頭疼了好幾年

Hyrum Wright 在 Google 工作時發現了一個規律，後來被稱為「Hyrum's Law」：

「當一個 API 有足夠多的使用者時，你在合同中承諾的行為並不重要——所有可觀察的行為，不管是否在設計規範中，都會被某個使用者依賴。」

對 Spotify 的實際意義：要把全公司後端的 gRPC 呼叫方式從舊 API 改成新 API，理論上很簡單。但實際上，這個「呼叫」在四千萬行代碼中存在成百上千種寫法——因為每個工程師都有自己的風格，有些甚至利用了 API 未文檔化的行為。

確定性的遷移腳本要處理這些情況，需要無窮無盡的 if-else。LLM 則可以「理解語意」——即使寫法不同，能判斷「這十幾個地方雖然長得不一樣，但它們在做同一件事」，並給出正確的變更。

啟示：AI 最強大的地方不是「執行規則」，而是「理解意圖」——這正是確定性腳本永遠無法完全做到的事。

II

兩種合併策略

自動化的核心是分流——不是所有 PR 都需要人類看，識別哪些可以完全自動化，是提升整體速度的關鍵。

🤖 Auto-merged（自動合併）

條件：自動化生成 + CI 通過 + 無邏輯影響

例如：依賴版本更新、格式修改、安全掃描修復、標準化配置調整

→ 工程師甚至不知道這個 PR 存在

👤 Human Review（人類審查）

條件：涉及核心商業邏輯、架構決策、安全敏感代碼

例如：新功能實作、API 界面變更、資料庫 Schema 修改、認證邏輯

→ 人類精力集中在真正重要的決策

目前 Fleet Shift 產生的 PR 中，「綠色（自動合併）」佔絕大多數。Niklas 說：「這些事情每天都在發生，我們每天都在出貨這些東西。」

III

Honk：Spotify 的 AI 工程師

當確定性腳本遇到瓶頸，Spotify 把 Claude 接進了 Fleet Shift——誕生了「Honk」，一個能理解語意、自我驗證、大規模並行的 AI 代理人。

名字傻，圖示也傻（一隻鵝），但如 Niklas 說的：「It's a very useful tool, as it turns out.」（事實證明，它是一個非常有用的工具。）

Claude
核心推理層 · Anthropic Claude（Agent SDK）· 語意理解 + 代碼生成 + 自我修正
Honk Harness
封裝層 · 內部 Harness 包裹 Agent SDK · 存取驗證工具集（CI 建置、多 OS 測試）
Kubernetes
排程層 · 每個 Honk 任務跑在一個 Kubernetes Pod · 可大規模並行，在雲端環境調度
Fleet Shift
協調層 · Fleet Shift 負責跨 Repo 的工作排程 · Honk 坐在中間執行實際代碼修改
給初學者 · 一句話搞懂「Honk 的架構」
像一間大工廠的生產線

想像 Honk 是一個工廠的生產線系統：

  • Fleet Shift（工廠排程系統）：決定今天哪 39 個 Repo 要升級 Java
  • Kubernetes（工廠的生產線）：同時開啟 39 條並行生產線，每條負責一個 Repo
  • Honk Harness（每條生產線的控制器）：管理工具存取、記錄狀態
  • Claude（每條生產線上的工人）：理解意圖、寫出正確代碼、執行驗證

整個系統讓「一個工程師管理 39 個並行 AI 工程師」成為可能。

III

驗證工具：Honk 的可靠性基石

Honk 之所以能「自動合併」，是因為它有能力驗證自己的工作——跨多個作業系統執行真實的 CI 建置和測試。

Spotify 的客戶端 App（iOS、Android、Desktop）跑在不同作業系統上。Honk 的驗證工具能：

01

在 CI 環境中建置代碼

觸發真實的 CI 流程，確保修改後的代碼能成功編譯。

02

跨 OS 執行測試

在 iOS、Android、Desktop 環境分別跑單元測試和整合測試，確保跨平台正確性。

03

自動修復然後重驗

測試失敗 → 分析錯誤 → 修改代碼 → 重新驗證 → 形成驗證迴圈直到成功。

量化成果 · Java 遷移的三天奇蹟

從「數百個團隊花數月」到「一個工程師三天搞定」

Spotify 的後端主要跑在 Java 上。每一次 Java 版本升級，過去的流程是：

「發遷移指南給幾百個團隊，說：『你們需要把你們的 Component 從這個 Java 版本升到那個版本。』團隊會去做，但這通常需要幾個月才能完成整個升級——這對所有人來說都不好玩。」

引入 Fleet Shift + Honk 後，最近一次的 Java 遷移只花了三天，且由單一工程師主導——其他人完全無感，那些自動合併的 PR 靜靜地完成了它們的任務。

啟示：「三天 vs 數個月」不只是速度提升，更是組織摩擦力的根本消除。數百個團隊不需要協調、不需要排程、不需要「記得去做」——這類組織協作成本往往比技術成本更高。

Honk v2：多人協作的 AI 工作坊

開發者自發把 Honk 帶進了 Slack——這推動了 Spotify 在 Hack Week 推出 Honk v2，引入多人協作的概念。

Honk v2 在演講當週的 Hack Week 剛發布 Alpha 版本。核心新功能：

🎭 Chirp 編排引擎

Spotify 內部的 Agent 編排工具，讓你同時調度和協調大量 Agent 會話。比 Agent SDK 有更多 Spotify 基礎設施整合。

👥 Multiplayer：共享 Agent 會話

多位開發者可同時共享同一個 Honk 會話——就像 Google Docs，但是協作 AI 代理人。一個大型專案，多人 + 多個 Honk 並行協作。

給初學者 · 一句話搞懂「Multiplayer AI Sessions」
從「一人一個 AI 助手」到「一個 AI 助手服務整個團隊」

目前大多數 AI 工具是「一個人對一個 AI」的關係，就像單人遊戲。Multiplayer 把它變成了多人遊戲。

  • 傳統：Alice 有她的 Claude，Bob 有他的 Claude，兩個人不能分享上下文
  • Multiplayer：Alice 和 Bob 共享同一個 Honk Session，都能看到對話歷史，都能給指令
  • 應用場景：兩個工程師一起用一個 AI 解決同一個架構問題，即時協作

這是「Claude 作為協作工具」的下一步進化——從個人工具變成團隊基礎設施。

"Imagine a Google Docs or something similar, but for Claude. You can have many sessions with Honk where you collaborate over working towards whatever that goal is." 「想像一個類似 Google Docs 的東西，但是是給 Claude 用的。你可以有多個 Honk 的 Session，大家一起協作達成目標。」

IV

十五年的「克制哲學」

Spotify 在 Niklas 加入之前就確立了一個信念：「使用的技術越少，前進的速度就越快。」這個決定在 AI 時代產生了意想不到的複利效應。

"The fewer technologies that we use, the faster we will be able to go." 「使用的技術越少，我們能前進的速度就越快。」

— Spotify 工程文化核心信念

這個信念來自三個實用的理由：

深度

深度專精 vs 廣泛涉獵

對少數技術達到極深的專精，能建構出更高品質的東西——而不是每個工具都只懂個七成。

決策

消除微小決策的認知負擔

技術選型是有認知成本的。當有「標準答案」存在，團隊可以把腦力用在真正的問題上，而不是「這個用 GraphQL 還是 REST？」

協作

跨團隊協作的摩擦力消除

如果所有後端服務都長得差不多，工程師換組後上手時間幾乎為零。代碼審查也更容易——看的人理解背景。

給初學者 · 一句話搞懂「技術標準化」的 AI 紅利
Claude 在一致的代碼庫裡表現顯著更好

想像你要教一個學生解數學題。如果每題格式都不一樣，他需要先弄清楚題型再作答。如果所有題目格式都一樣，他可以直接解題。

  • 標準化代碼庫：Claude 的 Context 質量高，出錯率低
  • 碎片化代碼庫：Claude 在不同服務間的表現有明顯落差
  • Niklas：「我們確實有代碼庫較碎片化的地方，能看到 Claude 表現明顯下降」

標準化是「免費的 AI 效能提升」——不需要更好的模型，只需要更一致的代碼庫。

IV

Backstage：開發者的單一窗口

Backstage 是 Spotify 開源給全世界的開發者入口網站（Developer Portal），也是他們讓 AI Agent 能「看懂整個組織」的基礎設施。

在 Backstage 誕生前，Spotify 有大約一百個獨立工具——部署監控一個、CI 狀態一個、AB 測試一個，每個都很難用。更麻煩的是：如果發生事故，想知道「這個服務是誰負責的」都很困難。

給初學者 · 一句話搞懂「Backstage」
就像公司的「內部 Google」+「軟體電話簿」

Backstage 有兩個核心功能：知道所有軟體的狀態，以及知道每個軟體是誰負責的。

  • 軟體目錄：公司所有服務的清單，含擁有者、依賴關係、健康狀態
  • 工具整合：部署、CI、AB 測試、文件——全都在一個介面
  • Soundcheck：讓團隊自我評估是否符合技術標準
  • AI 整合：把 Backstage 功能封裝成 MCP，讓 Claude 可以直接查詢和操作

Backstage 已開源，目前是業界主流的開發者門戶框架，被數百家公司採用。

📡 Golden State（黃金標準）

定義「這種類型的服務應該長什麼樣子」的最佳實踐模板。後端服務、iOS View、Android Fragment——每種都有推薦的技術棧和設計模式。

✅ Soundcheck（自我評估）

讓每個團隊可以在 Backstage UI 中自我評估：我的服務是否符合 Golden State？有沒有未定義擁有者？有沒有使用被廢棄的技術？

IV

Lint 作為 AI 的即時反饋系統

與其事後審查 Claude 是否遵循了標準，不如讓標準以「即時反饋」的形式嵌入開發流程——讓 Claude 在犯錯的瞬間就能自我修正。

Spotify 把 Golden State 的要求轉化成靜態分析規則（Lint Rules）。當 Claude 在代碼庫中工作時：

情境

Claude 寫出不符合規範的 gRPC 呼叫方式

假設 Spotify 的基礎設施要求用特定方式呼叫 gRPC，Claude 不知道，用了一個在業界常見但對 Spotify 基礎設施次優的方法。

觸發

Lint 系統立刻回饋

靜態分析工具掃到這個模式，產生具體的錯誤訊息：「偵測到不推薦的 gRPC 呼叫方式，建議改用 XXX。」

修正

Claude 自動閱讀反饋並修正

Claude 看到 Lint 錯誤，根據提示修改代碼，重新通過 Lint 檢查，完全無需人類介入指導。

Niklas 說：「這是一個令人驚嘆的方式來推動這種標準化。每次我在我們的代碼庫中使用 Claude，我都會看到 Claude 不斷觸發這些 Lint 檢查然後自我修正。」

實踐洞見 · CEO 原型設計的革命

當任何人都能在生產代碼庫中建立原型

Spotify 透過把內部工具封裝成 MCP 和 Skills，讓 Claude 能在真實的生產 Mono-repo 中操作。這帶來了一個意想不到的結果：

「現在在 Spotify，你可以打開我們客戶端 Mono-repo 中的 Claude，透過一套 Skills 和我們建立的基礎設施，提示 Claude 建立你想要嘗試或想象的任何功能。Claude 會幫你建立。你會得到一個 App，你可以安裝在你的設備上測試，並與 Spotify 內部的其他人分享，讓他們真正感受使用那個點子是什麼感覺。」

更令人印象深刻的是：連 Spotify 的 CEO 現在都在建立這樣的原型。不需要說服工程師花時間，不需要等待開發資源——任何人、任何點子，幾分鐘之內都能看到實際效果。

啟示：「原型設計的民主化」是 AI 代碼工具帶來的最深刻的組織變革之一。它不只是加速工程師，而是讓非工程師也能直接參與產品探索。

六道引導思考題

Niklas 的演講從大型工程組織的視角提出了幾個深刻的問題——關於技術決策、組織進化、以及 AI 時代的競爭優勢。

Q1 · Spotify 的「技術越少越快」策略在 AI 時代得到了回報。但這個策略在十五年前需要對抗工程師的自然傾向（工程師喜歡嘗試新技術）。對於一家小型新創公司，是應該從第一天就採用嚴格的技術標準化，還是應該先快速實驗、之後再收斂？

查看建議思考方向
  • 歷史對照：Amazon 和 Netflix 的早期技術棧都非常混亂，後來的標準化工作非常痛苦——但他們也因此學到了什麼技術真正適合自己。早期的「亂」是否有其學習價值？
  • 講者隱含論點：Niklas 說「我們想要的是有意識的差異化（intentional variance）」，不是零差異化——這個「有意識」的邊界在哪裡？
  • 職涯啟發：在小公司，你往往是推動或阻礙技術標準化的關鍵人。什麼時候堅持標準，什麼時候允許實驗，這是一個需要判斷力的問題。
  • 反向思考：過度標準化可能阻礙創新——當所有服務都長一樣，工程師也許會錯過一個更好的解決方案。如何防止標準化變成「技術停滯」？

Q2 · Spotify 現在有 76% 更多的 PR 需要審查，導致「Review Fatigue（審查疲勞）」。他們的解法是「自動核准低風險 PR」。但是「低風險」的判斷本身就可能出錯。你會如何設計一套「自動核准」系統的信任校準機制？

查看建議思考方向
  • 歷史對照：銀行的風險評分系統（信用評分、詐欺偵測）也面臨類似挑戰：自動化系統偶爾會犯錯。業界如何建立「犯錯後的修復機制」？
  • 講者隱含論點：Niklas 說自動核准的前提是「CI 通過」——但 CI 有盲點。什麼類型的問題是 CI 測試無法偵測的？
  • 職涯啟發：設計「以最小化誤判代價為目標」的系統，比設計「以零誤判為目標」的系統更實際——如何在「假陰性（漏掉壞PR）」和「假陽性（阻攔好PR）」之間找到正確的平衡點？
  • 反向思考：也許「審查疲勞」本身就是一個信號——告訴我們過去的 Code Review 文化可能本來就有過多的儀式性審查，而不是真正的品質把關。

Q3 · Hyrum's Law 指出「所有可觀察的行為，最終都會被某些使用者依賴」。這個定律在 AI 大規模參與代碼生成後，會更嚴重還是更輕微？

查看建議思考方向
  • 歷史對照：Hyrum's Law 的核心是「使用規模」——越多人使用一個 API，它的每一個細節就越可能被依賴。AI 讓使用規模暴增，這意味著 Hyrum's Law 的效應會加速？
  • 講者隱含論點：Spotify 用 LLM 的「語意理解」來繞過 Hyrum's Law 的確定性問題。但 LLM 生成的代碼本身是否也會產生新的「隱性依賴」？
  • 職涯啟發：理解 Hyrum's Law 是 API 設計的重要背景知識——設計 API 時，哪些行為你希望是「穩定的合約」，哪些是「實作細節」？
  • 反向思考：也許 AI 會創造一種新的 Hyrum's Law 變體：「當足夠多的 AI 都用同樣的模式生成代碼時，這些模式就會變成事實上的標準」——這是好事還是壞事？

Q4 · Niklas 說「當編碼不再是瓶頸，瓶頸轉移到人類決策速度——哪些點子值得做、如何更快驗證假設」。這對傳統的產品管理（PM）和業務策略有什麼影響？

查看建議思考方向
  • 歷史對照：電商讓「開店」的成本接近零，結果瓶頸從「能不能開店」轉移到「如何找到客戶、如何建立品牌」。同樣的動態是否也會發生在軟體產品上？
  • 講者隱含論點：Niklas 說 Spotify「有太多好點子，從來沒有足夠的開發資源去全部嘗試」。如果開發資源無限，決策能力就成了稀缺資源——好的 PM 比以前更重要，還是更不重要？
  • 職涯啟發：如果你正在考慮工程師以外的職業路徑，「產品決策能力」在 AI 時代的價值可能比過去任何時候都要高。
  • 反向思考：也許瓶頸的轉移會讓組織「實驗太多」——原型很快，但缺乏焦點，結果是一堆半成品，而不是真正做好的東西。「能力 → 克制」的悖論。

Q5 · Honk v2 的「Multiplayer」功能讓多人共享同一個 AI Agent 會話。這在協作方式上和傳統的「pair programming（配對編程）」有什麼本質的不同？這種新的協作模式會帶來哪些意想不到的挑戰？

查看建議思考方向
  • 歷史對照：Google Docs 改變了文件協作，但也帶來了「編輯衝突」、「所有權不清」等新問題。多人共享 AI Session 是否會帶來類似問題？
  • 講者隱含論點：Niklas 強調他「個人非常期待多人協作功能」——這暗示他認為目前「一人一 AI」的模式有什麼不足？
  • 職涯啟發：「多人 AI 協作」可能是下一代工程文化的核心實踐——如何在共享 AI Session 的環境中保持個人貢獻的可見性和責任歸屬？
  • 反向思考：也許「多人共享 AI」的最大挑戰不是技術，而是人——每個人都想讓 AI 做不同的事，共識形成的速度可能慢於單人使用 AI 的速度。

Q6 · 「把 Backstage 的功能封裝成 MCP，讓 Claude 可以查詢軟體擁有者、在 Slack 聯絡相關團隊。」這讓 AI Agent 有了「查詢組織圖」和「直接聯繫人類」的能力。這個設計的邊界在哪裡？你願意讓 AI Agent 自動代表你發送什麼類型的訊息？

查看建議思考方向
  • 歷史對照：Email 自動化（Newsletter、系統通知）已被廣泛接受，但 AI 自動發送「以你的名義」的 Slack 訊息呢？這個接受度的邊界在哪裡，為什麼？
  • 講者隱含論點：Spotify 已經實作了「Honk 在 Slack 聯絡代碼擁有者」——這暗示他們已經解決了授權和身份的問題。這樣的系統需要什麼前提才能被工程師接受？
  • 職涯啟發：設計「AI 代理人可以代表你行動的邊界」是未來系統架構的重要問題——這不只是技術問題，更是組織文化和信任問題。
  • 反向思考：如果 AI 可以自動聯繫任何人，組織內的「人際資本（Social Capital）」會不會因此貶值？當所有人都被大量 AI 生成的訊息轟炸，Slack 的信噪比會如何變化？

名詞速查表

Fleet Shift

Spotify 的大規模代碼變更引擎，能跨數千個 Repo 批量執行代碼修改腳本，並追蹤每個 Repo 的狀態。

Honk

Spotify 的 AI 工程師 Agent，底層用 Anthropic Agent SDK + Claude，跑在 Kubernetes Pod 中，擁有 CI 建置和跨 OS 測試的驗證工具。

Chirp

Spotify 的內部 Agent 編排工具，用於同時調度和協調大量 Agent 會話，Honk v2 建立在 Chirp 之上。

Backstage

Spotify 開源的開發者門戶框架，提供軟體目錄、工具整合、標準化評估（Soundcheck）等功能。

Soundcheck

Backstage 中的自我評估工具，讓團隊確認自己的服務是否符合 Spotify 的技術黃金標準（Golden State）。

Hyrum's Law

「當 API 有足夠多的用戶，所有可觀察的行為都會被依賴」——Google 工程師 Hyrum Wright 的發現，是大型代碼庫維護的核心挑戰。

Auto-merge

完全不需要人類介入的 PR 合併流程——自動化系統生成 PR、驗證 PR、合併 PR。大多數 Fleet Shift PR 採用此流程。
