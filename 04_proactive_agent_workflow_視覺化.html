I

從工具到隊友：一個範式的轉移

Maya 的演講從一個核心洞見出發：編碼代理不應該只是等著你按下 Enter。真正的「隊友」會主動察覺問題、自動修復，而不需要你每次都去喚醒它。

工具（Tool）

等待你輸入提示詞

等待你按下 Enter

你閉筆電，代理停止

無法在背景持續運行

Headless 模式看不到在做什麼

→

主動察覺程式碼斷裂

自動觸發，無需按 Enter

永遠在線，閉電腦也繼續

"Coding agents shouldn't wait for you to press Enter to get started. A teammate notices when something breaks and does something about it. A tool waits for you to enter your prompt and press Enter." 「編碼代理不應該等待你按下 Enter 才開始工作。真正的隊友，當它注意到某些東西壞掉時，會主動做點什麼。工具則只是等你輸入提示詞、按下 Enter。」

— Maya, Anthropic Applied AI Team

現行主動式代理的三大痛點

在 Routines 出現之前，想讓 Claude Code 「自動跑起來」，工程師必須自己解決三個問題：

①

運行環境困境：在哪裡跑？ 不能在本地筆電跑（關機即斷線），需要自己處理雲端託管、Session 狀態持久化、API 憑證管理——大量基礎設施樣板代碼。

②

觸發機制繁瑣：什麼時候跑？ 需要自行架設 Cron Job 或建立 API 端點，整合 GitHub 事件或外部系統需要額外工程——又是更多基礎設施。

③

缺乏人機協同：跑的時候怎麼辦？ Headless 模式下看不到代理在做什麼，無法即時觀看、引導、修正方向，更無法恢復中斷的 Session。

給初學者 · 什麼是 Cron Job？
定時自動跑程式的傳統方法

Cron Job 是 Unix/Linux 系統的定時任務排程器。你可以設定「每天凌晨 2 點自動備份資料庫」或「每週一早上自動發報告」等規則。

  • 語法範例：0 10 * * 1 = 每週一早上 10:00 執行
  • 問題：需要一台永遠開著的伺服器才能執行，還要自己管理失敗重試、日誌記錄等
  • Routines 的優勢：Anthropic 的託管基礎設施幫你管理一切，你只需要定義「做什麼」

Routines 讓你不需要知道 Cron 語法，用自然語言描述觸發條件，Claude 幫你處理底層細節。

Routines：一個指令建立主動式代理

# 在 Claude Code 終端機輸入： /schedule once a week, review all new changes merged to main against our documentation repo and create a PR to update docs if you see any changes

這一行自然語言，就是建立 Routine 的起點。Claude Code 會接著問你：幾點觸發、要不要透過 Slack 通知你——然後自動建立一個在 Anthropic 託管基礎設施上持續運行的工作流。

II

Routines 的三大設計支柱

Maya 解釋了 Routines 背後的三個核心設計原則，每一個都對應著現有主動式代理的一個痛點。

支柱一：永遠在線（Always Available）

Routines 運行在 Anthropic 的託管基礎設施上——這意味著：

  • 你的筆電關機不影響代理繼續運行
  • Anthropic 幫你處理雲端託管、Session 狀態持久化、連接器憑證管理
  • 你只需定義「做什麼」，不需要管「怎麼跑起來」
給初學者 · 什麼是「Session 狀態持久化」？
讓 AI 代理「記得自己做到哪了」

當一個代理執行長達數小時的任務時，它的「工作狀態」（讀了哪些文件、做了哪些決策、還有什麼待辦）必須被保存下來——這就是「狀態持久化」。

  • 沒有持久化：代理崩潰或重啟後，一切從頭開始，已完成的工作消失
  • 有持久化：代理可以從上次停下的地方繼續，或在人類介入後恢復
  • Routines 的做法：Anthropic 的基礎設施自動管理 Session 狀態，開發者不需要自己實作

支柱二：主動式觸發（Proactive Triggers）

⏰ 時間基準

排程觸發（Schedule-based）

  • 「每週一上午 10:00 執行」
  • 「每天凌晨比對 main 分支」
  • 「每月第一個工作日生成報告」
⚡ 事件基準

事件觸發（Event-based）

  • GitHub 原生事件（Issue 開啟、PR 合併、Release 發布）
  • CI/CD 完成後的 Webhook
  • 任何外部系統的 POST 請求
給初學者 · 什麼是 Webhook？
「事情發生了，馬上告訴我」的通知機制

Webhook 是一種讓外部系統在特定事件發生時自動通知你的機制——相對於你主動去查詢（Polling），Webhook 是對方主動推送（Push）通知。

  • 你的 CD 流水線：「部署完成了！」→ 傳送 POST 請求到你的 Webhook 網址
  • Routines 收到請求：「好，開始跑驗證代理」
  • 類比：與其每 5 分鐘查一次外送訂單狀態，不如讓外送員「到了就發訊息通知你」

支柱三：可引導性（Steerable）

每個 Routine 本質上都是一個 可以即時觀看和引導的 Claude Code Session。你可以：

A
即時串流觀看

在 Claude.ai 網頁端實時看到 Claude Code 正在執行的每一條命令和推理邏輯，就像在本地終端機一樣透明。

B
中途介入（Nudging）

當你發現 Claude 走錯方向，可以直接在 Session 中途輸入指令修正：「我已經做了這些變更，停止這個 Session」。代理立刻讀取並調整行為。

C
續接歷史 Session

可以隨時 Resume（恢復）過去已結束或中斷的 Routine Session，繼續對話——不必重新啟動整個工作流。

D
多平台控制

Web、CLI、Desktop 三種介面都可以查看和控制同一個 Routine Session，依個人偏好選擇操作方式。

"Every routine is really just a Claude Code session under the hood that you can open, watch, follow up on, steer, and resume — from web, CLI, and desktop." 「每一個 Routine，底層都只是一個 Claude Code Session——你可以打開它、觀看它、跟進它、引導它，以及恢復它——從網頁、CLI 或桌面端都可以。」

— Maya
III

建立 Routine 的三大核心決策

Maya 提出了一個清晰的「Routine 設計框架」：每次建立自動化工作流前，必須回答三個問題——When、What、How。這三個問題的答案，決定了代理的成效上限。

框架介紹 · Maya 的 Routine 設計思維

不是在配置參數，而是在設計一個數位員工的工作說明書

Maya 指出：建立 Routine 的過程，其實就像在寫一份「工作說明書」給一位新進員工——你需要告訴他：什麼時候該工作（Trigger）、需要哪些工具和資訊（Context）、工作後你要怎麼審核他的成果（Steerability）。

跳過任何一個環節，代理的輸出品質都會打折。其中最容易被忽略的是「上下文」——代理的成效上限，永遠受限於你給它的資訊邊界。

這個思維轉換很重要：你不是在「設定程式」，而是在「培訓一位能在背景獨立工作的員工」。

決策一：When——觸發時機

問自己：「這個任務應該在什麼時候自動啟動？」

排程觸發場景

定期維護類任務：文件同步、週報生成、數據清理、安全掃描。這些任務沒有特定的觸發事件，只需要固定的執行週期。

事件觸發場景

響應式任務：新 Issue 自動分類、部署後驗證、PR 合併後更新文件。這些任務的意義在於「即時響應」，固定排程會造成延遲。

決策二：What——上下文與連接器

問自己：「Claude 需要什麼資訊和工具，才能把這個任務做好？」

給初學者 · 為什麼「上下文」是代理成效的天花板？
你給的越多，Claude 能做的越好——但也要有所取捨

一個代理的智能，建立在它能「看到什麼」之上。如果一個文件同步代理沒有程式碼 Repo 的讀取權限，它根本無法知道哪裡有變動。

  • 代碼 Repo：知道「最新的程式碼長什麼樣」
  • 文件 Repo：知道「現有的文件長什麼樣，哪裡需要更新」
  • Google Drive：知道「行銷規範、品牌語氣，讓文件風格一致」
  • Slack Connector：知道「事情做完了要怎麼通知人」

Maya 的原則：「Whatever context Claude has, that's the ceiling of how successful Claude will be.」（無論你給 Claude 什麼上下文，那就是它成效的上限。）

常見連接器與工具：

→

GitHub：讀取程式碼、比對 Repo 差異、開立 PR、響應 Issue 和 Label 事件

→

Slack：在任務完成或異常時通知相關人員；也可讀取 Slack 頻道作為輸入上下文

→

Google Drive：讀取行銷文件、品牌語氣指南，讓代理輸出的文字符合公司風格

→

監控工具（DataDog/Grafana）：讀取服務指標，讓代理能做出 Go/No-Go 決策

→

自訂 MCP：透過 Model Context Protocol 接入任何第三方系統

決策三：How——引導與品質控制

問自己：「我要怎麼確保 Claude 的輸出品質？什麼時候需要人類介入？」

給初學者 · 什麼是 Generator-Critique 雙代理模式？
讓一個 AI 檢查另一個 AI 的工作

Generator-Critique（生成者-評論者）是多代理系統的經典設計模式：

  • Generator（生成者）：Routine A，負責執行主要任務——例如根據程式碼變更自動撰寫文件並開 PR
  • Critique（評論者）：Routine B，在 Generator 建立 PR 時自動觸發，對 PR 進行獨立審查並留下 Review Comments
  • 人類工程師看到的是「已經經過 AI 預審的 PR」，大幅降低審查負擔

這個模式把「品質控制」也自動化了，不只是讓 AI 做任務，而是讓 AI 也檢查 AI 的工作。

代理審查（Agent-on-Agent）

設定 Routine B 在 Routine A 建立 PR 後自動觸發，對 PR 進行獨立 Code Review。人類介入前，AI 已完成預審。

即時人類介入

在網頁端即時觀看 Session，發現問題隨時輸入指令修正方向。也可設定在關鍵節點（如「是否執行 Rollback」）強制等待人類確認。

輸出渲染驗證

對於文件類任務，直接渲染最終頁面確認樣式和內容——視覺驗證比 diff 更直觀，也更接近用戶最終看到的樣子。

信任度漸進授權

從「AI 建議、人類決定」逐漸演進到「AI 自動執行、人類抽查」——隨著信任度建立，逐步擴大代理的自主權。

IV

Anthropic 內部案例：文件自動化工作流

這不是假設性的場景，而是 Anthropic 工程師真實使用 Routines 解決的真實痛點：Claude Code 的快速發展，讓單一文件工程師無法跟上程式碼變更的速度。

Anthropic 內部實踐 · Sarah 的文件自動化閉環

一個工程師 vs 200% 增速的程式碼變更

自 2026 年初以來，Claude Code 的每週 PR 合併數增長了 200%。對 Claude Code 工程團隊來說，這是生產力飛躍的成果；對負責維護文件的工程師 Sarah 來說，這意味著永無止境的趕工壓力。

Sarah 成為 Routines 的早期採用者。她在終端機輸入的第一個 /schedule 指令，建立了這樣一個自動化閉環：

①
觸發：每週定時

每週一上午 10:00，Anthropic 的託管基礎設施自動啟動 Claude Code Session

②
Claude 讀取 Source Repo 與 Changelog

比對上週 main 分支的所有合併 PR，提取 API 變更、新工具、功能更新

③
比對文件 Repo

找出哪些程式碼變更在文件中沒有對應的說明，識別「文件缺口」

④
自動在文件 Repo 開立 PR

為每個發現的文件缺口撰寫補充說明，提交 Pull Request，等待人工審核

⑤
透過 Slack 通知 Sarah

「已建立 3 個文件更新 PR，請查看」——Sarah 收到通知，進行最終確認

給初學者 · 什麼是 PR（Pull Request）？
「我做了一些更改，請幫我審查後合併」

PR（Pull Request）是程式碼協作的標準流程：當你完成了一些程式碼變更，你不直接合併到主分支，而是發起一個「合併請求」，讓團隊成員審查你的改動。

  • 沒有 PR：每個人直接改 main 分支，容易引入 bug，難以追蹤變更
  • 有 PR：每次變更都有明確的記錄、審查者和審查意見，保證程式碼品質
  • Routines 的做法：Claude 自動建立文件更新 PR → Sarah 只需審核、批准或修改 → PR 合併，文件更新完成

Demo 二：事件觸發的即時響應

Maya 的第二個 Demo 展示了事件觸發的威力：每當有人在 Claude Code 文件 Repo 開立新 Issue，代理自動調查「這是否是文件缺口」，如果是，就自動開 PR 並在 Slack 通知。

Demo 亮點 · 即時人類介入的優雅展示

「我已經做了這些變更，請停止這個 Session」

在 Demo 過程中，當新 Issue 觸發代理開始工作後，Maya 發現她其實已經手動建立了一個相關的 PR。

她直接在 Claude.ai 網頁端的 Session 介面輸入：「我已經對這個議題開了一個 PR，請停止這個 Session。」

代理立刻讀取了這個指令，停止了正在進行的工作。

這個看似簡單的互動，展示了 Routines 最重要的設計原則：AI 代理的自主性，必須始終在人類可以即時介入的框架內運行。自主性和可控性不是矛盾的，而是共存的。

IV

延伸場景：把任何工程挑戰變成 Routine

Maya 提出兩個經典的工程場景，展示如何用「三大決策框架」設計 Routine，把過去需要工程師親手處理的繁瑣任務，轉化為自動化工作流。

場景一：自動化部署驗證器

每次部署後，你都要盯著 Grafana 看一個小時確保服務正常——這正是 Routine 可以解放你的場景。

T
觸發：CI/CD Webhook

CD 流水線完成部署後，自動發送 POST 請求到 Routine Webhook，啟動驗證代理

C
上下文：服務代碼 + 監控工具

連接服務的 Source Repo、DataDog/Grafana（讀取指標）、Slack/Twilio（發送警報）

S1
初期引導：人類做 Go/No-Go 決策

Claude 分析日誌與指標，給出詳細的 Go/No-Go 分析報告 → 人類在網頁端閱讀分析 → 決定是否 Rollback

S2
成熟期：授權 Claude 自主 Rollback

隨著信任度建立，可授權 Claude 在指標異常時自動執行 Rollback——無需人工確認

場景二：PM 待辦清單智能分類器

每週整理數百條 GitHub Issues 和 Slack 討論，手動分類優先級——這是每個 PM 的噩夢，也是 Routine 最適合接管的任務。

設計要素

觸發：每週定時排程
上下文：GitHub Issues 庫 + Slack 特定通報頻道
輸出：按優先級排序的清單，附上 AI 的分類理由

代理能做的事

自動讀取上百條未處理 Issue、分類為「文件缺失」「核心 Bug」「功能需求」，高優先級項目標記，甚至直接為簡單修復開立 PR

給初學者 · 什麼是 Backlog？為什麼要整理它？
產品待辦清單的混沌與秩序

Backlog 是所有「待完成工作項目」的清單——包括用戶反饋、Bug 報告、新功能想法。問題在於，沒有人整理的 Backlog 會快速失控：

  • 項目越積越多，最重要的需求被埋在雜訊中
  • PM 每週要花數小時手動閱讀和分類
  • 優先級判斷主觀，容易受近因偏差影響

用 Routine 自動分類 Backlog，讓 PM 每週只需做「最後的判斷」，而不是「大量的閱讀分類」——把人類的注意力集中在真正需要判斷的地方。

V

思考題與討論方向

Routines 代表了一種全新的人機協作模式。以下六個問題，邀請你思考「主動式代理」的邊界、設計哲學與深遠影響。

Q1 — Maya 說「proactive agents beat reactive agents（主動式代理優於被動式代理）」。但主動式代理也帶來了新的風險——代理可能在錯誤的時機、以錯誤的方式採取行動。你認為，主動式代理最容易在哪些情境下造成有害後果？如何在設計層面降低這種風險？

查看建議思考方向
  • 1. 量化風險：對比「程式碼自動化測試（CI/CD）」和「生產環境自動 Rollback」的風險量級差異——前者的錯誤是「多測一次」，後者的錯誤可能是「正常部署被錯誤回滾，造成服務中斷」。主動式代理的風險是否都可以被量化為「誤觸發的代價」？
  • 2. 隱含論點：Maya 設計的「信任度漸進授權」是一種風險控制機制——先讓人類做決策，再逐步授權 AI 自主。這個機制的本質是什麼？它是否能被形式化為一套通用的 AI 代理授權框架？
  • 3. 職涯啟發：「設計主動式代理的風險邊界」正在成為新的工程專業——你在目前工作中，有哪些任務是「可以被主動式代理接管但必須有嚴格護欄」的？如何設計這些護欄？
  • 4. 反向思考：有時候「什麼都不做」也是一種風險——被動式代理在緊急情況下無法即時響應。如何量化「過於保守」和「過於主動」的兩種風險，並找到平衡點？

Q2 — Routines 的核心設計之一是「可引導性（Steerability）」——人類可以隨時在正在運行的 Session 中介入、修正方向、終止任務。這個設計反映了什麼樣的 AI 治理哲學？它與「完全自主 AI」的設計取向有何根本差異？

查看建議思考方向
  • 1. 治理哲學：「Corrigibility（可糾正性）」是 AI 安全研究的核心概念——一個系統應該在任何時候都允許人類糾正或關閉它。Routines 的設計是否體現了 Corrigibility 原則？在哪些方面做得好，哪些方面有改進空間？
  • 2. 隱含的控制幻象：人類「可以介入」不等於「確實會介入」。如果一個 Routine 每天自動跑、很少被人觀看，那麼「可引導性」的安全保障實際上是名義上的。如何確保人類真的保持對代理的有效監督？
  • 3. 職涯啟發：作為工程師，設計「人類監督介面」正在成為 AI 開發的核心技能。一個好的監督介面需要滿足哪些條件（可觀察性、低認知負荷、清晰的介入點）？
  • 4. 反向思考：如果代理表現一直很好，人類會逐漸停止監督（信任度提升、注意力下降）。這種「逐漸喪失監督習慣」的過程是否會帶來長期風險？如何設計機制防止這種情況？

Q3 — Maya 展示了「Agent-on-Agent Review（代理審查代理）」模式——一個 AI 代理檢查另一個 AI 代理的工作。這個模式在提升品質的同時，是否也帶來了「AI 確認偏誤」的風險——Critic 代理可能偏向認同 Generator 代理的輸出？

查看建議思考方向
  • 1. 量化案例：在代碼審查中，研究顯示人類審查者也有「作者光環效應」——傾向於認可知名工程師的 PR。AI 代理是否也有類似的「生成者偏差」？有沒有已知的研究數據？
  • 2. 設計緩解：如果 Critic 使用不同的 Prompt、不同的上下文、甚至不同的模型（如一個 Generator 用 Opus、Critic 用 Sonnet），是否能減少確認偏誤？這樣設計的成本是多少？
  • 3. 職涯啟發：「設計有效的 AI QA 流程」正在成為新職位。如果你要設計一個可信任的 Agent-on-Agent Review 系統，你會加入哪些「多樣性機制」來避免兩個代理都犯同樣的錯誤？
  • 4. 反向思考：兩個 AI 代理互相審查，是否比一個人類花 5 分鐘看 PR 更可靠？在哪些任務類型上，AI-AI 審查比 AI-Human 審查更可靠？在哪些任務上反而更不可靠？

Q4 — Routines 讓「上下文邊界決定代理成效上限」這個原則變得非常明顯。未來如果代理能夠主動「請求更多上下文」（如主動詢問用戶、主動搜尋外部資訊），這個原則是否還成立？代理的「自主資訊獲取能力」應有哪些限制？

查看建議思考方向
  • 1. 現況分析：已有代理可以使用網頁搜尋工具主動獲取信息，但搜尋的範圍和行為是由開發者預先定義的。若代理能完全自主決定「去哪裡獲取什麼信息」，最大的風險是什麼（成本失控、意外訪問敏感系統、信息可靠性）？
  • 2. 隱含邊界：企業的資安框架（Least Privilege Principle，最小權限原則）要求：每個系統只能訪問完成任務所需的最少資源。如何把這個原則應用到 AI 代理的「上下文獲取權限」設計？
  • 3. 職涯啟發：「設計 AI 代理的上下文訪問策略」正在成為 AI 架構師的核心技能。在你目前熟悉的業務場景中，如果要部署一個 Routine，哪些資訊是必須給的、哪些是危險的、哪些是「可選但能顯著提升效果」的？
  • 4. 反向思考：也許「給代理太多上下文」和「給太少」一樣危險——過多的上下文可能分散代理的注意力，讓它陷入資訊過載而犯更多錯誤。你認為存在「最佳上下文密度」嗎？如何確定它？

Q5 — Routines 的設計讓「信任度漸進授權」成為可能：從人類做最終決策，到 AI 自主行動。在工程組織中，這個信任度的建立過程應該如何被規範化？是個人工程師的判斷，還是需要組織層面的授權流程？

查看建議思考方向
  • 1. 組織案例：對比「飛機自動駕駛儀的授權流程」——飛行員不能個人決定「我信任這個 AI 系統，讓它全程自動駕駛」，必須有標準化的認證和操作流程。AI 代理的「自主授權升級」是否也需要類似的組織規範？
  • 2. 隱含的責任問題：如果一個工程師個人決定「授權 Claude 自動 Rollback 部署」，而某次 Rollback 導致了意外服務中斷，責任歸屬是怎樣的？個人判斷和組織授權在責任分配上有何不同？
  • 3. 職涯啟發：制定「AI 代理授權升級標準（AI Agent Authorization Framework）」可能成為 AI 治理工程師的重要工作。你認為這個框架至少需要包含哪些要素（測試覆蓋率門檻、觀察期長度、回滾機制）？
  • 4. 反向思考：過於嚴格的授權流程會讓主動式代理的效益大打折扣——如果自動 Rollback 需要委員會審批，那它就失去了「即時響應」的意義。如何在授權嚴格性和操作效率之間取得平衡？

Q6 — Maya 指出 /schedule 是一個讓開發者「專注於領域專業，而非維護基礎設施」的工具。隨著 Routines 這類工具的普及，「基礎設施工程師」的角色是否會萎縮？AI 代理是否最終會接管大部分 DevOps 工作？

查看建議思考方向
  • 1. 歷史量化：回顧 Kubernetes 和 Terraform 的出現——它們確實降低了「人工配置基礎設施」的需求，但也創造了大量新職位（SRE、Platform Engineering）。主動式代理是否會走同樣的路：取代某些工作，同時創造新工作？
  • 2. 隱含的技能轉移：如果基礎設施的日常維護被代理接管，DevOps 工程師的核心工作可能轉變為「設計代理的工作邊界和安全護欄」——這是技能降級還是升級？
  • 3. 職涯啟發：如果你目前的工作中有 30% 是「重複性的維護任務（部署驗證、文件更新、Issue 分類）」，Routines 是否能替代這 30%？被替代後，你的時間應該花在哪裡才能創造最高價值？
  • 4. 反向思考：Maya 說「讓開發者專注於領域專業」——但「設計和維護 Routine 本身」也是一種新的基礎設施工作。Routine 出錯了怎麼辦？Routine 需要升級怎麼辦？「管理主動式代理的人」是否成為了新的 DevOps？

名詞速查表

A

Agent-on-Agent Review（代理審查）：借鑑 Generator-Critique 模式，由一個 AI Routine 生成輸出、另一個 AI Routine 在人類介入前自動審查並留下評論，形成 AI 品質控制層。

B

Generator-Critique Pattern：多代理系統的經典設計模式——Generator 代理負責生成輸出，Critique 代理負責評審和批評，兩者協作提升輸出品質。

C

Headless 模式：在沒有圖形介面的情況下執行程序。Headless Claude Code 代表在背景無視覺介面地運行代理，傳統上工程師無法即時觀看或干預——Routines 解決了這個問題。

D

Nudging（中途介入）：在 Routine 代理運行中途，人類直接輸入指令修正代理方向的能力——無需停止並重新啟動 Session，代理即時讀取並調整行為。

E

Routines：Claude Code 的新功能，允許開發者在 Anthropic 託管基礎設施上定義並運行自動化工作流。核心四要素：Prompt（提示詞）、Repos（程式庫）、Connectors（連接器）、Trigger（觸發條件）。

F

Steerability（可引導性）：Routines 三大設計支柱之一，指人類能隨時介入、觀看、引導或終止正在運行的代理 Session，確保 AI 自主性在人類可控範圍內。

G

Webhook：外部系統在特定事件發生時，主動向 Routines 端點發送 POST 請求以觸發代理的機制。相對於定時輪詢（Polling），Webhook 實現即時事件響應。

H

/schedule 指令：在 Claude Code 終端機中輸入此指令，即可進入 Routine 建立流程，透過自然語言對話定義觸發條件、上下文和連接器，無需手動配置 Cron 或 API 端點。
