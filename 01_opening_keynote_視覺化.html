Boris 的起點故事

創辦人的啟蒙故事 · 寫程式的純粹感動

13 歲，一台 TI-83 計算機，以及重新找回的魔法

Boris Cherny 在 13 歲時，用 TI-Basic 在德州儀器 TI-83 計算機上寫出數學公式程式，幫助自己與同學通過考試。同年，他學了 HTML，為 eBay 寶可夢卡牌賣場設計表格與閃爍文字——三張傳奇閃卡，每張 99 美分成交。

「從『我有一個點子』到『這個點子成功運行』之間的距離，正在重新瓦解。你只需要描述一個問題，分布式系統便會自動呈現。」— Boris Cherny

隨著時代發展，編譯器、類型檢查器、構建系統拉長了這段距離。而今天，Claude Code 正在讓這段距離再次消失——不是對 13 歲的孩子，而是對擁有 23,000 名工程師的 Mercado Libre，以及每天拯救寄養兒童的 Binty 新創公司。

啟示：技術的本質是讓點子可以快速實現。複雜性是敵人，Claude 是解方。

給初學者 · 一句話搞懂 Claude Code 的定位
為什麼「描述問題就能得到程式碼」這件事是大革命？

傳統寫程式就像蓋房子：你要先搭鷹架（設定環境）、再砌磚（寫核心邏輯）、最後拆鷹架（清理設定）。整個過程有 60% 的時間在蓋鷹架，不是在蓋房子。

  • 以前：設定 12 個設定檔 → 安裝 npm 套件 → 解決版本衝突 → 才能寫第一行邏輯
  • 現在：用中文描述問題 → Claude 理解意圖並輸出完整可運行代碼

這不只是省時間。對非工程師背景的創業者、社工系統開發者、甚至技術 VP，這意味著他們也能開始「交付代碼」——寫程式的魔法不再是少數人的特權。

"The default state of software engineering is no longer 'I prompt Claude Code.' The default state is 'I assign Claude to prompt Claude Code for me.'" 「軟體工程的預設狀態，不再是我來向 Claude Code 下 Prompt，而是我指派 Claude，由它來替我向 Claude Code 下 Prompt。」

— Boris Cherny, Claude Code 創始人

給初學者 · 一句話搞懂 Opus 4.7 的突破
什麼是「端到端主導最終成果」？

想像你告訴一個實習生「幫我搞定這個 Bug」。以前的 Claude 會問：「你要我用哪個方法？要修改哪個檔案？要怎麼測試？」現在的 Opus 4.7 會說：「我去看代碼、理解問題、選擇方法、修改、測試、回報——你只要確認結果就好。」

  • 以前的模型：需要人類詳細分解每一步，像在教一個新手
  • Opus 4.7：自己規劃「數百步的智慧循環」，像一個有判斷力的資深工程師

這就是「端到端主導最終成果（End-to-end driving of outcomes）」：從「接任務」到「交付成果」，中間不需要人類介入。

Mythos 的驚人發現 · 30 年歷史漏洞

OpenBSD 漏洞：Mythos 找到了人類與機器都錯過的一切

就在上個月，Anthropic 最新模型 Mythos 預覽版讀取了整個 OpenBSD 的原始碼樹（Source Tree），並成功找出了存在其中長達 30 年之久的安全漏洞。

這個漏洞在過去近三十年裡，躲過了所有人類審查員、所有模糊測試工具（Fuzzers）以及所有靜態分析器的檢測。

啟示：當模型能夠「讀完整個代碼庫」且「持續運行數小時」，它看到的不是片段，而是整個系統的全貌。這種「全局視角」正是人類工程師最缺乏的能力。

給初學者 · 一句話搞懂任務地平線
任務地平線（Task Horizon）是什麼？

就像人類工作時會「失去焦點」——午飯前的 2 小時你很專注，下午 3 點後開始想摸魚。Task Horizon 是指 AI 模型在執行長任務時，在「迷失方向」之前能持續工作多長的時間。

  • 短 Task Horizon：「幫我寫一個函數」→ 幾分鐘就完成，不需要長期規劃
  • 長 Task Horizon：「這週請幫我確保這個專案按時推進」→ 需要持續監控、主動回報、自我調整
  • 為什麼重要：Task Horizon 越長，AI 能完成的任務就越接近「人類的工作」而非「人類的工具」

當 Task Horizon 達到「永續運行」，AI 就從「工具」變成了「數位工作者（Digital Worker）」。

"We are seeing Anthropic release 8 frontier models in the past 12 months — each building on the last, helping you write better code and deliver more advanced experiences." 「我們在過去 12 個月內發布了 8 個前沿模型——每一個都以前一個為基礎，協助你們寫出更好的程式碼，並帶來更先進的體驗。」

— Lisa, Research Product Manager

Scaffolding 的枷鎖：Lisa 的核心警告

Lisa 提出了這場大會中最反直覺的洞見：隨著模型越來越聰明，你過去精心打造的「輔助架構」正在成為限制它的枷鎖。

給初學者 · 一句話搞懂 Scaffolding
Scaffolding（鷹架）是什麼？為什麼要拆掉它？

蓋高樓時，鷹架是必要的——它支撐工人完成工作。但當大樓蓋好後，鷹架必須拆掉，否則它會遮住大樓的原始樣貌。

  • AI 的 Scaffolding：複雜的 Prompt 模板、防錯機制（Guard Rails）、狀態機、層層巢狀的工具調用鏈
  • 問題：這些東西在「笨模型時代」有幫助，在「Opus 4.7 時代」反而干擾模型的判斷力
  • 解方：給模型乾淨的基礎原語（檔案系統 + 沙盒環境），讓它自己決定怎麼做

Scaffold 越少，越聰明的模型表現越好——這就是「減法哲學」。

"Smarter models generally need more generalized primitives — like a file system and a sandboxed compute environment — to go further." 「更聰明的模型通常只需要更泛用的原語——例如一個檔案系統和沙盒化的運算環境——就能走得更遠。」

— Lisa, Research PM，Anthropic

給架構師的實踐建議

為「下一代 Claude」設計，而非當前版本

歷史上，能在市場勝出的開發者，都是那些架構已提前準備好、能「吸收下一個模型大跨越」的人。

用更難的 Evals 驗證模型邊界

當一個過去總是失敗的任務突然開始通過，那就是「可以推出新功能」的強烈訊號。

讓「模型升級」成為商業機遇

透過自動化評估流程，讓模型版本更新變得簡單。每次升級都是差異化競爭的機會。

移除硬編碼規則，給乾淨的原語

以「泛用工具（Generalized Primitives）」取代「硬編碼流程（Hard-coded Workflows）」，讓模型的判斷力發揮。

顧問策略：以最低成本拿前沿產出

Angela 提出的「顧問策略（Advisor Strategy）」是一個極簡但強大的架構模式：將執行（小模型）與顧問諮詢（大模型）分離。法律新創 eve Legal 因此以五分之一成本拿到前沿模型的品質。

給初學者 · 一句話搞懂顧問策略
顧問策略（Advisor Strategy）怎麼運作？

想像一個診所：一般護士（Haiku/Sonnet）負責第一線接診，遇到複雜病例再去請教主治醫師（Opus）。這樣既省了醫師的時間，又確保困難案例有最好的判斷力介入。

  • 執行者：Haiku 或 Sonnet——體積小、成本低、處理大量日常任務
  • 顧問：Opus——在執行者卡關時介入，提供高品質判斷
  • 結果：Sonnet 作為執行者的效果，甚至超越 Sonnet 獨自運作——因為 Opus 在關鍵時刻給出更好的方向

只需要更新 Messages API 的 Tools 陣列設定，就能啟動這個模式。工程成本幾乎為零，但效果是前沿級的。

適用場景：旗艦產品

需要給用戶提供最佳體驗，又要嚴格控制成本。顧問策略讓你「貌若頂配、費用親民」。

適用場景：高流量 Agent 任務

每天數千次 API 調用的自動化流程，小模型處理 90% 的案例，大模型只在關鍵 10% 介入。

II

MCP Tunnels：突破企業資安防線

這是本場大會的最重大技術突破。過去，企業若要讓 AI Agent 使用內部工具，就必須把資料庫 API 暴露在公開網路——這在金融、法律、電商業是絕對禁區。MCP Tunnels 終結了這個困境。

給初學者 · 一句話搞懂 MCP Tunnels
MCP Tunnels 是什麼？為什麼它解決了資安最大痛點？

你的公司有一個秘密資料庫，裡面放著所有的用戶數據和業務數據。以前要讓 AI Agent 存取它，就必須在防火牆上打洞——相當危險。MCP Tunnels 的概念像是「企業的 VPN 給 AI 用」：

  • 企業在私有網路內部署一個 Gateway（閘道器）
  • 閘道器與 tunnel.anthropic.com 建立加密連線
  • 雲端的 AI Agent 透過這條隧道安全存取資料庫
  • 資料庫從未暴露在公開網路上

結果：AI 能用到所有內部工具，企業的合規要求也完全滿足。這個功能對金融、醫療、法律產業的意義，等同於「AI Agent 正式獲准進入企業」。

Counter 公司 Demo · Growthbot 的真實演示

一個 Slack Bot 如何在防火牆內安全改變生產環境

Anthropic 現場演示了虛擬公司 Counter 的 Growthbot：這個 AI Agent 透過 MCP Tunnel 讀取防火牆內的數據倉庫，判定哪個 A/B 測試勝出，然後自動拉 PR 清理舊代碼——整個過程在 Vercel 的自我託管沙盒上運行，代碼從未離開企業控制的伺服器。

「這些 MCP 伺服器都深鎖在 Counter 公司的防火牆後方，『只能』透過這個安全隧道來進行存取。這樣一來，Counter 就能完美滿足企業的安全合規需求。」— Caitlyn，平台團隊

Growthbot 甚至主動在 Slack 問：「我發現引導流程有 X% 的流失率，你希望我現在開始動手處理嗎？」
啟示：安全不必犧牲主動性。MCP Tunnels 讓 Agent 既安全又靈活。

給初學者 · 一句話搞懂自我託管沙盒
自我託管沙盒（Self-hosted Sandboxes）是什麼？

當 AI Agent 需要「執行程式碼」時，它需要一個隔離的環境——就像你不會在你的主電腦上直接跑來路不明的代碼，你會在虛擬機裡跑。

  • 以前：使用 Anthropic 提供的共享沙盒，代碼在 Anthropic 的基礎設施上執行
  • 現在：可以在你自己的 Vercel、Cloudflare、Daytona、Modal 帳戶上建立沙盒
  • 好處：代碼在你自己的伺服器上跑，機密代碼從未外洩，完全符合企業資安要求

搭配 MCP Tunnels，這就形成了完整的「雲端大腦下指令，本地私有環境執行」架構。

III

Claude 託管型 Agent（Managed Agents）

Angela 介紹的 Managed Agents 是 Anthropic 為「從原型到生產」的跨越提供的基礎設施。核心：內建多 Agent 編排、成果導向（Outcomes）、夢境機制（Dreaming）。

多 Agent 編排（Multi-agent Orchestration）

建立一整隊 Agent 艦隊，不同 Agent 負責不同專長，由主 Agent 統一協調調度。

成果導向（Outcomes）

直接定義「成功的終點是什麼」，讓 Claude 自己規劃達成路徑並不斷迭代，直到目標實現。

夢境機制（Dreaming）

Claude 在後台自我反思過去的執行日誌，進行非同步的自我學習與自我優化。就像人在睡夢中整理白天的記憶。

給初學者 · 一句話搞懂 Managed Agents
Managed Agents 跟自己寫 Agent 有什麼差別？

自己寫 Agent 就像自己架伺服器——你要處理所有的基礎設施問題：重試邏輯、錯誤處理、狀態管理、多工協調……Managed Agents 就像用 Vercel 部署網站——這些都已經幫你搞定，你只需要寫業務邏輯。

  • Asana 用 Managed Agents 打造「AI 隊友」，讓人類在 Asana 介面中直接和 Agent 協作
  • 部署速度從「幾個月」縮短為「幾天」

Angela 說：「10 倍部署速度。」這不是行銷語言，是工程現實。

Claude Code Desktop：多工管理的統整中心

Kad 指出，當工程師開始同時跑多個 Claude Code 實例（「Multi-clotting」），需要一個統整視覺看板。Claude Code Desktop 應用程式正是這個答案。

給初學者 · 一句話搞懂 Claude Code Desktop
Claude Code Desktop 解決什麼問題？

想像你同時在管理 5 個實習生：一個在修 Bug、一個在寫測試、一個在做 Code Review……你需要一個地方能看到他們所有人的進度。Claude Code Desktop 就是這個「Agent 管理中心」。

  • 看板功能：哪些 Agent 正在跑、哪些被卡住、哪些 PR 已完成
  • 即時預覽：直接渲染圖表和視覺化輸出
  • 跨環境管理：同時管理本地和雲端執行的 Agent

不想用 GUI？CLI 也有全新的「Claude Agents View」，支援行內回覆、在不同工作之間跳躍而不失去上下文。

Mercado Libre 的規模奇蹟 · 當 23,000 人都用 Claude Code

技術 VP 重新開始寫程式的那一天

Mercado Libre 是拉丁美洲最大電商，擁有超過 1 億活躍買家和 23,000 名工程師。當整個組織全面採用 Claude Code 後，奇蹟開始發生：

那些過去因為退居幕後做管理、已經長達好幾年沒有親自提交過一行代碼的技術經理和副總裁們，現在居然因為這個工具而重新開始動手交付代碼、寫程式了。— Kad

目標是 Q3 達成 80% 的自主化編碼（Autonomous Coding）。但更動人的不是數字，而是那些失去寫程式快樂 10 年的人，重新找到了它。

啟示：Claude Code 不是在取代工程師，而是在讓每個人都能「成為工程師」。

II

Routines：深夜自動執行的高階 Prompt

Boris 解釋，Routines（常規任務）是一種「高階函數（Higher-order Function）」——不是直接下 Prompt，而是寫一個「能自動向 Claude Code 下 Prompt 的 Prompt」。

給初學者 · 一句話搞懂 Routines
Routines 就像什麼？

Routines 就像公司的「SOP 流程手冊」自動執行版本。你不需要每次手動觸發，只需要設定好觸發條件，Claude 就會按照 SOP 自動工作。

  • 時間排程觸發：每天凌晨 2 點掃描所有開放的 GitHub Issue
  • Webhook 觸發：有新 Issue 進來，立刻喚醒 Claude 開始工作
  • API 觸發：任何外部系統都可以啟動一個 Routine

結果：你早上起床，發現 PR 已經準備好了。隔天的工作前晚已完成。

Acme Pay 的真實演示：退款功能的誕生過程

Boris 現場示範：Claude 負責開發具備冪等性的退款功能，寫完代碼後自動在瀏覽器後台觸發退款驗證，發現了「樂觀更新中的競態條件（Race Condition）」邊緣案例，自己修復後再次驗證，確認通過才交付。整個過程：零人工介入。

"I'm no longer the one prompting directly. I build Routines that prompt the first-line engineering agents on my behalf." 「主動去下 Prompt 的人早就不是我了。我現在的角色，是去建構 Routines，由這些 Routines 來負責去向第一線的工程師 Agent 下達 Prompt。」

— Boris Cherny, Claude Code 創始人
III

CI Autofix：PR 的全職保姆

每個工程師都有這樣的經歷：一個 PR 已經寫完了，但 CI 一直紅燈。排查、修復、等待……重複三遍。CI Autofix 讓這個過程消失。

自動修復 CI 錯誤

網路超時、資源不足、測試 flaky……CI Autofix 自動診斷並重試，甚至能修改底層根本原因（Root Cause）。

處理 Code Review 意見

Reviewer 留下的修改意見，由 AI 自動評估並實作修改，不再需要工程師逐條回應。

自動 Rebase

主幹有新提交導致合併衝突，CI Autofix 自動處理 Rebase，保持 PR 狀態健康。

給初學者 · 一句話搞懂 CI/CD 和 CI Autofix
CI（持續整合）是什麼？為什麼紅燈這麼令人崩潰？

CI 就像工廠的品質檢測線——每次有新代碼提交，系統自動跑所有測試，確保這批代碼沒有破壞既有功能。「紅燈」表示測試失敗，PR 無法合併。

  • 以前的紅燈：工程師停下所有工作 → 排查哪裡失敗 → 修復 → 等待重跑測試 → 可能再次失敗
  • CI Autofix：你看不到紅燈。Autofix 在後台悄悄把問題解決掉，紅燈變綠燈，你只需要合併

這為工程師省下的不只是時間，更是那種「被 CI 打斷深度工作」的心理成本。

V

深度思考題

六個開放性問題，幫助你深化對本場演講核心概念的理解，並思考它們對你的職涯和工作的意涵。

Q1. Boris 說「指數型能力曲線 vs. 線性採用路徑」之間存在巨大鴻溝。你認為是什麼原因導致企業採用速度仍然是線性的？這個鴻溝應該如何縮短？

查看建議思考方向
  • 歷史對照：雲端運算的採用也曾有 10 年的「遲疑期」，企業如何從 on-premise 轉向 SaaS？這個過程給了我們什麼啟示？
  • 講者隱含論點：Boris 暗示「開發者」是縮短鴻溝的關鍵橋樑。你同意嗎？還是企業領導者的態度更關鍵？
  • 職涯意涵：如果你在一家採用速度線性的公司，你能做什麼來加速？推動 AI 導入需要什麼樣的內部政治策略？
  • 反向思考：有沒有可能「不採用 AI」也是一個合理的策略？什麼類型的公司可能應該放慢採用速度？

Q2. Lisa 警告：「隨著模型越聰明，過去精心設計的 Scaffolding 會成為枷鎖。」如何判斷自己的系統中哪些 Scaffolding 應該保留，哪些應該移除？

查看建議思考方向
  • 量化指標：是否有一個「Scaffolding 必要性測試」——移除後模型性能下降多少？如果下降不到 5%，可能就是枷鎖。
  • 講者隱含論點：Lisa 的建議其實是「以 Evals 說話」。你如何設計一個能偵測 Scaffolding 是否有用的評估機制？
  • 職涯意涵：對於一個每天寫複雜 Prompt 的工程師，「Scaffolding 減法哲學」意味著什麼？你的技能組合是否需要轉型？
  • 反向思考：有沒有反例？在哪些情況下，更多 Scaffolding 反而讓模型更好？（提示：工具調用的精確性控制）

Q3. MCP Tunnels 解決了「資料安全 vs. AI 能力」的核心矛盾。這個突破會如何改變金融、醫療、法律等高合規要求產業的 AI 採用速度？

查看建議思考方向
  • 歷史對照：VPN 的普及如何改變了遠端工作的可能性？MCP Tunnels 是不是 AI 版的 VPN？
  • 講者隱含論點：Caitlyn 展示的不只是一個技術功能，而是在說「企業最後一個不採用 AI 的藉口」已經被移除了。你同意這個判斷嗎？
  • 職涯意涵：如果你在金融業或醫療業，現在是不是應該重新評估你們對 AI Agent 的部署計畫？
  • 反向思考：MCP Tunnels 需要企業在私有網路中部署 Gateway，這本身有多少技術門檻？中小企業能否輕易採用？

Q4. Routines 讓「深夜的程式碼自動生長」成為可能。這個典範轉移——從「人主動向 AI 下 Prompt」到「人建構 Routine 讓 AI 自我提示」——會如何改變工程師的職業角色？

查看建議思考方向
  • 歷史對照：自動化工廠的出現讓工廠工人從「操作機器」變成「監督流程」。工程師的角色轉變是否相似？
  • 講者隱含論點：Boris 的演示暗示「能寫好 Routines（高階 Prompt）」將成為新時代的核心技能。你現在是否有這個能力？
  • 職涯意涵：如果未來 80% 的代碼由 Routines 自動產出，那工程師的核心價值將是什麼？如何確保自己不被邊緣化？
  • 反向思考：非同步工作也帶來了新挑戰——如何驗證 AI 在無人監督時做了什麼？「驗證（Verification）」是否會成為新的瓶頸？

Q5. Mercado Libre 的案例中，「技術 VP 重新開始寫程式」是一個令人動容的細節。Claude Code 如何改變了不同角色（PM、設計師、資深工程師）與程式碼的關係？

查看建議思考方向
  • 歷史對照：Excel 讓非財務背景的人也能做財務分析；Claude Code 是否正在成為工程界的 Excel？
  • 講者隱含論點：Kad 說「Shopify 的 PM、設計師、資料科學家也都在高頻率使用 Claude Code」。這意味著什麼？
  • 職涯意涵：如果你是 PM 或設計師，現在應該開始認真學習使用 Claude Code 嗎？不學會有什麼競爭劣勢？
  • 反向思考：「人人都能寫程式」是否會貶低軟體工程師的價值？還是反而提高了對「真正的工程判斷力」的需求？

Q6. Boris 用他 13 歲時用 TI-83 計算機寫程式的故事開場，強調「寫程式的純粹感動」。你認為 AI 是否讓這種感動變得更容易獲得，還是讓它消失了？

查看建議思考方向
  • 哲學層面：當 AI 幫你實現點子，那個創造的滿足感還屬於你嗎？「我創造了這個」和「Claude 幫我創造了這個」有本質差別嗎？
  • 講者隱含論點：Boris 相信 Claude Code 「還原了」寫程式的魔法感，而不是取走了它。你認同嗎？
  • 職涯意涵：對於正在考慮進入軟體工程領域的新人，AI 的存在是機會還是威脅？如果你是他們，你會如何規劃學習路徑？
  • 反向思考：Boris 說他「教同學用計算機程式通過考試」——本質上是用技術取巧。AI 是否只是這個故事的更大規模版本？這有道德問題嗎？

Task Horizon（任務地平線）：模型在迷失方向前能持續工作的時間長度。一年前為「分鐘」，現在為「小時」，未來目標「永續運行」。
術語
MCP Tunnels（MCP 隧道技術）：允許雲端 AI Agent 透過加密隧道安全存取企業防火牆內部工具，而無需將內部伺服器暴露在公開網路。
術語
Scaffolding（鷹架）：圍繞 AI 模型設計的輔助架構，包含提示模板、防錯機制、狀態機等。模型越聰明，過多 Scaffolding 越可能成為枷鎖。
術語
Routines（常規任務）：一種「高階 Prompt」，設定觸發條件（時間/Webhook/API），讓 Claude Code 自動在無人監督下執行任務。
術語
Advisor Strategy（顧問策略）：架構模式，讓小型低成本模型負責日常執行，大型前沿模型作為顧問在關鍵時刻介入。實現「前沿品質、親民成本」。
術語
Self-hosted Sandboxes（自我託管沙盒）：企業可選擇在自有的 Vercel、Cloudflare 等帳戶中建立代碼執行沙盒，確保機密代碼不流出企業環境。
術語
Wall-to-wall 採用（全面鋪開）：整個組織所有角色（工程師、PM、設計師）全面採用某項工具，而非僅在特定部門試點。
