從原型到生產的三大系統性挑戰

①

長文本多步驟推理：代理必須在超長的上下文窗口中維持連貫的多步驟推理——不只是回答一個問題，而是在複雜任務中持續追蹤目標和進度。

②

系統可靠性：企業客戶要求高 SLA、內置可觀測性和完整的安全防護——模型能力強不夠，整個平台必須穩定可信。

③

外部工具與數據連接：代理必須能接入真實的業務系統——CRM、ERP、監控工具、通訊平台——才能真正替代人工完成業務任務。

給初學者 · 什麼是「Agentic Systems」？
從「問答機器」到「能辦事的數位員工」

傳統 AI 對話：你問 → AI 答（一問一答，每次互動獨立）。代理系統（Agentic Systems）：你給一個目標 → AI 自己規劃步驟 → 自己調用工具 → 自己做多步驟推理 → 最終完成目標（可能需要數十個步驟）。

  • 問答模式：「SAP 系統裡有哪些未付款的發票？」→ AI 告訴你清單
  • 代理模式：「處理本月所有未付款發票」→ AI 自動登入 SAP、篩選發票、發送提醒郵件、更新追蹤表格

關鍵差異：代理系統需要「記憶」（知道做到哪）、「規劃」（決定下一步）和「執行」（呼叫外部工具）的能力。

Microsoft × Anthropic：Better Together 的具體體現

與 AWS + Anthropic 的合作不同，Microsoft Foundry 的整合重點在於將 Claude 嵌入 Microsoft 已有的企業生態系（Azure、M365、Copilot Studio、VS Code），讓企業的現有投資直接受益於 Claude 的能力提升。

Claude 的頂級推理能力（Opus 4.7、Sonnet 4.6）、長文本理解、精確的工具呼叫執行——這些讓代理能真正完成複雜任務，而非只是表演。

Microsoft 帶來的

企業安全框架（Defender + Purview + Entra ID）、1,400+ 企業連接器、Azure 的全球基礎設施、VS Code 和 GitHub 的開發體驗——讓 Claude 能安全地在生產環境中落地。

Microsoft Foundry 平台架構

Microsoft Foundry 是 Microsoft 為大規模構建 AI 應用和代理打造的統一平台。它的定位不是「另一個 AI API」，而是「把 AI 代理從原型帶到生產的完整企業基礎設施」。

🤖

Foundry Models

原生支援 Claude Sonnet 4.6、Opus 4.7 等 Anthropic 模型，直接在 Playground 中測試和對比不同系統提示詞的效果

⚡

Agent Service

內建的代理托管服務，提供代理編排（Orchestration）、工具執行管理和 Session 狀態持久化，無需自建托管基礎設施

🔌

工具與連接器

1,400+ 內建連接器，支援 SAP、ServiceNow、Microsoft 365、GitHub、Slack 等企業系統，代理可直接在真實業務系統上採取行動

📊

ML 服務

內建模型評估、提示詞優化、微調（Fine-tuning）等 ML 工具，讓代理從實驗走向優化的迭代循環更高效

多工具支援：在你喜歡的環境工作

Foundry 支援開發者在以下環境中工作，不需要切換生態：

  • GitHub：直接在 Repo 中管理代理代碼，整合 CI/CD 流程
  • Visual Studio Code：本地 IDE 直接連接 Foundry 端點，支援本地測試
  • Copilot Studio：低代碼介面，適合非工程師的業務分析師構建代理
給初學者 · 什麼是「統一平台」？為什麼重要？
避免「AI 工具孤島」的系統整合挑戰

沒有統一平台時，企業 AI 開發通常長這樣：

  • 用 A 家的 API 呼叫模型
  • 用 B 家的工具做向量搜尋
  • 自己寫代碼整合 C 家的監控
  • 再寫代碼整合公司現有的 IT 安全系統

每個點都需要維護，任何一個環節出問題都可能影響整個代理。「統一平台」的意義在於：這些整合由平台提供商負責維護，開發者只需專注於業務邏輯。

III

MCP 整合實戰：打破 AI 的數據孤島

MCP（Model Context Protocol，模型上下文協定）是 Anthropic 推出的開放標準，讓 AI 代理可以透過統一的介面存取任何外部系統的函數、提示詞和資源。Marlene 用杯子蛋糕店的 MCP 伺服器，直觀地展示了 MCP 的必要性。

Claude 代理

強大的推理能力，但只知道訓練資料截止前的靜態知識——不知道「今天有哪些杯子蛋糕口味」

⇌

MCP 伺服器

連接外部業務系統的橋樑——動態提供函數、提示詞、資源；代理透過這個標準介面「看見」真實世界

給初學者 · MCP 解決了什麼根本問題？
AI 代理與外部世界的「通用翻譯官」

每個外部系統（GitHub、SAP、你的自製資料庫）都有自己的 API 格式。如果沒有統一標準，每接一個系統就要寫一段不同的整合代碼。

MCP 是一個統一的標準：

  • 系統提供商：「我按照 MCP 標準發布我的 API」
  • AI 代理：「我按照 MCP 標準讀取任何系統的功能」
  • 開發者：「一次整合，到處可用」

類比：就像 HTTP 是網頁的通用協定，讓所有瀏覽器都能讀取任何網站——MCP 是 AI 代理存取外部系統的通用協定。

MCP 伺服器的三種資源類型

Functions（函數）

代理可以呼叫的操作——如「查詢庫存」「建立用戶帳號」「發送 Slack 通知」

Prompts（提示詞）

可複用的指令片段——如「Sparkles 客服代理的歡迎語」，由 MCP 統一管理，確保所有代理使用一致的語氣

Resources（資源）

可透過 HTTP 傳輸給代理的結構化數據——如「今日可用口味清單」「優惠券代碼」等動態業務數據

"With MCP, one of the great things is you just need a URL to plug it in, and you get a whole complete API, formatted in a way that's very easy for your agent to read." 「有了 MCP，其中一個很棒的優點是你只需要一個 URL 把它接入，你就能直接獲得一整套完整的 API，而且格式非常易於讓你的代理讀取。」

— Marlene Mangami

MCP 的「可複用性」優勢

在杯子蛋糕工作坊中，Marlene 展示了一個實用的 MCP 設計模式：把提示詞（Prompts）存在 MCP 伺服器上，而不是硬編碼在每個代理裡。

傳統做法：提示詞硬編碼

系統提示詞寫在每個代理的代碼裡。更新歡迎語？要修改每個部署版本。加入新的語氣指南？又要重新部署所有代理。

MCP 做法：提示詞集中管理

歡迎語和代理指令存在 MCP 伺服器上。更新歡迎語？修改 MCP 伺服器一個地方，所有代理下次啟動時自動取得最新版本。

給初學者 · 為什麼「集中管理提示詞」很重要？
AI 的「配方書」不應該分散在每個廚師手上

想像一家連鎖餐廳：如果每家分店都有自己的一份食譜，當你想更新一道菜的配方，就要通知每一家分店更新——費時費力，容易漏掉某家。

集中管理的「中央配方書」（MCP 伺服器的 Prompts）讓所有分店（代理）從同一個來源取得最新配方——一次更新，全面生效。

對企業來說，這意味著「合規語言」「品牌聲音」「風險警語」等必須一致的指令可以被統一管理，而不是分散在數十個代理的代碼裡。

MCP 教學時刻 · 「沒存檔」意外成為最佳演示

Claude 誠實地說「我不知道」——而不是胡說

在 Demo 中，Marlene 向代理問「你們今天有什麼口味？」代理回答：「我可能有些困惑，我是一個 AI 助手……」

原因很簡單：她在代碼中加了 MCP 連接，但沒有存檔——Claude 沒有讀到 MCP 數據，於是坦誠地表示它不知道。

這個意外揭示了一個重要特性：Claude 選擇誠實地表示「不知道」，而不是憑空捏造口味清單。 這正是把 Claude 用於業務代理的重要理由——在上下文不足時，它不會幻覺，而是如實反映自己的知識邊界。

IV

企業安全：告別 .env 管理噩夢

Marlene 用一句話點出了企業 AI 部署的最大痛點：「你不需要整天在處理那些 .env 環境變數檔案，我們真的非常專注於安全。」這句話引發了現場工程師的共鳴。

為什麼 .env 在生產環境是安全定時炸彈？

工作坊中，Marlene 當場展示了如何把 API 金鑰貼進 .env——她自己也說：「雖然在台上直接把金鑰貼出來不是很理想，但今天沒關係。」

給初學者 · 什麼是 .env 檔案？它為什麼危險？
把密碼寫在一個「希望沒人看到」的文字檔裡

.env 是存放環境變數的設定檔，通常包含 API 金鑰、資料庫密碼、服務端點等敏感資訊。

問題在於：

  • 開發者常常不小心把 .env 提交到 Git（GitHub 上有數以萬計的洩漏金鑰）
  • 生產伺服器的 .env 需要手動同步，容易出現版本不一致
  • 員工離職後，他們曾使用的 API 金鑰往往沒有被及時撤銷
  • 難以稽核「誰在何時用了哪個 API 金鑰做了什麼」

Microsoft Foundry 的解法：把憑證管理完全交給 Entra ID（企業身份系統），代理的存取權限和員工的 IT 帳號掛鉤——員工離職，代理存取權自動撤銷；所有行為都有完整的稽核軌跡。

Foundry 的三層安全架構

1
Microsoft Entra ID（身份管理）

取代 .env 中的 API 金鑰管理——代理的存取權限由企業統一的身份系統控制，支援 SSO、OAuth、IAM Role，與現有 IT 治理框架無縫整合

2
Microsoft Defender（威脅防護）

企業級的安全威脅偵測和防護，自動監控代理的行為模式，偵測異常調用（如突然大量呼叫外部系統）

3
Microsoft Purview（數據合規）

數據治理與合規框架，確保代理在處理敏感資料時符合 GDPR、HIPAA 等法規——自動標記、分類和保護敏感數據

"You don't need to be dealing with .env files all day. We're really focused on security — agents built on Foundry come with built-in security, observability, and governance." 「你不需要整天在處理那些 .env 環境變數檔案。我們真的非常專注於安全——基於 Foundry 構建的代理，開箱就具備內置的安全防護、可觀測性和治理功能。」

— Marlene Mangami

在 Foundry Playground 中直接對比不同模型、不同系統提示詞的效果，無需額外搭建評估基礎設施。

Foundry 提供免費額度讓開發者試用，降低入門門檻；完整學習資源在 Microsoft Learn 上免費提供。

V

思考題與討論方向

Microsoft Foundry + Claude 的組合代表了「雲端供應商整合 AI」的企業路線圖。以下六個問題邀請你思考這個方向的機遇、風險與深層意義。

Q1 — Marlene 強調 Foundry 讓開發者「從構建原型走向生產環境」。但「生產就緒」和「可信任」是兩件不同的事。你認為，要讓一個 AI 代理真正「可信任」到可以在生產環境自主運行，需要滿足哪些條件？

查看建議思考方向
  • 1. 量化可信任度：對比航空業的「適航證書」制度——飛機不只是「能飛」，而是在特定條件下「被認證為安全」。AI 代理的「適航認證」標準應包含哪些維度（準確率門檻、失敗降級機制、監控覆蓋率）？
  • 2. 隱含論點：Marlene 的工作坊把「生產就緒」定義為「有安全框架 + 可觀測性」。但真實的業務風險（代理做出錯誤決策的代價）是否需要更高的標準？
  • 3. 職涯啟發：「AI 代理的生產可信任度評估」正在成為新的工程專業。如果你要設計你公司第一個生產級 AI 代理的評估框架，你會從哪裡開始？
  • 4. 反向思考：過於嚴格的「可信任標準」可能讓 AI 代理永遠停在測試階段。如何平衡「等到足夠可信再部署」和「在真實使用中積累可信度」的兩難？

Q2 — 杯子蛋糕 Demo 中，Claude 在缺乏上下文時選擇誠實地說「我不知道」，而非捏造答案。這種「誠實性」對業務代理至關重要，但 Claude 在哪些情況下可能仍會產生幻覺（Hallucination）？如何在代理設計中系統性地防範？

查看建議思考方向
  • 1. 幻覺的觸發條件：研究顯示 LLM 在「問題涉及最近事件」「問題有隱含前提但前提是錯的」「用戶的問法強烈暗示某個答案」時更容易幻覺。對業務代理設計有什麼啟示？
  • 2. Grounding 機制的作用：AWS Bedrock 的 Grounding 和 Foundry 的 Guardrails 都提供幻覺控制機制。這些機制的原理是什麼？它們能捕捉到哪些類型的幻覺，哪些類型仍然無法防範？
  • 3. 職涯啟發：「設計 Anti-hallucination 機制」正在成為 AI 工程師的核心技能。在你熟悉的業務場景中，哪些代理任務的幻覺代價最高？應該怎麼設計防護？
  • 4. 反向思考：有時候「誠實地說不知道」會讓用戶體驗很差（代理無法完成任務）。從用戶體驗的角度，如何設計「優雅地承認無知，同時仍然提供價值」的代理行為？

Q3 — Microsoft 的 1,400+ 連接器生態系讓 Claude 代理可以「直接在真實的系統上採取行動」——包括 SAP、ServiceNow 等核心業務系統。這個「直接行動」的能力應該如何被限制和治理？誰應該有權授權代理在哪些系統上採取什麼行動？

查看建議思考方向
  • 1. 責任鏈問題：當 AI 代理在 SAP 中刪除了一批財務記錄（即使是因為代理「理解錯誤指令」），責任歸屬是怎樣的？代理開發者、系統管理員、授權用戶各自的責任邊界在哪裡？
  • 2. 最小權限原則：在安全設計中，「最小權限」是基本原則——每個系統只能訪問完成任務所需的最少資源。如何把這個原則應用到 AI 代理的連接器授權設計？
  • 3. 職涯啟發：「設計 AI 代理的授權治理框架（Agent Authorization Governance）」正在成為企業 IT 架構師的新工作。你認為這個框架的核心要素是什麼？
  • 4. 反向思考：過於細粒度的授權（代理必須申請每一個操作的單獨許可）會讓代理效率大幅降低。如何在「充分授權以提升效率」和「嚴格限制以降低風險」之間找到平衡？

Q4 — Foundry 把代理「嵌入 Microsoft 企業生態系」的策略，和 AWS 把代理「嵌入 AWS 雲端基礎設施」的策略，代表了兩種不同的 AI 企業市場進入路徑。哪種路徑在長期更有競爭優勢？

查看建議思考方向
  • 1. 歷史量化：微軟 Office 系列在企業市場的統治地位，很大程度來自「整個組織都在用同一套工具」的網路效應。AI 代理是否也會形成類似的企業生態鎖定效應？
  • 2. Copilot vs. Foundry 的市場定位：Microsoft Copilot 面向一般用戶，Foundry 面向開發者。這個雙軌策略如何幫助微軟覆蓋「不同技術水平的企業用戶」？Anthropic 在這個生態中的角色是什麼？
  • 3. 職涯啟發：企業 AI 的競爭格局中，AWS Bedrock、Microsoft Foundry、Google Vertex AI 三足鼎立。作為一個要學習 AI 開發的工程師，選擇哪個平台作為主要技能方向？
  • 4. 反向思考：深度嵌入某個雲端生態的代理，是否會面臨「供應商鎖定（Vendor Lock-in）」的風險？如果企業決定從 Microsoft 遷移到 Google，這些代理的可攜性如何？

Q5 — MCP（Model Context Protocol）是 Anthropic 提出的開放標準，但現在已被 Microsoft、AWS 等主要雲端供應商採用。這種「Anthropic 設計標準、行業廣泛採用」的模式，對 Anthropic 的競爭地位意味著什麼？

查看建議思考方向
  • 1. 標準制定的戰略價值：對比 TCP/IP（ARPA 設計、全世界採用）——設計者不一定是最大受益者。Anthropic 設計 MCP 的目的是什麼？它如何從廣泛採用中獲益？
  • 2. 競合關係的複雜性：如果 GPT-4 也支援 MCP，那 Claude 的 MCP 生態護城河就消失了。Anthropic 如何在「推動 MCP 開放標準化」和「保持 Claude 差異化優勢」之間取得平衡？
  • 3. 職涯啟發：「掌握 MCP 開發」正在成為 AI 工程師的通用技能（類似掌握 REST API 設計）。你認為未來 5 年，MCP 會成為像 REST 一樣的行業標準嗎？
  • 4. 反向思考：開放標準有時會讓差異化變得更難——如果所有 AI 都用 MCP，那「誰的 AI 接 MCP 接得最好」的競爭點在哪裡？是模型能力、工具呼叫精確度，還是其他維度？

Q6 — Marlene 示範的工作坊，讓非工程師（至少技術門檻很低的用戶）也能在「Skillable 虛擬環境 + 點擊填入」的幫助下完成代理構建。AI 代理的「低代碼化」趨勢，對軟體工程師職業的長期影響是什麼？

查看建議思考方向
  • 1. 量化歷史：Excel 的出現讓「不懂程式的業務分析師」能做複雜的數據分析——它降低了對工程師的需求嗎？還是它讓工程師可以把時間花在更高價值的工作上？AI 代理的低代碼化是否會走同一條路？
  • 2. 「低代碼」的天花板：Copilot Studio 適合業務分析師，Foundry 適合開發者——這個分層本身說明了什麼？複雜的企業代理為什麼仍然需要工程師？
  • 3. 職涯啟發：如果低代碼工具能覆蓋 70% 的代理構建需求，工程師應該把技能重點放在那剩下的 30%（需要深度客製化、系統整合、安全治理的場景）。你認為這 30% 最需要哪些技能？
  • 4. 反向思考：「降低技術門檻」不等於「降低出錯機會」——沒有工程背景的用戶可能更容易構建出「看起來能用但有安全漏洞」的代理。低代碼 AI 代理的品質問題誰來負責？

名詞速查表

A

Agentic Systems（代理系統）：具備規劃、推理和執行能力的 AI 系統，能自主完成多步驟任務，而非只回答單一問題。

B

Microsoft Agent Framework：Microsoft 開源的 Python/TypeScript 代理框架，提供 Track Client 等工具，讓本地代理代碼可以直接連接 Foundry 端點。

C

Microsoft Entra ID：企業身份與存取管理系統（前稱 Azure Active Directory），Foundry 用它取代 .env 中的 API 金鑰管理，與企業現有 IT 治理框架整合。

D

Microsoft Foundry：Microsoft 的統一 AI 平台，支援在 Claude 等 1,400+ 模型和連接器的基礎上大規模構建 AI 代理，內建企業級安全（Defender + Purview + Entra ID）。

E

MCP（Model Context Protocol，模型上下文協定）：Anthropic 提出的開放標準，讓 AI 代理透過統一介面存取外部系統的函數、提示詞和資源；只需一個 URL 即可獲得完整 API 存取。

F

Observability（可觀測性）：讓工程師能實時監控系統的指標、日誌和追蹤鏈的架構能力，是生產環境 AI 代理的必備特性。

G

Skillable：用於工作坊和培訓的虛擬化實作環境平台，可自動提供預配置的開發環境，讓學員無需自行安裝工具即可跟著操作。
