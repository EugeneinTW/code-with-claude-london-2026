I

瓶頸從智商移向基礎設施

三年前的限制是模型不夠聰明。今天的限制是圍繞模型的基礎設施不夠完善。這個認知轉移，是整場對談的核心前提。

"We found that the bottleneck to improving AI capability had actually become the infrastructure surrounding these models, not the intelligence of the models themselves." 「我們發現，提升 AI 能力的瓶頸，實際上已經變成了圍繞著這些模型的基礎設施，而不是模型本身的智能程度。」

— Harrison, VP Technology, Anthropic

給初學者 · 為什麼「基礎設施瓶頸」很重要？
模型再聰明，沒有好的「跑道」也飛不起來

想像你有一輛 F1 賽車（超強模型），但跑道（基礎設施）是泥路。賽車的馬力根本發揮不出來。Anthropic 發現：

  • 最新模型的智能已超過大多數任務的需求上限

  • 解法：把跑道（基礎設施）造好——Managed Agents 就是鋪路的工程

II

四大沙盒夥伴的底層賭注

自我託管沙盒讓企業可以把代理人的執行環境帶回自己的 VPC，同時享用 Managed Agents 的所有上層服務。四大夥伴各有截然不同的技術哲學。

Vercel · Luke

流動運算（Fluid Compute）

所有基礎設施建構在同一核心之上，函式和沙盒可以無縫互通。沙盒防火牆讓你過濾流量並注入憑證，一次建構、處處重用。

賭注：VM 與函式的互通性
Modal · Ashot

規模與速度的極致

自研調度器在幾分鐘內拉起數十萬個沙盒，全球區域部署確保低延遲。GPU 沙盒是持續增長的核心使用場景——強化學習、推論優化。

賭注：大規模 + GPU 沙盒
Daytona · Ivan

「人需要的，Agent 也需要」

提供多規格（CPU/RAM/OS）、可暫停、可恢復、可分叉的完整沙盒。Agent 能夠多路並行嘗試不同結果，就像人類的多工思維。

賭注：Agent 生命週期管理
Cloudflare · Mike

雙軌制：MicroVM + Isolates

MicroVM 給「像開發者一樣行動」的代理人；Isolates 給未來海量微型代理——毫秒啟動、單進程多租戶、激進擴展。

賭注：Isolates 的規模化優勢

MicroVM vs Isolates：技術對比

MicroVM（微型虛擬機）
✓完整的 Linux 環境
✓可運行 CLI 工具、編譯 Rust
✓GPU 支持（Modal, Daytona）
✓可暫停、可恢復、可分叉
✗啟動速度較慢（秒級）
✗資源消耗較高
✗海量代理場景成本高
Isolates（沙盒隔離）
✓毫秒級啟動
✓單進程多租戶，資源效率極高
✓可寫入文件、執行任意代碼
✓為海量代理爆發設計
✗不支持完整 Linux 環境
✗不直接支持 GPU
✗複雜開發任務能力受限
給初學者 · 為什麼 Cloudflare 賭 Isolates？
當 AI 成本像電費一樣便宜時，會出現什麼？

Mike（Cloudflare）的核心論點：兩年後，Sonnet 級別的模型將能在 Opus 的規模上運行，而且成本極低。到那時，你不需要一個能幫你「完成複雜開發任務」的大型代理人——你需要的是數十億個執行簡單任務的微型代理人。

  • 微型代理：「幫我檢查這個網頁是否可訪問」「幫我發送這封郵件」
  • 這類任務不需要完整 Linux 環境，但需要極快啟動速度
  • Isolates = 為這個未來設計的基礎設施
未來應用 · 每個工程師都有了幕僚長

過去是高階主管的特權，現在每個人都有

Vercel 的 Luke 分享了他對這個技術浪潮最激動的想法：

「每個人現在都擁有了一位幕僚長。這過去是一個高階主管的職位，但現在每位工程師都可以擁有——幫你做市場研究、試著進一步了解你正在開發的產品、客戶如何使用它等等。」

過去需要「同步開會」決定的事（產品對齊、市場研究）現在可以委派給代理人在後台非同步執行，工程師只需要在需要時查看結果。

GPU 沙盒 · DoorDash 的帳戶管理自動化

Claude 自己運行 Nvidia Profiler，讓推論效能爬山優化

Modal 的 Ashot 分享了一個令人印象深刻的應用：

「我們讓 Claude 去優化推論。你給它一個工作負載和一個基準測試，它基本上就會進行爬山演算法並讓它變得更好。它會運行 Nvidia 的 Profiler、去讀取 Profile，然後全力以赴把事情做好。」

關鍵：這整個過程運行在 Modal 的 GPU 沙盒上。未來，這將成為在 Slack 中運作的背景代理人——工程師不需要手動調優，代理人自己做。

給初學者 · 什麼是 Dreaming？
Dreaming = AI 版本的「睡眠記憶固化」

人類在睡覺時，大腦會反覆重播白天的記憶，整理哪些重要、哪些可以遺忘。Dreaming 讓 Claude 做同樣的事：

  • 在代理人「空閒時」並行處理成千上萬個過去的 Session
  • 反思：「這次對話中我學到了什麼新知識？」
  • 更新長期記憶：刪除過時記憶、強化重要知識
  • 結果：每天自動「更聰明」，不需要人類手動更新

Dreaming 讓記憶從「人工策展」走向「自主進化」。

Pascal 演示：代理人觀測的三個層次

Harrison 透過虛擬雜貨分析代理人 Pascal 展示了 Managed Agents 的 Console 體驗：

即時觀測
Console 展示事件流中的所有工具調用、Agent 事件——即時了解代理人「正在做什麼、為什麼這麼做」
代理人定義
右側欄顯示 System Prompt、模型選擇、MCP 工具配置——一目了然代理人的「身分與能力清單」
Ask Claude 按鈕
點擊後 Claude 自動閱讀 Session 逐字稿，提出優化建議——示範中識別到一個 Python 腳本執行超過 20 秒，建議優化
持續學習 · 每天早晨都有更聰明的模型

Daytona 的客戶如何用 RL 讓模型日日進化

Daytona 的 Ivan 分享了最令他興奮的應用場景：

「我們有一些客戶，他們基本上正在讓客戶能夠針對他們每天、每週、每個月所擁有的數據去進行強化學習。然後到了隔天早上，他們實際上就擁有了一個更聰明的模型。」

這是「Continuous Learning」的最直接應用——企業私有數據每天驅動模型微調，模型品質呈現漸進式提升，而不是等待下一個基礎模型發布。

IV

Auth Chain：生產部署的阿基里斯腱

把代理人從「感覺編碼」推向真正的生產環境，最大的阻礙不是模型能力，而是身分認證鏈（Auth Chain）——當代理人開始生成其他代理人時，誰負責什麼、被允許做什麼，整個行業尚無答案。

"When that agent then goes and creates another agent, and that agent goes and creates another site, how do we correctly push identity all the way down the chain and filter egress traffic?" 「當那個代理人又去建立了另一個代理人，而那個代理人又去建立了另一個站點時，我們該如何正確地沿著這個鏈條一路向下分配身分，並過濾出站流量？」

— Mike, Cloudflare

認證鏈樹狀圖

用戶（人類）明確授權
↓派生
Agent A（Coordinator）繼承什麼權限？
↓派生
Agent B（Specialist）繼承 A 的哪些權限？
↓建立
外部服務/站點應接受哪個身分？
❓ 目前問題：整個行業尚無統一協議規範如何在代理人樹中傳遞身分 Token 與過濾出站流量（Egress）
給初學者 · 為什麼 Auth Chain 是難題？
你能授權給你的代理人，但它能再授權給它的子代理人嗎？

想像你授權你的助理（Agent A）可以代表你訂飯店。助理又委託另一個助理（Agent B）去比價。Agent B 需要訪問你的信用卡信息嗎？它應該有多少你的「身分」？

  • 人類世界：有「委任書」這類法律工具規範授權邊界
  • AI 代理人世界：這套「委任書」協議還沒有建立
  • 當前風險：代理人可能攜帶過多或過少的權限執行任務
  • 最危險的場景：Sub-agent 被惡意 Prompt Injected，用繼承的完整權限執行惡意操作

四大安全挑戰

Egress 過濾
如何正確過濾代理人的出站流量？確保它只能訪問被授權的外部系統，不能「打電話回家」
身分分配
如何為代理人分配正確的身分？它代表誰行事？能存取什麼數據？這個身分在多代理人鏈中如何傳遞？
跨公司認證
如何讓不同公司的服務都能理解和驗證代理人的身分？目前沒有行業標準的「代理人認證語言」
Vibe-coded 到 Production
大多數公司內部部署的代理人仍是「感覺編碼」級別，推向生產環境將面臨全面安全審查

多人遊戲邏輯被打破

Vercel 的 Luke 提出了另一個深刻的挑戰：當用戶從人類轉變為代理人時，協作模式需要根本性重建。

人類協作的特點

人類可以「分道揚鑣（Diverge）」——根據彼此反饋隨時調整計劃，一個人說「我認為計劃改變了」，大家都跟著調整。

代理人協作的困境

代理人一旦進入軌道就「極致追目標」——它知道要達到哪個函式，就衝刺過去，無法像人類一樣靈活「中途改道」。

給初學者 · 什麼是 Resumability（任務可恢復性）？
代理人被打斷後，如何從原來的地方繼續？

想像你讓一個代理人去幫你做一個大型研究報告，需要三天。第二天你的電腦重啟，代理人的狀態全丟了。Resumability 是指：

  • 代理人的執行狀態被持久化儲存（不只是對話記錄）
  • 重新啟動後，代理人知道「我做到了哪裡」，繼續往下做
  • 甚至幾週後重新接入，代理人能「記得」上次的進度和上下文

Resumability 是長任務（M&A 分析、大型代碼重構）變得可行的前提。這也是 Daytona「暫停/恢復」功能的核心價值。

未來基礎設施 · 「時空旅行」與儲存設計空間

當代理人能夠分叉並回溯時，調試將徹底改變

Modal 的 Ashot 分享了他對下一代基礎設施的思考：

「你能不能生成一個子代理人，從你目前的狀態分叉出去，然後也許它們可以進行時空旅行（Time travel）並回到先前的狀態？」

這不只是技術上的噱頭，而是實際工程需求：當代理人嘗試了多條路徑，其中一條走進了死胡同，「回溯到分叉點」的能力意味著你不需要從零開始。Daytona 的「分叉（Forking）」功能已是這個方向的初步實現。

Q3 · Mike（Cloudflare）直言「Auth Chain 是大規模部署的阿基里斯腱，整個行業在認證協議上尚未統一」。你認為解決這個問題最可能的路徑是什麼——標準委員會、大廠主導、還是市場自然演化？

查看建議思考方向
  • 歷史參照：OAuth 2.0 解決了「第三方服務授權」問題，花了多久才成為事實標準？AI 代理人的 Auth Chain 問題是否比 OAuth 更複雜？
  • 講者隱含論點：Mike 說「我們應該要統一（協議）」——這暗示他認為市場自然演化不夠快，需要行業共同行動
  • 職涯啟發：如果你的公司今天要部署多代理人系統，在行業標準未定的情況下，你如何設計「臨時的」Auth Chain 架構，同時讓它易於遷移到未來標準？
  • 反向思考：是否有場景讓「Auth Chain 不統一」反而是優勢？（各公司用私有協議建立競爭壁壘？）

Q4 · Dreaming 讓 Claude 在後台自主更新長期記憶。但「自主修改記憶」帶來了一個深刻問題：如何確保 Claude 不會在 Dreaming 過程中形成偏見、強化錯誤記憶、或「遺忘」對企業重要但低頻的知識？

查看建議思考方向
  • 歷史參照：人類的記憶也會隨重複強化——政治廣告通過反覆曝光改變記憶，AI 的 Dreaming 機制有沒有類似的「漸進式偏移」風險？
  • 講者隱含論點：Harrison 把 Dreaming 定位為讓長期記憶「始終是最頂尖的」——但誰來定義什麼是「頂尖」？這個評分標準由誰設定？
  • 職涯啟發：如果你負責設計一個企業的 AI 記憶系統，你希望有哪些「記憶稽核」機制，讓人類能檢查和干預 Dreaming 的結果？
  • 反向思考：是否有場景讓「Dreaming 關閉」反而是更好的選擇？（如法律合規系統，需要記憶是不可變的歷史記錄）

Q5 · Luke（Vercel）說每個工程師現在都有了「幕僚長」。Ivan（Daytona）則提出「數字工人（Human Emulator）」——代理人可以登入 legacy 系統、安裝 Windows 應用。這兩個願景對「工作」的本質有什麼不同的預設？

查看建議思考方向
  • 歷史參照：助理（Chief of Staff）模型是「增強人類決策」的，而工廠機器人模型是「取代人類勞動」的——AI 代理人更接近哪個？兩者可以共存嗎？
  • 講者隱含論點：Luke 的「幕僚長」框架假設人類仍在決策中心；Ivan 的「數字工人」框架暗示某些任務可以完全由代理人自主完成——你認為哪個更快到來？
  • 職涯啟發：在「幕僚長」和「數字工人」兩種模式下，工程師需要的技能有何不同？哪個框架對你的職涯規劃更具指導意義？
  • 反向思考：「數字工人」進入 legacy 系統的場景帶來了什麼額外的安全風險，是「幕僚長」模式不存在的？

Q6 · Luke（Vercel）指出：人類協作可以「分道揚鑣」，但代理人一旦進入執行模式就會「極致衝刺」。如何設計一個代理人協作系統，讓它既能自主高效執行，又能在計劃改變時靈活調整方向？

查看建議思考方向
  • 歷史參照：軍事上的「指揮官意圖（Commander's Intent）」——士兵被允許在細節上偏離命令，只要符合更高層的目標意圖。這能否移植到代理人設計中？
  • 講者隱含論點：Luke 描述的問題本質是代理人缺乏「元認知」——它知道如何做，但不知道什麼時候應該停下來重新評估。如何注入這種能力？
  • 職涯啟發：如果你是 Agent Framework 的設計者，你如何在「自主執行效率」和「計劃靈活性」之間找到平衡點？具體機制是什麼？
  • 反向思考：是否有任務類型讓「極致衝刺、不分道揚鑣」恰恰是理想的代理人行為？（如：高度結構化的合規檢查？）
