I

核心論題

Prompting 不是感覺，而是工程紀律。Margot Vanlar 帶領我們走過兩個最真實的場景——維護一個千行爛 prompt，以及從零打造一個排班 Agent——展示如何用評估套件、XML 結構與多代理迴圈讓 AI 系統真正可靠。

"Prompting is arguably one of the first skills, if not the first skill, that we had to learn as engineers when we first started to work with LLMs — and even now it continues to be one of the most critical skills to building effective AI systems." 「Prompting 可說是我們工程師最早需要學習的技能之一——即便到現在，它仍然是建構有效 AI 系統最核心的技能之一。」

— Margot Vanlar, Applied AI Engineer, Anthropic
給初學者 · 一句話搞懂 Prompting
什麼是 Prompt Engineering？

Prompt 是你寫給 AI 的「工作說明書」。但和一般員工不同，AI 字面理解每一句話——說錯了就會錯、說太多了會混亂、說太少了會亂猜。

  • 好的 Prompt：結構清楚、角色分明、規則一致
  • 壞的 Prompt：一大段、混雜新舊規則、靠「語氣」要求結果
  • 最壞的誤解：「寫 CRITICAL: 這很重要！」以為會讓模型更努力

Prompt Engineering 的本質是：設計讓模型做出你期望行為的說明系統。

本講兩大場景

場景 A

維護現有 Prompt

模型升級後，測試案例突然失敗。堆疊多年的防禦性補丁、無主的協作 Prompt、混雜的政策與歷史規則——如何系統性清理？

場景 B

從零打造新 Agent

面對複雜的零售排班問題（多員工、硬性限制、長輸出），如何從最小 Prompt 開始、選對模型、最終用 Generate-Evaluate-Repair 迴圈解決？

十大核心洞見

01
評估先行——任何改動之前，先建立 Eval Suite。沒有評估，你不知道是否真的進步了。
02
Eval 三型案例——控制組（必過基準）、邊緣案例（曾失敗的規則測試）、邊界案例（該拒絕 or 轉接真人的情境）。
03
XML 標籤立竿見影——將 role / policy / guidelines / tone 分開標注，立刻提升模型對規則的理解與遵循。
04
Stop Sequence > 口頭規範——要求 JSON 輸出，用 API 層 stop_sequences 比在 Prompt 裡叮囑更可靠。
05
版本控制防禦性補丁——記錄每個防禦性指令的原因，否則新模型會把舊補丁過度優化。
06
指令 ≠ 能力——寫「務必精確計算」無法讓模型更擅長心算。給工具（Tool）才能真正擴展能力。
07
說完整故事——不要只告訴模型「避免 X 的成本」，也要說「不避免 X 的損失」，模型越聰明越需要全貌。
08
模型 vs Prompt vs 架構——當一個 Prompt + 大模型解不了時，考慮架構拆分，而非無限加大模型。
09
Generate-Evaluate-Repair——三個簡單 Prompt 的迴圈，成本和延遲都低於 Opus 4.7 + Adaptive Thinking。
10
軟性約束 Runtime 注入——GER 迴圈的評估 Prompt 可以在執行時動態加入軟性限制，無須修改後端程式碼。
II

案例背景：Meridian Mobile

本演講以電信客服機器人「Meridian Mobile」為範例 Prompt——一個被多人協作維護、混雜各時期補丁、在模型升級後出現多個測試失敗的典型生產環境 Prompt。

給初學者 · 什麼是生產環境 Prompt？
為什麼 Prompt 會變得「爛掉」？

想像一份員工手冊，一開始只有10條規定，三年後有50人各自加了一條，還有人加了「如果碰到上次的問題，不要這樣做」。沒有人知道哪條規定是為什麼加的，有些規定甚至互相矛盾。

  • 無明確擁有者：多人協作，誰都可以加，沒人負責刪
  • 歷史補丁堆積：每次出問題就加一條禁令，從不移除
  • 模型遷移失效：舊模型的防禦規則，新模型過度遵循

這就是為什麼「先有 Eval，再動 Prompt」是工程紀律，而不只是建議。

✓
控制組 · Control Cases

無爭議基準案例

模型必須穩定通過的基本問題。例：「基本方案的數據上限是多少？」不應受任何調整影響。

⚠
邊緣案例 · Edge Cases

曾失敗的問題

以前出過錯、因此加了防禦規則的案例。例：按比例計費計算、帳務爭議升級。用來驗證防禦規則是否仍有效。

⛔
邊界案例 · Boundary Cases

能力邊界與拒絕測試

模型應該拒絕回答或轉接真人的情境。例：複雜帳務爭議 → 應升級給客服人員而非自行診斷。

I

Prompt 通識衛生

在針對失效模式做出修改之前，先做一輪「通識衛生清理」——就像整理一間凌亂的房間，讓模型（和你自己）能清楚看到規則的全貌。

給初學者 · Prompt 衛生是什麼？
為什麼 Prompt 需要「整理」？

如果一份工作說明書讓你都看不懂哪段是規定、哪段是舉例、哪段是警告——模型也看不懂。這不是模型能力的問題，是資訊組織的問題。

  • 把所有規則塞在一段 → 模型不知道優先級
  • 從網站貼上資料，連 Hero Image 說明都帶進來 → 模型讀了無用資訊
  • 說「你是 Alex，一名客服人員」→ 欺騙模型，可能引發不可預期行為

解法：用 XML 標籤讓結構一目了然。

四大常見反模式

反模式 1

假裝模型是人類

「你是 Alex，一位真人客服員工。」— 這不是事實，且會讓模型在邊界情境行為不一致。

反模式 2

複製貼上垃圾內容

從網站貼上政策文件時，Hero Image 說明、Cookie 警告也一起進了 Prompt。模型要處理無用資訊。

反模式 3

一巨段塞所有規則

政策、語氣、流程、歷史補丁全塞在一個大段落。沒有結構，模型難以區分規則的重要性。

反模式 4

過期的防禦性補丁

「絕不要給客戶錯誤資訊，請指向 URL。」這條舊規則在新一代模型上被過度遵循，反而隱瞞正確資訊。

解法：XML 標籤結構化

"If you're reading a prompt and you can't tell guidelines from policy from data, most likely the model isn't able to either." 「如果你讀一份 Prompt 時分不清哪段是指引、哪段是政策、哪段是資料——模型很可能也分不清。」

— Margot Vanlar
給初學者 · 什麼是 XML 標籤？
XML 標籤如何幫助模型？

XML 標籤就像幫文件加上標題牌。<role>...</role> 告訴模型這段是關於它的身份；<policy>...</policy> 是規則。

  • 之前：「你是一個客服助手。以下是一些規定...（一大段）」
  • 之後：<role>客服助手</role> + <policy>...</policy> + <tone>...</tone>

結果：模型對各類規則的理解與遵循度顯著提升，尤其在複雜案例上。

輸出契約（Output Contract）

除了整理 Prompt 內容，還需要確保輸出格式的一致性。Margot 展示了三個層次的輸出控制：

I

Prompt 內定義輸出格式

在 Prompt 中加入 <output_format> 區塊，告訴模型用 XML 標籤包住回應。基礎層次，但不是最可靠的。

II

API 層 Stop Sequences

在 API 呼叫中設定 stop_sequences，偵測到 XML 閉合標籤時就停止生成。比口頭規範更可靠。

III

Structured Outputs（最強）

使用 Structured Outputs API，以程式化方式強制模型輸出符合 JSON Schema 的格式。複雜巢狀 JSON 場景的最佳選擇。

給初學者 · Stop Sequence 是什麼？
為什麼不能只靠「請輸出 JSON」？

就像請人寫信「請在最後一段結束」——他可能多寫了幾行。Stop Sequence 是給模型的「停筆訊號」：看到這個字符，就立刻停。

  • 如果模型輸出是 <response>...</response>，設定 stop_sequences = ["</response>"]
  • 模型一寫完閉合標籤，API 就截斷輸出
  • 對複雜 JSON Schema → 直接用 Structured Outputs，不要靠 stop_sequences

越往 API 層控制，越可靠；越靠 Prompt 口頭叮嚀，越脆弱。

這個故事的啟示 · Prompt 清理立竿見影

只是「整理」，就讓測試通過率立刻上升

Margot 在現場展示：取出一個已有測試失敗的 Prompt，僅做三件事——1. 移除「你是真人」的說法，2. 刪除從網站複製進來的垃圾資料，3. 加上 XML 結構標籤——重新跑 Eval 後，某個邊緣案例立刻從失敗變為通過。

"Simply clearing up the prompt with a better structure, with a better role description, has improved the performance."

沒有改任何「規則」，只是改了「組織方式」。這說明一件事：很多時候模型表現差，不是因為它不夠強，而是因為 Prompt 讓它沒辦法發揮。

防禦性補丁的版本控制

給初學者 · 什麼是防禦性補丁（Defensive Patch）？
為什麼舊規則會「反咬」你？

過去模型常常會犯特定錯誤，工程師就在 Prompt 裡加了一條「絕對不要做 X」。問題是，新一代模型指令遵循能力更強——它會「更認真地」不做 X，即使 X 其實是正確答案。

  • 舊場景：模型 A 經常給出錯誤合約資訊 → 加補丁「有疑問就給 URL」
  • 新場景：模型 B 指令遵循更強 → 即使知道正確答案，也一律給 URL
  • 解法：用版本控制記錄「這條補丁是為了解決 X 問題在 Y 日期加入的」

這樣你才能在模型升級後，有根據地決定哪些補丁可以移除。

I

失效模式一：資訊隱瞞

模型不只會幻覺——它也會隱瞞它本來就擁有的正確資訊。Hotspot 案例揭示：過度優化的防禦性補丁，會讓新模型選擇沉默而非誠實。

關鍵故事 · Hotspot 的消失

模型知道答案，卻選擇不告訴你

客戶問：「我的無限方案有多少 Hotspot 流量？」客戶帳號資料中清楚寫著 5 GB。但模型的回答是：「無限方案一般包含 4 GB，但您的是舊方案，詳情請查看 URL。」

"The customer test case that we're dealing with is on a legacy plan. The customer data which we're feeding to the prompt includes the amount of hotspot data — they have 5 gigabytes. But the model is telling the customer to go check this out themselves."

為什麼？因為 Prompt 裡有一條舊補丁：「切勿給客戶錯誤的方案細節，有疑問請提供 URL。」這條規則在舊模型時代是防止幻覺的，但新模型指令遵循更精確——它把「有疑問」解讀成「只要是舊方案客戶，就是有疑問情境」。

教訓：幻覺（編造資料）和隱瞞（不說已知答案）是同樣危險的兩個極端，而過度防禦會把模型推向後者。

解法：更新資訊來源的優先級

修改：明確告訴模型「客戶帳號資料是最可靠的資訊來源，遇到舊方案客戶，直接使用帳號資料回答，而非轉向一般政策文件。」

給初學者 · 資訊隱瞞 vs 幻覺
兩種模型可能犯的相反錯誤

大多數人擔心模型「亂說」（幻覺），但鮮少有人注意它「不說」（隱瞞）。這兩種問題都源於 Prompt 設計。

  • 幻覺：模型沒有資訊，卻編造一個 → 過去常見，現在模型變強後少見
  • 隱瞞：模型有正確資訊，但因為防禦規則選擇沉默 → 隨新模型指令遵循能力增強而增加

解法：在 Prompt 中明確定義資訊優先級，而非一味「有疑問就不答」。

II

失效模式二：指令 ≠ 能力

告訴模型「這非常重要，請精確計算」不會讓它更擅長心算。能力邊界只能透過工具（Tool）來擴展，而不是語氣強度。

"The key lesson to take away here is instructions don't add capability. Telling the model it's critical to do a calculation right doesn't make it better at mental math. So the correct approach was to give it a tool overall." 「核心教訓是：指令無法增加能力。告訴模型這個計算非常重要，並不會讓它更擅長心算。正確的做法是給它一個工具。」

— Margot Vanlar
關鍵故事 · 按比例計費的失敗

說「很重要」不能讓 AI 會算數學

客戶問：「如果我月中升級到 30 GB 方案，下個月帳單會是多少？」這需要精確的按比例計費（Proration）計算。

"In the prompt it says: 'critical: always calculate any pro-rated amounts correctly.' Telling the model to do a good job isn't particularly helpful when we don't give the model the capability to actually do a good job."

模型的回應？看起來在思考，做了點「心算」，但給出的是模糊的估算而非精確數字。原因很簡單：LLM 的訓練目標不是精確算術，它最擅長的是語言推理。

解法：在 API 中定義一個 calculate_proration 工具，告訴模型「需要計算時，呼叫這個工具」。結果：所有測試案例立刻通過，計算精確無誤。

給初學者 · 什麼是 Tool Use？
為什麼 LLM 需要「工具」？

把 LLM 想成一個超級聰明但沒有計算機的人。它能解釋題目、判斷邏輯、提取參數——但精確計算不是它的強項。給它一個計算機（Tool），它就能做精確計算。

  • 模型的強項：語言理解、邏輯推理、參數提取、決策判斷
  • 工具的強項：精確計算、資料庫查詢、即時資訊獲取、外部 API 呼叫
  • 正確分工：模型決定「何時需要計算」，工具「執行計算」，模型「解釋結果」

Rule of thumb：如果你在 Prompt 裡寫「請精確計算 X」，那就應該給它一個 Tool 來做 X。

工具定義三步驟

I

Prompt 中說明工具用途

「需要計算按比例費用時，使用 calculate_proration 工具。」告訴模型工具存在及使用時機。

II

API 中定義工具 Schema

告訴 API：「這個工具叫什麼、接受什麼參數（升級日期、當前計費周期、舊方案費用、新方案費用）。」

III

後端實作工具邏輯

寫真正的數學計算程式。模型負責提取正確參數，工具負責精確計算，兩者各司其職。

III

失效模式三：片面故事

當你只告訴模型「避免 X 的代價」，它會全力避免 X——即使不避免才是更好的選擇。越智能的模型，越需要完整的兩面資訊。

關鍵故事 · 升級真人的代價

模型知道規則，但不知道規則背後的故事

帳務爭議情境：客戶有一筆不明費用，理應升級給真人客服處理。但模型反而試圖自行診斷問題。

"The initial instructions said: 'Avoid escalating or transferring to a care specialist unless absolutely necessary, as it costs approximately $8 and it counts against our team's fast contract resolution.' This is only giving one side of the story, right? We're telling it what the cost is to escalating, but not the benefit."

模型的「邏輯」完全正確——它確實在最小化升級成本。但 Prompt 忘了告訴它：如果複雜帳務問題沒解決，客戶可能離開，損失遠大於 $8 的升級成本。

修改後的 Prompt：「升級給真人花費 $8。但帳務問題若未解決，客戶可能要求退款並流失，損失遠高於此。請在兩者之間做出最佳判斷。」結果：模型在帳務爭議情境下正確升級。

"Specifically as models become more intelligent, we need to remember to state both sides of the trade-offs because our models are becoming better themselves at making those trade-offs themselves." 「隨著模型越來越聰明，我們需要記得說出權衡的兩面——因為我們的模型正在變得更擅長自己做出這些權衡。」

— Margot Vanlar
給初學者 · 什麼是目標過度優化？
為什麼聰明的模型更危險？

想像你告訴員工「盡量不要讓客戶等待」——一個普通員工可能做到 80% 的程度。但一個超級聰明的員工可能為了不讓人等待，做出了完全錯誤的決定（例如：直接拒絕複雜申請）。

  • 舊模型：指令遵循能力弱，過度優化風險低
  • 新模型：指令遵循能力強，給了單一目標就會不計代價達成
  • 解法：提供完整的 Trade-off 資訊，讓模型自己做出平衡決策

黃金法則：你在 Prompt 裡提供的資訊，決定了模型做決策時的全部依據。

I

從零打造排班 Agent

面對複雜的零售排班問題——8 名員工、硬性限制、一週排程輸出——Margot 系統性地比較了四種方法，最終找到了在成本、延遲、彈性三方面最優的架構。

給初學者 · 為什麼排班是 Hard Problem？
排班問題比看起來複雜得多

零售排班聽起來很日常，但對 AI 是個組合爆炸問題：8 個員工 × 7 天 × 多個班次，還有「每人每週不超過 X 小時」「至少 Y 人值大夜班」等硬性規則。

  • 長輸出問題：完整一週排程輸出很長，容易超出 max tokens
  • 多規則驗證：需要同時確保所有約束都被滿足，難以口頭叮囑
  • 軟性約束：「盡量不排某兩人同班」這種非硬性規則難以程式化

這是測試「一個 Prompt vs 多個 Prompt」最好的場景。

E · GER 迴圈（Sonnet 4.6）
全通過 · 最低成本
給初學者 · Adaptive Thinking 是什麼？
讓模型自己決定「想多少」

一般模型是：你問問題→模型直接回答。Adaptive Thinking 是：你問問題→模型先「想一想」→再回答。就像讓聰明的朋友先在腦子裡推演，再告訴你答案。

  • 好處：難題（如複雜排班）的解題正確率大幅提升
  • 代價：思考過程消耗大量 Token，延遲 3 倍起跳
  • 適用場景：一次性的複雜推理任務，不適合高頻呼叫

Margot 的結論：Adaptive Thinking 有效，但代價高。GER 迴圈更適合生產環境。

II

Generate-Evaluate-Repair 迴圈

三個簡單的 Prompt，各司其職，互相協作。GER 迴圈不只比單一超大 Prompt 更有效，還能在執行時動態注入軟性約束，展現出架構上的彈性。

1
Generate

生成草稿

Simple Prompt 讓模型生成第一版排班表。不強求完美，允許有違規——有後續環節處理。

→
2
Evaluate

LLM 評估器

第二個 Prompt：檢驗每條規則是否違規，提供具體證據（「王小明在週三已排班 8 小時，再排違反上限」）。不靠程式碼，靠語言推理。

→
3
Repair

修復補丁

第三個 Prompt：收到違規清單，針對性地微調排班表，而非重新生成全部。精準、高效。

"We have three very simple prompts but they're now running independently rather than trying to do everything in one large prompt." 「我們有三個非常簡單的 Prompt，各自獨立運行，而不是試圖用一個巨大的 Prompt 做所有事情。」

— Margot Vanlar

GER 迴圈的隱藏優勢：軟性約束

給初學者 · 硬性約束 vs 軟性約束
什麼是「盡量」？什麼是「必須」？

排班有兩種規則：必須滿足的（硬性：每人每週不超過 40 小時）和盡量滿足的（軟性：兩位員工最好不要同班）。

  • 硬性約束：用 Python 程式碼驗證，違反就視為失敗
  • 軟性約束：無法程式化，需要語言理解（「John 和 Mary 最近有衝突，盡量分開」）
  • GER 的優勢：軟性約束可以直接寫在評估 Prompt 裡，每次執行時動態調整

這意味著：不需要改程式碼，只需要改 Prompt，業務需求的調整就能立刻生效。

GER 迴圈中，軟性約束可以在 Runtime 直接注入到 Evaluate Prompt：

傳統方法

改後端程式碼

「John 和 Mary 最好不要同班」→ 寫 Python 邏輯→ 部署→ 測試→ 上線。幾天的工作量。

GER 方法

Runtime 注入

在 Evaluate Prompt 裡加一行：「As a soft constraint: try to separate John and Mary.」立即生效。

兩條可行的生產路線

路線 A
Opus 4.7 + Adaptive Thinking：最簡單的架構，一個 Prompt 解決。代價：成本和延遲 3 倍，不適合高頻呼叫。適用：低頻、高價值的關鍵任務。
路線 B
GER 迴圈 + Sonnet 4.6：三個簡單 Prompt，成本低、延遲低、支援軟性約束動態調整。適用：生產環境、需要定期執行、有業務規則靈活性需求。
關鍵故事 · 架構選擇的本質

不是要更大的模型，而是更好的架構

Margot 的排班實驗展示了一個普遍規律：當一個單一 Prompt + 大模型解不了問題時，工程師的第一反應往往是「換更大的模型」——但這不總是最好的答案。

"Rather than using one prompt to address everything, we're actually isolating different tasks where it's easy and repeatable to separate out the steps."

GER 迴圈的每個節點都非常簡單：Generator 不需要驗證，Evaluator 不需要生成，Repairer 不需要重新推理全部。這種「分工」讓每個 Prompt 都能用更小的模型完成，總成本反而更低。這是系統設計思維在 AI 應用中的具體體現。

給初學者 · 多 Prompt 架構是「Agentic」嗎？
Agent 不一定要做很多事——它只需要「協調多個步驟」

很多人以為「Agent」是一個能自主做很多事的大系統。但 GER 迴圈展示的是：Agentic 架構的核心是把複雜任務分解成多個簡單步驟，讓每個步驟都能做好自己的事。

  • Single Prompt Agent：一個人做所有事，容易超載
  • Multi-Prompt Agent：流水線，每個 Prompt 是一個專責環節
  • 關鍵洞見：分工後，每個節點都能用更小、更便宜的模型完成

這是 CS 153 Week 1 講的「Orchestrator / Subagent」模式在真實場景的落地。

V

Guided Discussion

六道思考題，搭配每題的建議思考方向——幫助你把本講知識轉化為實際判斷力。

Q1

Margot 強調「先建評估套件，再動 Prompt」。但在實際工作中，許多團隊沒有 Eval Suite。你認為，為什麼 Eval Suite 在 AI 工程中比在傳統軟體工程中更重要？你的組織或個人專案應該如何開始建立第一個 Eval？

查看建議思考方向
  • 歷史對照：傳統軟體的行為是確定性的（輸入 → 輸出固定），AI 模型是機率性的。沒有 Eval，你無法區分「本次成功」和「每次都成功」。
  • 講者隱含論點：Margot 說「我們很少從頭寫 Prompt，更常是 debug 現有 Prompt」——Eval Suite 是 debug 的前提，沒有它你只能靠直覺。
  • 職涯啟發：能設計好 Eval Suite 的工程師，是 AI 時代真正稀缺的技能。設計 Eval 本身就是對「什麼叫成功」的清晰定義。
  • 反向思考：「我們的任務太複雜，難以量化評估」——Margot 用 Python 函數驗證排班規則，反駁了這個說法。複雜任務往往可以分解成可量化的規則。
Q2

「版本控制防禦性補丁」是 Margot 提出的一個有趣建議：記錄每條防禦規則的原因和添加時間。在你看來，這個做法在工程文化上面臨的最大阻力是什麼？如何在一個快速迭代的團隊中落地這個實踐？

查看建議思考方向
  • 歷史對照：這和技術債（Technical Debt）的管理類似——當下的「快速修復」在未來成為維護負擔。差別在於 AI Prompt 的後果更隱蔽。
  • 講者隱含論點：Prompt 的版本控制不只是「記錄」，是在建立可以安全刪除補丁的條件。沒有記錄，你不敢刪；有了記錄，你知道何時可以刪。
  • 職涯啟發：把 Prompt 當作「程式碼」管理（git commit、PR review），而不是隨手修改的文件。這個思維轉換本身就是 AI 工程師的成熟標誌。
  • 反向思考：有沒有補丁是「永遠需要的」？那些應該被提升為正式的 Policy，而不只是 Patch。
Q3

「指令無法增加能力」是本演講最有力的洞見之一。請思考：除了「精確計算」，還有哪些常見任務是 LLM 在沒有工具的情況下本質上做不好的？你的專案中有哪些地方可能需要引入工具而不是更多提示詞？

查看建議思考方向
  • 歷史對照：ChatGPT 早期版本在算術上的失敗案例廣為人知。後來的 Code Interpreter（Python 執行）和 Web Browsing 都是「給工具」的解法。
  • 講者隱含論點：LLM 的能力邊界是確定的，Prompt 無法突破——只有架構（Tool、RAG、Retrieval）才能擴展。
  • 職涯啟發：辨識「哪些任務需要工具」是 AI 工程師的核心判斷力。類比：知道什麼情況用 for loop、什麼情況用資料庫的判斷力。
  • 反向思考：給工具也有代價（延遲、成本、複雜性）。什麼情況下可以接受模型做「夠好但不精確」的計算？
Q4

Margot 指出「模型越聰明，越需要完整的 Trade-off 資訊」。這個觀察暗示了什麼？隨著模型能力持續提升，Prompt 的設計哲學將如何演變？你認為 2 年後的 Prompt 工程會和今天有多大不同？

查看建議思考方向
  • 歷史對照：早期 ChatGPT（GPT-3.5）時代的 Prompt 技巧（few-shot、chain-of-thought）到 Claude 3.5 時代已基本過時。模型進化讓 Prompt 技巧的半衰期很短。
  • 講者隱含論點：未來的 Prompt 可能更像「目標設定」而非「步驟指令」——你告訴模型想要什麼、為什麼，讓模型自己找方法。
  • 職涯啟發：與其學記哪些 Prompt 技巧有效，不如理解模型的推理原理。後者的半衰期更長。
  • 反向思考：有些 Prompt 原則（XML 結構、Eval Suite）可能永遠有效，因為它們是「溝通清晰度」的問題，而非模型能力的問題。
Q5

Generate-Evaluate-Repair 迴圈的核心架構是「把複雜任務分解成多個簡單任務」。這和傳統軟體工程的哪些設計原則有相似之處？你認為 GER 迴圈的主要風險是什麼？什麼情況下反而不適合用 GER？

查看建議思考方向
  • 歷史對照：類似 Unix 哲學「一個程式只做一件事，多個程式組合解決複雜問題」，也類似微服務架構的解耦思想。
  • 講者隱含論點：GER 的每個節點都「更容易測試和除錯」——這是分工的隱性好處，不只是性能優化。
  • 職涯啟發：學會識別「這個任務可以用 GER 解構」是 AI 系統設計的核心能力。這需要你先能清晰定義「什麼叫成功輸出」。
  • 反向思考：GER 增加了系統複雜度（三個 API 呼叫、錯誤處理、狀態傳遞）。對於簡單任務，一個 Prompt 就夠了。過度拆分也是一種浪費。
Q6

Margot 展示了「軟性約束可以在 Runtime 動態注入到 Evaluate Prompt」。這個能力對商業應用有什麼深遠的影響？你能想到三個真實場景，在那裡「Runtime 軟性約束」會比「硬編碼規則」帶來顯著的商業價值？

查看建議思考方向
  • 歷史對照：傳統業務規則引擎（如 Drools、Camunda）需要 IT 部門修改和部署，週期長。GER 讓業務人員可以用自然語言即時調整規則。
  • 講者隱含論點：軟性約束是「人類判斷的延伸」——那些難以程式化的偏好、情境判斷、臨時需求。這正是 LLM 最擅長的領域。
  • 職涯啟發：能設計「可接受軟性約束的 Agent 架構」，讓業務人員自行調整，是一個極具商業價值的技能。
  • 反向思考：軟性約束也帶來風險——如果業務人員注入了互相矛盾的約束怎麼辦？系統應該如何處理衝突？

名詞速查表

GER
Generate-Evaluate-Repair：三節點代理迴圈。生成草稿 → LLM 評估違規 → 修復補丁。比單一超大 Prompt 更高效。
Eval
Evaluation Suite：評估套件。包含控制案例、邊緣案例、邊界案例，用於量化 Prompt 改動的效果。
Stop Seq
Stop Sequences：API 層的停止信號。偵測到特定字符後截斷輸出，確保格式一致性。
Proration
按比例計費：月中升級方案時，依剩餘天數計算費用。需要精確計算，必須用 Tool 而非口頭指令。
Adaptive Thinking
讓模型自決「需要思考多少」的模式。提升複雜推理正確率，但消耗 3× Token 和延遲。
Soft Constraint
軟性約束：「盡量」而非「必須」的規則。GER 迴圈允許在 Runtime 動態注入，無需改程式碼。
