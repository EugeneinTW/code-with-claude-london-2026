I

Claude Code 讓 CTO 重拾程式碼

Omni CTO 演講的開場，不是產品 roadmap，而是一張 commit 趨勢圖。2025 年初到 2026 年初，那條曲線從平緩變成垂直——而這一切，都從 Opus 模型的 Claude Code 發佈那天開始。

這不只是一個生產力故事。這是一個文化轉型的故事。Omni 的 25 人工程團隊從「讓我們試試看」的實驗心態，到 2026 年初全面「升級技能、找到門路」。那條曲線上有一段極小的貢獻是 CTO 自己的——一個管理數百家客戶、快速成長公司的 CTO，依然能持續提交高品質程式碼。

"I thank Claude very much for making me still able to do some software engineering from time to time. That's been a really fantastic sort of unexpected benefit." 「非常感謝 Claude，讓我偶爾還能做一些軟體工程工作。這真的是一個令人驚喜的意外收穫。」

— Omni CTO
給初學者 · 一句話搞懂「CTO 為什麼通常不再寫程式碼」
技術管理者的「彼得原理」陷阱

在大多數科技公司，工程師晉升為技術主管、再晉升為 CTO 後，就幾乎不再寫程式碼了——因為管理工作（1:1 會議、路線圖規劃、客戶溝通）佔滿了所有時間。這被稱為「技術負債之外的管理負債」。

  • 問題：離開技術前線太久的 CTO，往往失去了對系統複雜度的直覺感知
  • Claude Code 的效應：把「寫程式碼」的時間成本降低到「偶爾 30 分鐘」也能有實質貢獻
  • 更深的意義：繼續動手寫程式的 CTO，對「工程師的痛點」有更真實的體感

Omni CTO 的故事暗示：AI 工具不只是提升工程師的生產力，也在重塑技術管理者的工作方式。

「Ship It」文化：速度即競爭力

Omni 的文化有兩個核心價值觀：Ship it（推出產品）和透明度。每週五全員大會，前 10 分鐘公告，後 50 分鐘全部用於產品 Demo。週六早上，CEO Colin 親自剪輯 Demo 影片並上傳 YouTube。

文化設計的啟示

CEO 週六上午的例行工作

大多數公司的透明度是聲明式的——我們承諾開放。Omni 的透明度是行為式的：CEO 每週六早上親自剪輯 Demo 影片上傳，讓客戶、潛在客戶和整個技術社群都能即時看到進展。

"Our CEO's favorite thing to do is wake up Saturday morning, cut these demos, and upload to YouTube to share with the world."

這個設計有兩層效果：第一，它迫使工程師每週必須有「可展示的進展」（而不只是「在做」的進展）；第二，它建立了公司在技術社群中的品牌——你不是在說「我們很快」，而是每週讓全世界看到「我們有多快」。Claude Code 讓這種節奏變得可持續。

六大核心洞見

I

開發者即使用者，共鳴才有深度

Omni 團隊既是 Claude Code 的重度用戶，又是 Blobby 的建造者。這種「身份重疊」讓他們對好的 Agent 框架應有的樣子，有著難以複製的直覺。

II

上下文越局部越有效

就像 .claude.md 放在程式碼旁邊效果更好，語義層的定義也應該「局部化」到它所指的欄位旁邊，而非放在遠端文件中。這個洞察直接來自 Claude Code 的使用經驗。

III

錯誤預算是 Agent 品質的最佳槓桿

給 Blobby「錯誤恢復預算」和高品質錯誤訊息，是成本最低、收益最高的品質提升手段——評估分數大幅上升，比換模型的效果更快更明顯。

IV

分裂大腦是 Agent 架構的隱形陷阱

外部規劃 Agent 和內部執行 Sub-agent 之間，一旦出現「知識不對稱」——一個不知道另一個能做什麼——系統就會進入死循環。這是 Multi-agent 設計的核心反模式。

V

讓 Claude 做它最擅長的事

SQL Parser 的復活，本質上是「停止限制 Claude，讓它用熟悉的語言（CTE SQL）表達」。這個決定需要對 Claude 的能力有信心——而 Omni 團隊押注 Anthropic 會持續強化 SQL 能力。

VI

Traces 可視化是 Agent 調試的核心工具

在發現「分裂大腦」問題之前，Omni 的做法是深入研究 Traces——Agent 的內部對話記錄。可觀測性（Observability）是 Agent 系統品質的前提條件，不是事後補救措施。

II

語義層：讓 LLM 理解你的業務

Claude 能告訴你「企業一般如何運作」，但它不知道你的公司如何定義「上個季度」、你的資料庫中哪一張「收入表」才是唯一可信源。這就是語義層存在的原因。

"'Last quarter' in our product and engineering team refers to the calendar year, but in our sales team, it refers to our fiscal quarter. All of these things need to be encoded." 「『上個季度』在我們的產品和工程團隊中指日曆年；但在銷售團隊中，指的是財政季度。所有這些都需要被編碼進去。」

— Omni CTO
給初學者 · 一句話搞懂「語義層」
語義層是 AI 與企業數據之間的「翻譯官」

想像你是一個剛入職的新員工（這就是 LLM 的處境）。你懂商業邏輯，但你不知道：公司哪張表才是「官方」的收入數據、「季度」是日曆年還是財政年、「用戶」在不同部門的定義是否一致。

  • 沒有語義層：LLM 在 1000 張相似的表中猜測，結果不穩定
  • 有語義層：語義層說「這張表是唯一可信源，那 99 張可以忽略」
  • 核心類比：程式碼中的 .claude.md 文件 ≈ 語義層中的欄位定義

語義層不改變 AI 的能力，它改變 AI 的「背景知識」，讓 AI 的通用智慧能正確應用到特定業務。

「數值局部化」的天才設計

讓 Claude「嚐一口」數據的滋味

在第一代 Blobby 中，Omni 發現了一個微妙但關鍵的洞察：當用戶問到「Region（區域）」這個欄位時，如果直接給 LLM 一些代表性數值（如 EMEA, NA, APAC），它的準確性會大幅提升。

"If you can give the LLM a little taste of what the values in that field are, it's very helpful. It can infer what the next 10 values are, because it sees 'these are abbreviations for world regions'."

這個設計解決了兩個問題：拼字錯誤的模糊匹配（用戶輸入 "United States" 可以正確匹配到 "US"），以及 LLM 對「這個欄位長什麼樣」的幻覺（不必猜測，看到示例就知道）。

這個洞察直接對應到 Claude Code 中的局部化 .claude.md 原則：上下文越靠近它所描述的事物，效果越好。

給初學者 · 一句話搞懂「模糊匹配」
為什麼 AI 需要「容忍錯字」？

現實中的用戶不會每次都打對字。「New York」可能被打成「Nw York」，「Q3」可能被理解成「第三季度」或「Q3 2025」。

  • 精確匹配：字不對就找不到，系統回報錯誤
  • 模糊匹配：計算字串相似度，「Nw York」≈ 「New York」
  • Blobby 的做法：結合 LLM 語義理解 + 代表性數值，讓 Claude 能推斷「這個用戶想問的是 NA（North America）」

好的 NLP 系統設計，總是假設用戶會犯錯，然後優雅地處理這些錯誤。

III

Blobby 的四個演進階段

Blobby 的今天不是設計出來的——它是在真實失敗中迭代出來的。18 個月，四個主要階段，每一次演進都解決了當時 AI Agent 設計的一個核心難題。

1
單問單答（Single Q&A）
挑戰：LLM 無法理解企業私有數據結構和術語，拼字錯誤導致查詢失敗
解法：引入語義層 AI 上下文、示例查詢、局部化數值（EMEA/NA/APAC）、模糊匹配。每次詢問都能得到較準確的單次回答。
2
Agentic Loop + 錯誤預算
挑戰：單次問答無法應對需要多步驟的複雜問題；錯誤發生時直接崩潰
解法：建立專屬 Agent 框架 + 任務清單；引入錯誤恢復預算和高描述性錯誤訊息；將底層模型從 Haiku 升級為 Sonnet。評估分數急劇上升，使用量開始指數成長。
3
Blobotomy：合併大腦手術
挑戰：外部規劃 Agent 和 SQL 生成 Sub-agent 之間「分裂大腦」，導致死循環
解法：Joel 將 SQL 生成工具「上拉」至外部 Agent 框架，消除子系統間的知識不對稱。不可預測的奇怪行為大幅減少。
4
SQL Parser 復活 + CTE 一擊必殺
挑戰：自製 JSON 查詢格式限制太多，無法發揮 Claude 的 SQL 天賦
解法：Steven 重啟塵封多年的 SQL 解析器，允許 Claude 直接生成 CTE SQL。複雜查詢從 3-4 次嘗試壓縮至一次成功。
給初學者 · 一句話搞懂「Agentic Loop（Agent 循環）」
Agent 循環讓 AI 從「答一題」變成「解一個問題」

想像你請一個顧問幫你分析財報。「單次問答模式」是：你問一個問題，顧問答一個答案，結束。「Agent 循環模式」是：顧問拿到問題後，自己列出調查計劃、一步步執行，遇到錯誤自己修正，最後給你完整報告。

  • Tasks（任務清單）：Agent 的工作計劃，分解複雜問題
  • 錯誤恢復預算：允許 Agent 嘗試幾次失敗後自我修正，而非直接崩潰
  • 高品質錯誤訊息：告訴 Agent「失敗原因」和「如何修復」，而非只說「failed」

Blobby 加上 Agent 循環後，突然能處理「我剛問了這個問題，以前要花好幾個小時手動分析，現在它兩分鐘搞定了」這類問題。

Haiku 到 Sonnet 的轉折

Token 花更多，但客戶願意買單

升級到 Sonnet 是一個有趣的商業決策：更長的對話、更多的 Token 消耗，也就是更高的成本。但 Omni 做了這個決定，理由很簡單：它解鎖了客戶原本無法解決的問題。

"We suddenly started hearing customers say 'wow, I just asked this question, this is either something I could never answer myself, or it would have taken me hours, and it got it in two minutes.'"

這個故事印證了 Talk 17 的核心原則：「最便宜的不是單 Token 成本最低的，而是每次成功結果成本最低的。」用 Sonnet 花更多 Token，但把「需要人類花數小時」的問題壓縮到兩分鐘，這個 ROI 毋庸置疑。

給初學者 · 一句話搞懂「錯誤恢復預算」
為什麼要給 Agent「失敗幾次的權利」？

人類做事情也會失敗。好的工程師不是第一次就寫出完美程式碼的人，而是能快速找到錯誤並修復的人。Agent 也一樣。

  • 沒有錯誤預算：第一次 SQL 查詢失敗 → 系統報錯 → 用戶看到錯誤頁面
  • 有錯誤預算：第一次 SQL 查詢失敗 → Agent 分析錯誤訊息 → 修改 SQL → 重試 → 成功
  • 關鍵設計：錯誤訊息必須「有用」——告訴 Agent 「SYNTAX ERROR at line 12: missing comma」遠比「ERROR: query failed」有價值

Omni 發現，單單改善錯誤訊息的品質，就讓困難評估的通過率大幅上升。這是「最便宜的品質提升」。

IV

Blobotomy：腦葉切除術

CEO Colin 是 Blobby 最嚴苛的測試者。他不接受「LLM 本身有不可預測性」的說法。正是這個拒絕妥協的態度，逼出了 Omni 最重要的架構發現：分裂大腦問題。

"You want to be careful not to have a split brain between any sort of sub-agent system and outer agent system... Joel referred to this as consolidating the brain." 「在任何子 Agent 系統和外部 Agent 系統之間，你必須小心不要產生分裂大腦⋯⋯ Joel 把這稱為合併大腦。」

— Omni CTO

分裂大腦 vs 合併大腦

❌ 分裂大腦（問題架構）

Blobotomy 之前

外部 Agent：負責規劃任務清單，了解所有可用數據
↓ 委派任務（但不知道子 Agent 的能力邊界）
子 Agent：只負責「生成一個 SQL 查詢」
↓ 問題爆發
外部 Agent：「分析 GitHub PR + Support 數據的趨勢」
子 Agent：「我無法在單個查詢中回答這個問題」→ 死循環
✓ 合併大腦（解法）

Blobotomy 之後

外部 Agent：負責規劃，且直接擁有所有查詢工具
↓ 外部 Agent 自己執行查詢工具
工具「上拉」：SQL 生成工具直接整合進外部 Agent
↓ 結果
外部 Agent 知道「什麼能在單個查詢中完成」
不可預測行為大幅減少，複雜評估顯著改善
給初學者 · 一句話搞懂「分裂大腦問題」
為什麼「分工」有時反而更糟？

想像一個大廚（外部 Agent）和一個助手（子 Agent）合作做料理。大廚說：「你去幫我同時炒魚和燉牛肉」。但助手只有一口鍋，只能做一件事。問題在於，大廚不知道助手的限制，所以一直下達超出能力的任務。

  • 症狀：系統看起來很聰明地規劃了任務，但執行時不斷失敗
  • 根本原因：上層 Agent 缺乏對下層 Agent 能力邊界的認知
  • 解法：把工具直接給大廚，讓他自己知道能做什麼

「合併大腦」的代價是放棄了「模組化」和「工具複用」——那個查詢生成子 Agent 原本可以在其他場景複用。但換來的是系統行為的可預測性，這個交換值得。

Traces 可視化的關鍵作用

看見 Agent 的內心世界

發現分裂大腦問題的不是直覺，是 Traces 可視化。當 Omni 團隊開始認真研究 Agent 的內部對話紀錄時，他們看到了一個隱藏的模式：外部 Agent 不斷下達子 Agent 無法執行的任務，子 Agent 不斷回報失敗，形成無法退出的循環。

"When you look at traces, you can see the inner workings of how the agent is conversing with itself and reacting and responding in these loops. This really clarified why some seemingly oddly-orchestrated conversations could be traced back to real root problems."

這個故事的教訓是：Agent 系統的 Debugging 和傳統系統完全不同。你需要能「觀察 AI 在想什麼」，而不只是觀察它做了什麼。可觀測性工具是 Agent 工程的前提條件。

IV

SQL Parser 復活：讓 Claude 用最愛的語言說話

Omni 曾經建構了一個完整的 SQL 解析引擎，然後因為「不夠可靠」把它束之高閣多年。當 Claude 出現後，工程師 Steven 決定把它從架子上取下來——因為一切條件都變了。

舊系統：JSON 化查詢格式

為了讓 LLM 能生成語義查詢，Omni 設計了一套高度結構化的 JSON 格式。問題是：這種格式受限極多，無法表達複雜的多步驟查詢邏輯，LLM 需要反覆嘗試。

新系統：讓 Claude 寫 SQL

SQL Parser 復活後，Claude 可以直接生成它最擅長的 CTE SQL。解析器負責把這個 SQL 翻譯成 Omni 的語義查詢格式，再丟進數據倉儲執行。

"Claude seems to really love writing SQL with CTEs... And our parser is really good at parsing those. This turned into a really beautiful combination." 「Claude 似乎非常喜歡用 CTE 寫 SQL⋯⋯而我們的解析器非常擅長解析這些。這演變成了一個非常美妙的組合。」

— Omni CTO
給初學者 · 一句話搞懂「CTE（公用表表達式）」
CTE 是 SQL 的「函式」

普通 SQL 把所有邏輯擠在一個查詢裡，很難讀也很難調試。CTE（Common Table Expressions）讓你把複雜查詢拆分成有名字的小步驟。

  • WITH monthly_revenue AS (SELECT ...),
  • top_customers AS (SELECT ... FROM monthly_revenue ...)
  • SELECT * FROM top_customers WHERE ...

就像寫程式碼時把邏輯拆成有名字的函式一樣，CTE 讓複雜的多步驟數據分析變成可讀、可理解、可調試的結構。Claude 特別擅長用 CTE 思考，因為它的訓練數據中包含大量高品質的 CTE 範例。

節省

這個案例背後有一個更深的設計原則：讓 Claude 做它最擅長的事，而不是讓它學你的語言。 Omni 最初設計 JSON 格式是為了「控制 LLM 的輸出」，這種控制是有代價的——它剪除了 Claude 最強大的 SQL 生成能力。SQL Parser 的復活，是從「控制 AI」到「信任 AI」的哲學轉變。

給初學者 · 一句話搞懂「AI 建設 vs UI 驗證」
Omni 的產品哲學：角色分工

Omni 的 Demo 展示了一個很清晰的產品哲學：AI 負責生成，人類負責驗證和精煉。

  • AI 做：根據問題生成 SQL、建立儀表板結構、提供視覺化方案
  • UI 做：讓人類可以「打開工作簿」，直接看到 SQL 和篩選條件，一鍵修改
  • 原則：AI 不是要取代人的判斷力，而是把「從零到 80%」的工作做掉，讓人專注在「80% 到 100%」的精煉

這個設計讓 AI 的「不完美」變得可接受——用戶知道他們可以輕鬆驗證和修改結果，不需要盲目信任 AI 的輸出。

V

深度思考題

從 Blobby 的演進故事出發，這六個問題探索 Agent 架構設計、組織文化，以及 AI 輔助分析的深層邏輯。

Q1：Omni 的 commit 曲線在採用 Claude Code 後呈指數成長。但 commit 數量真的等於開發品質嗎？有哪些情況下更多的 commit 可能是一個負面訊號？

查看建議思考方向
  • 歷史對照：「代碼行數」曾是衡量工程師生產力的錯誤指標。Commit 數量是否是另一個代碼行數陷阱？哪些指標能更好地衡量 AI 輔助開發的品質？
  • 講者隱含的論點：CTO 展示 commit 曲線，重點不是在說「多就是好」，而是在說「我們的文化和工具允許持續交付」。曲線是文化的代理指標，而非直接的品質指標。
  • 職涯啟發：如果你是技術主管，如何設計評估 AI 工具效益的指標體系？除了 commit 數量，還應該追蹤什麼？（例如：bug 率、PRD 到上線週期、客戶滿意度）
  • 反向思考：大量的 AI 輔助 commit 可能意味著程式碼品質的一致性下降——每個開發者用 AI 的方式不同，可能導致代碼風格更混亂。Omni 如何處理這個問題？

Q2：語義層解決了「LLM 不理解企業私有數據」的問題。但語義層本身需要人工維護（定義欄位、更新上下文）。隨著公司規模增長，語義層的維護成本會不會成為新的瓶頸？

查看建議思考方向
  • 歷史對照：數據倉儲的「數據字典」（Data Dictionary）曾是企業數據治理的核心痛點——很快就過時，沒人願意更新。語義層是否面臨同樣的問題？
  • 講者隱含的論點：Omni 提到「反饋循環」——用戶的問答歷史可以反饋更新語義定義。這暗示語義層應該是自我維護的，而非依賴人工。這個假設成立嗎？
  • 職涯啟發：如果你是數據工程師，在公司引入 AI 數據分析工具的第一年，應該投入多少精力在語義層的「冷啟動」建設上？這是 80/20 的哪 80%？
  • 反向思考：語義層的存在本身是否創造了組織內的「數據知識壟斷」——只有維護語義層的人才知道「正確答案」，這是否反而阻礙了數據民主化？

Q3：Blobotomy 的解法是「合併大腦」——把子 Agent 的工具拉回外部 Agent。但這是否犧牲了系統的模組化？如果 SQL 生成工具被固定在外部 Agent，未來要在新場景中複用它，難度是否增加了？

查看建議思考方向
  • 歷史對照：軟體工程中「單一職責原則」和「高內聚低耦合」的設計準則。Blobotomy 的解法看起來違反了這些原則——它增加了外部 Agent 的職責。長期看是否有更好的架構方案？
  • 講者隱含的論點：CTO 承認「原始設計有點太聰明了（a bit too clever）」。這是一個難得的承認——高明的設計有時反而帶來隱性複雜度，而「笨一點」的設計更可靠。
  • 職涯啟發：在 Multi-agent 系統設計中，「合理的分工」和「過度的分工」之間的邊界在哪裡？有沒有比「合併大腦」更好的方式讓外部 Agent 理解子 Agent 的能力邊界？
  • 反向思考：如果你給外部 Agent 一個「關於子 Agent 能力的說明文件」，讓它在規劃任務時知道子 Agent 的邊界，是否能解決問題而不需要合併大腦？

Q4：Omni 選擇復活舊的 SQL 解析器，並賭注「Anthropic 會持續強化 Claude 的 SQL 能力」。這種「押注供應商路線圖」的策略有哪些風險？

查看建議思考方向
  • 歷史對照：企業技術選型中「押注某個供應商」的歷史案例——Oracle 鎖定、AWS 服務深度整合、Salesforce 客製化。這些案例的教訓是什麼？
  • 講者隱含的論點：CTO 說這是「a pretty safe bet」，因為 SQL 是 AI 訓練的高價值技能，且數據分析市場足夠大，Anthropic 有商業動機持續投資。這個邏輯成立嗎？
  • 職涯啟發：作為工程師，你如何評估「押注 AI 供應商能力提升」的技術決策？有哪些領域是「安全的賭注」，有哪些是高風險的？
  • 反向思考：如果 Anthropic 未來改變 Claude 的 SQL 生成行為（例如為了安全考量限制某些 SQL 操作），Omni 的 SQL Parser 系統將面臨什麼挑戰？

Q5：Omni 的 Eval 系統有兩個層面：內部研發用和客戶用。把評估系統「開放給客戶」意味著什麼？客戶是否能透過評估系統的反饋，間接影響 Omni 的模型行為？

查看建議思考方向
  • 歷史對照：開源軟體中「用戶報告 bug」的機制。Omni 的客戶 Eval 系統類似於「可量化的 bug 報告」——客戶明確指出「這個問題的答案是錯的」，這比傳統反饋表單更精確。
  • 講者隱含的論點：CTO 強調「可預測性和品質」是 Omni 提供的核心價值，而客戶 Eval 系統是量化這個承諾的工具。這暗示 B2B AI 產品的競爭力，部分來自「可測量的品質保證」。
  • 職涯啟發：如果你要為 B2B AI 產品設計客戶 Eval 系統，最難的部分是什麼？（例如：如何定義「正確答案」、如何防止客戶操縱評估結果）
  • 反向思考：客戶 Eval 系統是否創造了「為評估而優化」的風險——系統在評估集上表現好，但在真實使用中反而退步？這是 Goodhart 定律的 AI 版本嗎？

Q6：「開發者即使用者」的原則讓 Omni 工程師對好的 Agent 框架有深度共鳴。但如果你的工程師不是你產品的目標用戶（例如你在做醫療 AI、但工程師不是醫生），這個原則如何應用？

查看建議思考方向
  • 歷史對照：用戶研究（User Research）和同理心設計（Empathy Design）的出現，正是為了解決「開發者不是用戶」的問題。Airbnb、Dropbox 的早期用戶研究故事。
  • 講者隱含的論點：CTO 的重點不只是「用自己的產品」，而是「從使用中建立直覺——知道一個好的框架應該是什麼感覺，然後把這個直覺投射到你為客戶建的系統中」。這個「直覺轉移」的能力才是核心。
  • 職涯啟發：在你目前的工作中，有哪些機會讓你「成為自己產品的用戶」？如果沒有，你如何建立對用戶痛點的直覺感知？
  • 反向思考：Omni 的工程師使用 Claude Code（他們的開發工具），然後把這個體驗投射到 Blobby（他們的產品）上。但這兩種工具的用戶期望完全不同——開發者的容忍度 vs. 業務用戶的容忍度。這個類比是否存在盲點？
