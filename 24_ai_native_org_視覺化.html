I

過去有效的，現在不一定有效

Fiona Fung 帶領著每天使用 Claude Code 打造 Claude Code 的團隊。她的演講核心是一個成長心態的宣言：情況已改變，不審計舊規範就是對團隊的傷害。

多年來，工程頻寬一直是最昂貴的資源。所有的流程——深入的規劃、嚴格的審查、謹慎的程式碼所有權——都是為了保護這個資源而設計的。現在，這個前提已經不成立了。

"What worked for you in the past might not work anymore." 「過去對你有效的方法，現在可能不再有效。」

— Fiona Fung, Anthropic

Fiona 的演講不只是關於工具——它是關於組織的成長心態。當瓶頸轉移，那些「理所當然」的流程就需要重新被質疑：它是否還符合其目的？

給初學者 · 一句話搞懂 AI-native 工程組織
什麼是「AI-native 工程組織」？

不是「用 AI 輔助的普通工程團隊」，而是從根本上重新設計了假設的團隊——假設編碼不再是瓶頸，假設驗證是新的核心能力，假設每個人都應該每天使用自己打造的產品。

  • 普通團隊用 AI：加了一個 GitHub Copilot，流程其他不變
  • AI-native 團隊：重寫 Onboarding、規劃流程、程式碼審查、組織架構——一切都基於「Claude 在團隊裡」這個新前提
  • Fiona 的關鍵問題：「每個流程——它是否還符合其目的？如果不符合，殺掉它。」

AI-native 是一種組織心態，不是一個工具清單。

10 個核心洞見

01 瓶頸轉移：從「工程頻寬（Coding）」→ 雲端時代「建置流程」→ AI 時代「驗證、知識共享、維護」。
02 程式碼勝出（Code wins）：技術爭論不用白板——生成 3 個 PR 版本，讓程式碼本身說話。「建置便宜，爭論昂貴。」
03 TDD 終於有趣了：以前寫測試像吃花椰菜，有了 Claude，TDD 的「稅賦」被拿掉了——測試驅動開發第一次變得令人享受。
04 Left Shift 驗證：增加吞吐量後，驗證必須更靠近源頭——自動化機制比等用戶遇到 Bug 更好，比我先遇到 Bug 更好。
05 程式碼是唯一事實來源：文件在高吞吐量時代更容易過期。把規格（Spec）簽入程式碼庫，讓程式碼成為 Single Source of Truth。
06 Ant food（螞蟻食物）：Anthropic 版的 Dogfooding。主管必須親自使用產品，才能「在骨子裡感受它」——不靠 Dashboard，靠體驗。
07 主管從 IC 開始：Claude Code 每位主管都從個人貢獻者（IC）做起。維持與程式碼庫的連結是保持產品洞察力的關鍵。
08 跨功能鴻溝由 Claude 填補：設計師用 Claude 直接做 UI 微調；工程師用 Claude 做內容設計——Claude 增強你較弱的領域。
09 明確允許殺死舊流程：流程不會自己消亡。組織需要明確的文化授權——「如果它不再符合目的，你可以殺掉它」。
10 自上而下 + 自下而上：全團隊必須做的「強迫函數」（頂層對齊）+ 各小組有自由 Claude 化自己的工作流（底層適應）並行。
II

瓶頸的歷史轉移

Fiona 帶大家坐上時光機回到 2000 年代初期：那時沒有雲端，Visual Studio 的建置是一個佇列，一次只能合併 6 個 PR。每個時代都有它的瓶頸。現在到了 AI 時代。

2000s
初期
瓶頸

建置流程（Build Pipeline）

Visual Studio 建置佇列：一次合併 6 個 PR，測試失敗要找是哪個 PR 造成的。機房值班一週，一切圍繞著保護建置資源。

2010s
雲端時代
瓶頸

工程頻寬（Engineering Bandwidth）

雲端解放了建置問題，但工程師時間成為最貴的資源。所有流程都是為了保護工程師：深入規劃、嚴格審查、謹慎的程式碼所有權。

2026
AI 時代
新瓶頸

驗證、知識共享、維護

編碼不再是瓶頸。吞吐量大幅增加後，「它是否正確」比「能否寫出來」更重要。Verification、知識共享、維護成本是新挑戰。

TDD 的頓悟時刻

測試驅動開發的「稅賦」消失了

Fiona 加入 Claude Code 時想嘗試 TDD（測試驅動開發）——先寫測試、確保它失敗、再修正 Bug。但她記得以前寫測試的感覺：

「就像吃花椰菜——你知道它對你好，但它不是流程中最令人享受的部分。我總是很癢心，想趕快去建置產品。」

有了 Claude，她先寫失敗的測試，讓 Claude 幫助修正 Bug，完成後測試自然存在了。

「那第一次的體驗，TDD 實際上變得有趣得多、令人愉悅得多——它有點像是把 TDD 中的『稅賦』給拿掉了。這對我來說是一個很大的 Aha moment。」

教訓：不只是速度更快，而是高品質工程實踐的「心理負擔」大幅降低——原本覺得吃力才能做到的好習慣，現在變成自然而然的事。

給初學者 · 一句話搞懂 TDD
什麼是 TDD（測試驅動開發）？

先寫測試（先讓它失敗），再寫代碼讓測試通過——不是先寫代碼再補測試。順序是：紅（失敗）→ 綠（通過）→ 重構。

  • 好處：測試反映真實需求，而不是為了通過而寫的假測試
  • 以前的問題：需要額外的時間和紀律——「稅賦」很高
  • 現在：Claude 幫你生成測試框架，TDD 的摩擦力降到幾乎為零

Fiona 的洞見：Claude 不只讓你寫得更快，它讓「做對的事」變得更容易。

"In technical debate, code wins. Build is cheap, argument is expensive." 「在技術爭論中，程式碼勝出。建置很便宜，爭論很昂貴。」

— Fiona Fung
技術爭論新範式

Fiona vs Boris：三個 PR 取代白板

Fiona 在 Onboarding 期間想做一些程式碼重構。她和 Boris（團隊工程師）對方法有幾次爭論。

「在過去，我差點就說『Boris，我們去會議室用白板畫出來吧。』但我心想：等等，有了 Claude，我可以生成三個不同版本的 PR。」

她生成了三個不同重構方案的 PR，不是為了看程式碼如何實作，而是要看對整體架構的影響。這三個 PR 成為她和 Boris 進行技術討論的素材——比白板更具體、更可執行、更可以直接 merge 其中一個。

這個故事的核心：在 AI 時代，「先做出來再討論」比「先討論再做」更有效率。建置成本下降後，原型（Prototype）和 PR 取代了設計文件和白板作為技術討論的主要媒介。

III

Verification：新時代的核心能力

當吞吐量大幅提升，每個變更都需要更高的驗證信心。Left Shift 是關鍵——讓自動化機制在更靠近源頭的地方抓住問題，而不是等到用戶回報。

Fiona 說：「當我聽到客戶遇到 Bug，感覺真的很糟糕。我總是想著：我真希望我們能先抓到它。但你知道有什麼比我先遇到 Bug 更好的嗎？就是我們有自動化機制在線，在更接近源頭的地方就抓住它。」

給初學者 · 一句話搞懂 Shift Left
什麼是「Left Shift」（左移）？

想像一條時間軸，左邊是「寫程式碼」，右邊是「用戶回報 Bug」。Left Shift 就是把「發現問題」的時機從右邊的（用戶遇到）移到左邊（寫程式碼時）。

  • 最右邊（最糟）：用戶在生產環境遇到 Bug → 最貴，已造成傷害
  • 中間：工程師在 Code Review 發現 → 還可以，但有浪費
  • 最左邊（最好）：自動化在提交前就抓住 → 最便宜，問題最小

AI 時代的 Shift Left：不只是自動測試，更是讓 Claude 在提交前驗證規格是否偏離（Spec Drift Detection）。

I

Claude 程式碼審查 — 擅長的領域

風格（Style）、Lint、明顯的 Bug、規格偏離（Spec Drift）——把規格簽入程式碼庫，Claude 能自動驗證實作是否符合規格。

II

人類在流程中 — 不可替代的領域

法律審查、風險承受度判斷、信任邊界評估、產品感與品味（Taste）——這些需要人類的上下文理解和價值判斷。

III

自動化早晨常規（Morning Routine）

Fiona 的做法：每天早上喝咖啡時，自動整合所有回饋管道（用戶回饋、App Store 評論等），Claude 協助識別主題和趨勢。

產品感 × 人類審查

把 Claude 變成花生先生：設計師的眼睛

去年 12 月，Fiona 想給 Claude CLI 加節慶主題——把 Claude 的 ASCII 藝術做成一個雪人。她做出了範例，然後去問設計師意見。

「設計師說：『不，這看起來完全不像雪人，你把 Claude 變成花生先生了。』」

花生先生（Mr. Peanut）是美國花生品牌的吉祥物——一個穿著燕尾服的花生人偶。Fiona 沉默了一秒，然後說：「天哪，完全沒錯。我以為他是雪人，但你是對的。」

這個故事的意義：產品感（Product Sense）和美學品味（Taste）是人類在 AI 時代程式碼審查中最不可替代的貢獻。Claude 無法感知「這看起來像花生先生而不是雪人」——這需要人類對文化語境的理解。

知識共享：程式碼作為單一事實來源

傳統上，設計文件和 Wiki 是知識庫。但在 AI 時代，高吞吐量讓這些文件更容易過期——因為它們不在「更新迴圈」裡。

給初學者 · 一句話搞懂 Source of Truth
為什麼「程式碼是唯一事實來源」？

想像一本菜譜（設計文件）和廚師實際做的菜（程式碼）——如果兩者不一致，你要相信哪個？廚房裡的實際做法，不是過期的菜譜。

  • 設計文件的問題：很少有人記得在改程式碼後同步更新文件 → 文件很快過期
  • 程式碼的優勢：任何人都可以跑它、Claude 可以讀它、它永遠是最新的
  • Fiona 的做法：把「規格（Spec）」寫成可執行的程式碼或測試，簽入 codebase → 規格自動保持更新

在 Claude Code 團隊，如果有人問技術問題，Fiona 說：「這是個好問題，我們去問 Claude」——而不是「去看文件」。

"Code is our single source of truth. No matter what your source of truth is, turn it into a skill and check it into the codebase — because then you can actually keep it updated." 「程式碼是我們的唯一事實來源。不論你的事實來源是什麼，把它變成一個技能並簽入程式碼庫——因為這樣你才能真正保持它的更新。」

— Fiona Fung
IV

Ant Food：在骨子裡感受產品

Anthropic 版的 Dogfooding 叫「Ant food（螞蟻食物）」。Fiona 的核心主張：產品感不是來自儀表板，而是來自親身使用——讓決策者「在骨子裡感受到產品」。

"Most of the time, anytime I join a new team, that's one of the first things I do — dogfooding the product to make sure I understand it. This way you really start feeling your product in your bones." 「大多數時候，每當我加入新團隊，這是我首先做的事情之一——內部試用產品以確保我理解它。透過這種方式，你真的開始在骨子裡感受到你的產品。」

— Fiona Fung
小企業 Onboarding 田野研究

餐廳老闆、Claude for Small Business 與謙卑的教訓

Fiona 對小企業有個人熱情。Anthropic 宣布 Claude for Small Business 後，她親自拜訪在舊金山開餐廳的朋友，引導他們使用 Co-work。

「我得到了如此寶貴的回饋。看到我們在引導流程中，其實有那麼多事情是我們可以做得更好的，這真的讓人感到謙卑。」

這就是「在骨子裡感受產品」的意義：不靠 Dashboard 數字，不靠 PowerPoint，而是看著真實用戶——一個餐廳老闆，不是工程師——試圖使用你的產品，感受他們的困惑和摩擦。這種體驗是任何指標都無法替代的。

給初學者 · 一句話搞懂 Dogfooding / Ant food
什麼是 Dogfooding？

「吃自己的狗糧」——讓自己的團隊成為自己產品的用戶。Google 工程師用 Google Search，Apple 工程師用 iPhone，Anthropic 工程師用 Claude Code 開發 Claude Code。

  • 目的 1：第一手感受產品的痛點，比任何用戶回饋都直接
  • 目的 2：讓主管「有資格」做決策——因為他們親身體驗過，不是靠數字猜測
  • Anthropic 的版本：「Ant food」（螞蟻食物），因為 Anthropic 的英文縮寫是 Ant

Fiona 說：「否則，一段時間後你可能發現自己做出的產品決策全都是基於指標、儀表板或 PowerPoint。」這是對脫離現實的管理層的最嚴厲批評。

主管從 IC 開始：技術連結的強迫函數

Fiona 在加入 Claude Code 後，首先改變的事情之一就是組織架構：所有主管都從個人貢獻者（Individual Contributor）開始做起。

給初學者 · 一句話搞懂 IC vs Manager
什麼是 IC（Individual Contributor）？

IC = 個人貢獻者，就是親自寫程式碼、做設計、做工程的人。Manager 是帶領 IC 的人，但通常隨著管理職責增加，逐漸脫離實際工程工作。

  • 傳統觀點：主管應該管理，IC 應該執行——角色分明
  • Fiona 的觀點：主管如果完全脫離程式碼庫，決策容易流於表面，失去對產品的真實感受
  • Claude Code 的做法：每位主管從 IC 開始，有時間建立技術直覺，再承擔管理責任

另一個好處：主管重新進入程式碼庫，不再需要佔用工程師的時間——有 Claude 可以回答所有技術問題。

V

思考題 · 六個值得帶回團隊的問題

Fiona 用一個「帶回家練習」結束演講：「挑出你最嘈雜的工作流程，問自己它是否還符合其目的，然後一步一步地去執行。」這六個問題是更深層的延伸。

「最嘈雜工作流程」練習框架

1
識別（Identify）

你現在最耗時、最貴、大家最不享受的工作流程或會議是哪個？想想那個每週有 15 個人參加、大家都在用筆電的會議。

2
審計（Audit）

這個流程最初是為了解決什麼問題？它現在還在解決那個問題嗎？如果原來的問題已經消失了，這個流程是否還有存在的必要？

3
Claude 化（Claude-ify）

這個流程的哪些部分可以由 Claude 完成或輔助完成？從一個小部分開始，一步一步地去執行。不要試圖一次革命所有流程。

Q1 · Fiona 說「過去有效的，現在不一定有效」——當工程頻寬不再是瓶頸，所有圍繞著保護它而設計的流程都需要重審。你的團隊現在有哪些「悄悄停止運作」的流程？最後一次有人認真審計這些流程是什麼時候？

查看建議思考方向
  • 歷史對照：Fiona 舉了她待過一個擁有「許多層 SLA」的團隊——P0 Bug SLA、事件回應 SLA——堆積到工程師都不知道應該先做哪個。SLA 的初衷是保護客戶，但過多的 SLA 變成內部協調負擔，反而傷害客戶體驗。
  • 講者隱含論點：Fiona 說「流程不會自己死去」——它需要主動的審計文化和明確的終止授權。「允許殺死舊流程」是一種文化設計，不是自然發生的。
  • 職涯啟發：能夠識別「這個流程的存在是為了解決哪個已不再存在的問題」的工程師/主管，在組織轉型中有巨大價值。這需要同時理解歷史和當前現實。
  • 反向思考：「殺死舊流程」有風險——它可能在解決你沒有意識到的問題。如何設計一個「安全試驗：暫停這個流程 4 週，看看有什麼崩潰」的機制？

Q2 · Fiona 的 TDD 頓悟：有了 Claude，「吃花椰菜」的稅賦消失了，TDD 第一次變得令人享受。在你自己的工程實踐中，有哪些你知道「對的做法」但因為摩擦力太高而常常跳過的事？Claude 有可能讓這些事情變得自然？

查看建議思考方向
  • 量化參考：研究顯示，TDD 能減少 40-80% 的缺陷，但採用率一直很低——因為「先寫測試」的心理負擔太高。如果 Claude 讓 TDD 的摩擦力降到接近零，TDD 採用率的理論上限是什麼？
  • 講者隱含論點：Fiona 的頓悟不只是關於速度，而是關於「正確的工程實踐變得可負擔」。這暗示 Claude 的最大影響之一，是讓「最佳實踐」從「知道但不做」變成「自然而然做」。
  • 職涯啟發：有一份「你知道但因為太麻煩而跳過」的工程實踐清單，然後系統性地用 Claude 逐一試驗降低摩擦力——這是非常值得做的個人工程改善實驗。
  • 反向思考：如果 Claude 讓「吃花椰菜」變得容易，是否也有風險讓某些應該有摩擦力的決策（如刪除生產資料庫）變得太容易了？「正確的摩擦力」在哪裡？

Q3 · Fiona 說「程式碼是唯一事實來源」，並建議把規格（Spec）簽入程式碼庫。但現實中很多組織的「事實來源」是分散的——Confluence、Notion、Slack、口頭約定。你的組織現在的「事實來源」在哪裡？將它遷移到程式碼庫需要面對什麼阻力？

查看建議思考方向
  • 歷史對照：敏捷宣言（2001 年）其中一條是「可運作的軟體勝過詳盡的文件（Working software over comprehensive documentation）」——Fiona 的「程式碼即事實來源」是這個原則在 AI 時代的延伸。25 年後，AI 讓這個原則更容易實踐。
  • 講者隱含論點：Fiona 說高吞吐量讓「不在更新迴圈裡的文件更容易過期」——這意味著以前「花時間維護文件」在低吞吐量時代還可以，但現在已經跟不上。
  • 職涯啟發：設計「讓規格可執行、可測試、可自動驗證偏離」的架構是一個高價值技能。能把「業務邏輯」轉化為「可測試的程式碼規則」的工程師，在 AI 時代很稀缺。
  • 反向思考：不是所有知識都能轉化為程式碼——架構決策背後的「為什麼」、設計哲學、商業背景通常不能。程式碼告訴你「做什麼」，但不告訴你「為什麼」。如何在「程式碼即事實」和「保留決策上下文」之間取得平衡？

Q4 · 花生先生事件：Claude 無法判斷 ASCII 雪人「看起來像花生先生」，需要設計師的文化語境和美學品味。在你的工作中，哪些任務需要這種「人類才有的文化/美學/上下文判斷」？AI 的邊界在哪裡？

查看建議思考方向
  • 歷史對照：自動駕駛能辨識紅燈，但難以辨識「這個十字路口的人是在等待還是要過馬路」——因為這需要理解人類的隱含意圖和社會脈絡。AI 的能力邊界通常在「顯式規則」和「隱含文化脈絡」之間。
  • 講者隱含論點：Fiona 說「人類的產品感和品味」是程式碼審查中不可替代的，並且用「花生先生」這個故事來說明這個不可替代性不是抽象的，而是非常具體的。
  • 職涯啟發：識別哪些工作需要「文化語境 + 美學判斷 + 隱含人類理解」，刻意培養這些能力——這是 AI 無法短期取代的人類競爭優勢。設計師的「品味」是護城河。
  • 反向思考：隨著 AI 接觸更多人類文化數據，它的文化語境理解能力也在提升。「花生先生問題」在三年後的 Claude 版本中還會是問題嗎？人類的「不可替代優勢邊界」本身也在移動。

Q5 · Fiona 主張每位主管都應該從 IC 做起，直接進入程式碼庫。但在許多組織中，「主管親自寫程式碼」被認為是「不信任工程師」或「微管理」的訊號。你如何區分「保持技術連結」和「過度介入」？這條線在哪裡？

查看建議思考方向
  • 歷史對照：Amazon Jeff Bezos 在 Amazon 初期親自寫程式碼，但隨著公司增長必須過渡到純管理。另一方面，Linus Torvalds 繼續參與 Linux kernel 的技術決策，而不只是管理。不同規模和類型的組織有不同的答案。
  • 講者隱含論點：Fiona 強調的是「IC 開始再過渡到管理」，而不是「永遠既是 IC 又是 Manager」——目的是讓主管有足夠的「肌肉記憶」和技術直覺，而不是永遠自己做工程。
  • 職涯啟發：「技術連結」和「微管理」的區別在於：技術連結是為了建立判斷力，微管理是控制工程師的決策。前者增加對話的深度，後者減少工程師的自主性。
  • 反向思考：如果 Claude 讓主管可以更輕鬆地「重新進入程式碼庫」，是否也帶來了主管「對技術過度自信」的風險？工程師的實際深度和主管的 Claude 輔助深度之間，有多大的差距？

Q6 · Fiona 的演講是整個 Code with Claude London 2026 的最後一場，她帶來的訊息是「組織要保持成長心態，不斷審視舊規範」。回顧這兩天 24 場演講，哪一個觀點最讓你感到「我需要改變我目前的做法」？你打算明天做什麼不同的事？

查看建議思考方向
  • 歷史對照：每一次重大技術轉型（網際網路、雲端、智慧手機）後，「最快適應的組織」不一定是最大的，而是最願意重新質疑自己假設的。這兩天的演講集體指向一個方向：AI 不是工具升級，而是假設系統升級。
  • 講者隱含論點：Fiona 的最後訊息是「變化的速度在加快，成長心態不是選擇，而是生存要求」——這也是整個 Code with Claude London 的隱含主題。
  • 職涯啟發：從 24 場演講中，每個人應該識別出 1-3 個具體的、可立即採取的行動——不是「我要更多使用 AI」這種模糊目標，而是「明天我要做 X」的具體行動計劃。
  • 反向思考：Fiona 說「不要只看端到端指標，要把它拆解成各個漏斗階段」。你現在對自己效率的衡量方式，是否存在同樣的盲點——你在衡量的指標，是否真的反映了你想提升的東西？
給初學者 · 名詞速查
本場演講關鍵術語
  • Bottleneck（瓶頸）：限制整個系統吞吐量的最薄弱環節——當它改變，整個組織的最佳策略也改變
  • TDD（Test-Driven Development）：先寫測試、確保失敗、再寫代碼讓測試通過的開發方式
  • Shift Left（左移）：把問題發現的時機向「源頭」移動，越早發現越便宜
  • Spec Drift（規格偏離）：實作隨著時間偏離了原始設計規格，而沒有人注意到
  • Ant food：Anthropic 版的 Dogfooding，讓自己的團隊使用自己的產品
  • Product Sense（產品感）：對產品應該如何工作的直覺，來自大量使用和觀察用戶
  • IC（Individual Contributor）：個人貢獻者，親自做工程/設計工作，非管理角色
  • Source of Truth：唯一正確的、最新的資訊來源，其他地方的記錄都以它為準
  • Forcing Function（強迫函數）：讓某個行為不得不發生的機制或結構
