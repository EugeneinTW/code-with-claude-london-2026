I

無摩擦開發的願景

Emanuele Nardini 的演講從一個場景調查開始：大多數人這週使用了 AI 編碼工具，但極少數人真正將其部署至 Google Cloud。這場演講的目標，就是補上這道落差。

企業軟體開發向來需要一整個團隊：產品經理提出構想、UI/UX 設計師將其視覺化、後端工程師建構核心邏輯、安全工程師確保合規、數據分析師收集洞察。而 Claude Code 的核心主張是：它不取代這個團隊，而是增強整個生命週期的效能。

演講者以一個「五帽實驗」來驗證這個主張——一個人，在 28 分鐘內，完成一個從草圖到生產環境部署的完整 Feedback App，並實時從 BigQuery 產出 Looker 互動儀表板。

"Claude Code provides a set of capabilities that will essentially augment them across this entire software life cycle." 「Claude Code 提供了一系列能力，能夠在整個軟體生命週期中強化每一個角色的效能。」

— Emanuele Nardini, Google Cloud Developer Advocate
給初學者 · 一句話搞懂「軟體生命週期」
為什麼一個 App 需要「五種人」？

想像你要蓋一棟房子。你需要：建築師畫圖、室內設計師規劃空間、工程師建結構、消防局驗安全、物業管理收集住戶反饋。軟體開發也一樣——每個階段有不同的專業。

  • PM（產品經理）：決定蓋什麼，畫草圖
  • UI/UX 工程師：設計外觀和使用動線
  • 後端工程師：建造水電管線（邏輯和資料庫）
  • 安全工程師：消防驗收，確保沒有漏洞
  • 數據分析師：記錄住戶行為，持續優化

Claude Code 的野心是：它能協助所有五個角色完成工作，不是取代任何一個，而是讓每個人都跑得更快。

核心洞見速覽

I

Planning Mode 先思考後行動

Claude Code 的規劃模式讓 AI 在撰寫程式碼前先提案、確認，減少因誤解需求造成的無效工作。這是「副駕駛→專案代理人」進化的關鍵一步。

一個指令，三個子代理人同時建構 API、數據管線和儀表板，就像 Agile Sprint 中的跨功能小隊——只是全部都在 terminal 裡發生。

大語言模型有訓練截止時間，但 Developer Knowledge API MCP Server 讓 Claude 永遠拿到最新的 Google Cloud 文件，消除 Hallucination 的根源。

II

為什麼在 Google Cloud 上執行 Claude？

Google Cloud 不只是 Claude 模型的一個執行平台，它帶來了傳統 API 存取無法提供的企業級優勢：免密鑰存取、資料主權、高可用性，以及專屬的開發者工具生態。

給初學者 · 一句話搞懂「Application Default Credentials」
ADC 是什麼？為什麼「免密鑰」很重要？

想像你在公司辦公室用員工識別證刷卡進門，這就是 ADC 的概念——你的「身份」由環境決定，不需要額外拿一把鑰匙（API Key）。

  • 傳統方式：申請 API Key → 存在環境變數 → 每隔幾個月手動輪換 → 萬一洩漏就全完了
  • ADC 方式：登入 Google Cloud 帳號 → 環境自動識別 → 無需任何密鑰

對企業來說，消除 API Key 就是消除最常見的安全漏洞來源之一。

「什麼都沒變」的魔法

Emanuele 在說明設定流程時強調：「Nothing change with respect to your environment.」工程師繼續使用熟悉的 Claude Code 介面，繼續寫同樣的提示詞——唯一改變的是底層的計費和資料路由，以及多了一層企業安全保障。

這是設計上的智慧：最好的安全機制，是讓工程師感覺不到它的存在。ADC 讓安全合規變成「預設狀態」，而非「額外負擔」。

給初學者 · 一句話搞懂「多區域端點」
為什麼「多區域」對企業很重要？

想像你是一家跨國公司的工程師。如果 AI 服務只在美國有伺服器，那你在歐洲的用戶每次呼叫都要繞地球一圈——速度慢、GDPR 合規也有風險。

  • 多區域端點：Claude 在各大洲都有伺服器，請求就近處理
  • SLA 保障：服務水準協議確保 99.9%+ 可用性，宕機有賠償
  • 合規優勢：歐洲資料可設定留在歐洲，符合 GDPR

對企業 IT 部門來說，這三點合在一起，才是真正「可以採購」的條件。

CI/CD 是什麼？為什麼重要？

CI/CD 就像汽車工廠的自動化生產線。工程師的程式碼（零件）一進倉庫，流水線自動測試品質（CI）、然後自動裝配出廠（CD），不需要人工逐步操作。

  • CI（持續整合）：每次提交程式碼，自動執行測試，確保沒有破壞現有功能
  • CD（持續部署）：測試通過後，自動部署到伺服器
  • Cloud Build：Google 的 CI 服務，負責測試和打包
  • Cloud Deploy：Google 的 CD 服務，負責版本管理和環境升級

Claude Code 在這裡的貢獻是：自動生成 CI/CD 的配置文件（YAML/Dockerfile），把本來需要 DevOps 專家數小時才能設置好的流水線，壓縮到幾分鐘。

關鍵時刻

merge 那一秒，流水線啟動了

演講現場，當 Emanuele 在 GitHub 點下 Merge PR 按鈕的那一刻，Google Cloud 的 Cloud Build 自動觸發了一個構建流水線。台下觀眾在大螢幕上實時看到：API 的構建日誌、BigQuery 管線的初始化、最終 Cloud Deploy 推送一個新版本到開發環境，並自動等待升級確認。

"Once this pipeline finish to run, it will push a new release on the continuous deployment tool... and it will trigger a release."

這個 Demo 的設計很聰明：它不是事先準備好的結果展示，而是讓觀眾即時看到整個自動化鏈條的每個環節。這種「真實性」比任何 slide 都更有說服力。

III

Hat 4：安全工程師與 Plugin 機制

在應用程式進入生產環境前，安全工程師的帽子上場了。這個環節展示了如何把公司特有的安全規則，編碼成 Claude Code Plugin，讓安全審查變成可重複、可自動化的流程。

Plugin 做了什麼？

自訂 OWASP Top 10 漏洞掃描 + Service Account 權限最小化審查。Claude 發現 Input Validation 不足後，自動修正程式碼、重跑測試、重新提交 PR。

為什麼用 Plugin 而非手動 review？

每家公司的安全要求不同。Plugin 讓這些特有規則變成可版本控制、可共享、可演化的知識資產，而非靠個別工程師的記憶傳承。

給初學者 · 一句話搞懂「OWASP Top 10」
OWASP Top 10 是 Web 安全的「必考題清單」

OWASP（Open Web Application Security Project）每幾年發布一次最常見的 Web 安全漏洞前 10 名。這是業界共識的「最低標準」。

  • No.1 Broken Access Control：未授權用戶存取了不該存取的資源
  • No.3 Injection：SQL 注入、命令注入等
  • No.7 Identification & Auth Failures：認證漏洞
  • Input Validation（本次 Demo 發現的）：對用戶輸入未做充分驗證

Claude 的安全 Plugin 掃描這些已知漏洞，就像讓一個熟記所有常見漏洞的資安工程師，自動跑了一遍程式碼審查。

升級（Promotion）機制值得特別注意：Cloud Deploy 的設計要求在每次從 Dev 升級到 Production 時，有一個明確的人工確認步驟。Claude Code 自動建立了這個審核節點，確保自動化不會完全繞過人類判斷。

給初學者 · 一句話搞懂「BigQuery vs Firestore」
兩種資料庫，兩種用途

就像「冰箱」和「糧倉」——兩個都是存食物的地方，但用途完全不同。

  • Firestore（冰箱）：快速存取、即時讀寫、適合 App 的操作型資料（用戶提交的 Feedback 表單）
  • BigQuery（糧倉）：大量歷史資料、複雜分析查詢、適合計算統計、挖掘趨勢
  • 數據管線：把冰箱的食材定期搬到糧倉，讓分析師可以對大量歷史數據做研究

Claude 在 Hat 3 同時建構了這兩個資料庫的 Schema，以及把 Firestore 數據流向 BigQuery 的管線——而且全部基於官方 MCP 文件中的最佳實踐。

一個鏈接，從終端機到 Looker

Hat 5 的流程展示了現代 AI 輔助分析的極端效率：BigQuery MCP Server 直接連接倉儲，分析 Firestore 導入的原始表單數據，在終端機中計算出平均填寫時間和統計分布。

但數據還在終端機裡——PM 看不懂。這時，第二個 MCP Server（Looker MCP）接手，把那些數字自動轉換成互動式儀表板，包括時間分布圖、百分位數比較，以及概率分布疊加圖。

"Even if you don't know like a dashboard tool, you were able to build this using Claude Code and the MCP server that we provide."

PM 打開那個鏈接，看到的是一個完整的 Looker Dashboard——可以篩選時間範圍、對比不同頁面的填寫時間，甚至識別出哪個步驟讓用戶花最多時間。這個洞察直接指向下一輪 UX 優化的方向。

商業智能（BI）工具解決的問題

資料庫是給工程師看的。PM 和管理層需要的是「圖表和趨勢」，不是一行行數字。BI 工具就是把資料庫裡的數字，轉換成可以互動、可以展示、可以做決策的視覺化介面。

  • Looker（Google）：雲端 BI 工具，可連接 BigQuery
  • Tableau / Power BI：市場上的競品，原理相同
  • 過去的問題：需要專門的 BI 工程師花幾天建立儀表板
  • Looker MCP：Claude 直接調用 Looker API，自動生成儀表板結構

這是「民主化分析」的下一步：不只是工程師能寫 SQL，連 MCP Server 都能幫你把 SQL 轉成儀表板。

V

深度思考題

這六道問題從 Emanuele 的演講出發，向外延伸至更廣的技術趨勢和職涯思考。每道題附有四個建議切入角度。

Q1：ADC（Application Default Credentials）消除了 API Key，但這是否也引入了新的安全風險？「隱式憑證」比「顯式密鑰」是否真的更安全？

查看建議思考方向
  • 歷史對照：OAuth 2.0 和 Service Account Key 的安全事故記錄——哪個洩漏頻率更高？Google 的 Key Guardian 統計數據。
  • 講者隱含的論點：Emanuele 的重點不只是「安全」，而是「安全的摩擦力」——過高的安全摩擦（複雜的密鑰管理）會導致工程師走捷徑，反而更不安全。
  • 職涯啟發：如果你在企業內部推廣 AI 工具，安全合規通常是最大的阻礙。理解「零摩擦安全」的設計哲學，能讓你的提案更容易被 CISO 接受。
  • 反向思考：ADC 依賴環境身份，意味著「誰控制了執行環境，就控制了 API 存取權限」。這在 CI/CD 環境中可能引入新的攻擊面——如果有人汙染了 CI 伺服器，後果是什麼？

Q2：Planning Mode 讓 AI 先「思考後提案」再執行。這個設計和傳統軟體工程中的「需求文件」有什麼本質差異？這對敏捷開發（Agile）的衝擊是什麼？

查看建議思考方向
  • 歷史對照：瀑布式開發（Waterfall）強制要求詳細需求文件，Agile 刻意避免過早規格化。Planning Mode 是第三條路嗎？
  • 講者隱含的論點：Planning Mode 解決的不是「需要不需要規格」的問題，而是「AI 在不確定時傾向猜測而非詢問」的行為模式。這是 AI 代理人特有的挑戰。
  • 職涯啟發：PM 和工程師的溝通成本是軟體開發的最大浪費之一。如果 Planning Mode 能機器化地降低這個成本，那 PM 的核心價值在哪裡？
  • 反向思考：有時候「先做後改」比「先規格後做」更快。Planning Mode 是否增加了整體交付時間？在什麼場景下應該跳過它？

Q3：MCP Server 每 24 小時刷新文件，幫助 Claude 避免 Hallucination。但 MCP Server 本身提供的文件，Claude 是否會完全相信而不做批判性驗證？這是否只是把 Hallucination 的源頭移了位置？

查看建議思考方向
  • 歷史對照：RAG（Retrieval-Augmented Generation）技術同樣試圖用「查詢再生成」解決知識截止問題，但 RAG 準確性高度依賴檢索品質。MCP 和 RAG 的區別是什麼？
  • 講者隱含的論點：Emanuele 聲稱 MCP 解決了 Hallucination，但演講中的例子都是「查詢成功、架構正確」的案例。系統性錯誤的場景呢？
  • 職涯啟發：如果你負責建構公司內部的 MCP Server（如公司私有 API 文件），你需要考慮哪些資料品質保障機制？文件本身的錯誤、過時或不完整該怎麼處理？
  • 反向思考：訓練數據中的「通用程式設計知識」vs. MCP Server 中的「特定服務文件」——哪個對 Claude 的決策品質影響更大？是否存在兩者相互矛盾的情況？

Q4：Sub-agents 並行開發能加速工作，但也引入了協作一致性的風險——三個 Sub-agent 分別建構 API、數據管線和 Dashboard，它們的介面定義如何保持同步？

查看建議思考方向
  • 歷史對照：微服務架構（Microservices）同樣面臨服務間介面一致性問題，業界發展出了 Contract Testing、API Schema Registry 等解決方案。Sub-agents 是否需要類似的機制？
  • 講者隱含的論點：演講中，三個 Sub-agents 是由一個 API Specification 作為「合約」來協調的。這個細節很重要——「共享合約先行」是分散式開發的核心原則。
  • 職涯啟發：作為一個架構師，設計 Multi-agent 系統時，「合約定義」在哪個時間點、由哪個 Agent 負責生成，是關鍵的設計決策。
  • 反向思考：並行化的效益在什麼條件下會消失？如果 Sub-agents 生成的程式碼有衝突，解決衝突的成本是否可能超過並行帶來的收益？

Q5：「客製化 Security Plugin」讓安全規則可程式化、可版本化。但這是否意味著只有規則清單內的安全問題才會被檢查？未知漏洞（Zero-day）和特定業務邏輯漏洞怎麼辦？

查看建議思考方向
  • 歷史對照：靜態程式碼分析工具（如 SonarQube）的侷限性——它們能找到已知模式，但無法理解業務語義。Claude 的 LLM 能力是否改變了這個侷限？
  • 講者隱含的論點：Plugin 的設計目的是「把公司已知的安全要求自動化」，不是「發現未知漏洞」。這是一個謙遜但誠實的定位——把 AI 放在對的位置。
  • 職涯啟發：在現有工作流中，哪些安全檢查是「已知規則的機械性執行」？哪些需要「人類的判斷力」？明確這個邊界，是 AI 輔助安全工程的起點。
  • 反向思考：如果 Claude Code 的安全 Plugin 系統被廣泛採用，攻擊者是否會反過來研究常見 Plugin 規則，設計能繞過這些規則的攻擊方式？

Q6：從草圖到 Looker Dashboard，整個演示在 28 分鐘內完成，而且大部分由 AI 執行。這個未來裡，軟體工程師的核心競爭力是什麼？「寫程式」的技能是否正在被「定義問題」的能力取代？

查看建議思考方向
  • 歷史對照：計算機工業史上每次技術抽象層級提升（組合語言→高級語言→框架→無代碼）都引發了「程式員要消失了嗎」的討論，但程式員人數每次都增加了。這次有什麼不同？
  • 講者隱含的論點：Emanuele 始終強調人類在每個節點的「確認」和「決策」：確認 Planning Mode 的規格、審查 PR、批准 Security 通過、升級到 Production。AI 執行，人類決策。
  • 職涯啟發：如果「寫程式」的門檻降低，「產品判斷力」的稀缺性就會升高。未來工程師最有價值的能力，可能是知道「什麼該做」，而不是「怎麼做」。
  • 反向思考：Emanuele 的 Demo 非常順暢，但他本身是有深厚技術背景的 Developer Advocate。如果一個真正沒有技術背景的 PM 來嘗試，結果會一樣嗎？技術背景在 AI 時代的作用是什麼？
